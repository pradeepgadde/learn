io.StringIO and BytesIO: in-memory files

Pass in-memory text or bytes wherever a file object is expected. Unit-test file-writing code, stream without disk I/O, wrap network buffers.

30 Mar 2026 • ~2 min read • Python 3.0+
TL;DR

io.StringIO() is an in-memory text file; io.BytesIO() is an in-memory binary file. Both have the full file API (read, write, seek, getvalue). Use them to test code that takes a file object, to avoid temp files in tests, or to build buffers that get POSTed over HTTP.

Any function that does f.write(data) or f.read() works with a real file or with a StringIO/BytesIO — they implement the same interface. Knowing this buys you a lot of flexibility with zero change to the underlying code.

StringIO: in-memory text

pythonimport io, csv

# Build a CSV string without touching disk
buf = io.StringIO()
writer = csv.writer(buf)
writer.writerow(["hostname", "role", "ip"])
writer.writerow(["sw-01", "access", "10.0.0.1"])

csv_string = buf.getvalue()
print(repr(csv_string))
# 'hostname,role,ip\r\nsw-01,access,10.0.0.1\r\n'

csv.writer expects a file object; StringIO satisfies that without a file on disk. getvalue() returns everything that was written.

Read mode — feed a string into something that expects a file: 

pythonraw = "hostname,role\nsw-01,access\nsw-02,core\n"
for row in csv.reader(io.StringIO(raw)):
    print(row)

BytesIO: in-memory binary

pythonimport io, zipfile

# Build a ZIP in memory, return as bytes — no temp file needed
def make_zip(files: dict[str, str]) -> bytes:
    buf = io.BytesIO()
    with zipfile.ZipFile(buf, "w") as z:
        for name, content in files.items():
            z.writestr(name, content)
    return buf.getvalue()

zip_bytes = make_zip({
    "configs/sw-01.cfg": "interface gi0\n shutdown\n",
    "configs/sw-02.cfg": "interface gi0\n no shutdown\n",
})

buf.getvalue() gives all the bytes. Serve it directly as an HTTP response body or write it to S3 — no temp file, no cleanup.

Testing file-writing functions

The most common use: test code that writes to a file without creating a file. 

pythondef write_report(devices: list, f) -> None:
    for d in devices:
        f.write(f"{d.hostname},{d.role}\n")

def test_write_report():
    buf = io.StringIO()
    write_report([
        Device("sw-01", "access"),
        Device("sw-02", "core"),
    ], buf)
    assert buf.getvalue() == "sw-01,access\nsw-02,core\n"

No tmp_path, no file cleanup, no disk I/O. The test is faster, cleaner, and exercises the exact same code path.

seek and tell

After writing, the position is at the end. Call seek(0) to rewind before reading: 

pythonbuf = io.StringIO()
buf.write("hello\n")
buf.seek(0)
print(buf.read())          # "hello\n"

buf.seek(0, 2)             # seek to end (whence=2)
print(buf.tell())           # 6 — current position in bytes/chars

Wrapping network data

Code from libraries like paramiko sometimes writes to file objects. BytesIO captures the output without touching disk. 

pythonbuf = io.BytesIO()
sftp.getfo("/etc/hostname", buf)    # paramiko writes into buf
content = buf.getvalue().decode()
print(content.strip())                # "sw-01"

Sending in-memory bytes over HTTP

python# Upload a generated ZIP directly — no temp file
zip_data = make_zip(configs)
resp    = httpx.post(
    "https://store.example.com/upload",
    content=io.BytesIO(zip_data),
    headers={"Content-Type": "application/zip"},
)

TextIOWrapper: bridge between bytes and text

For APIs that write bytes but you want to read text (or vice versa): 

pythonraw_buf  = io.BytesIO()
text_buf = io.TextIOWrapper(raw_buf, encoding="utf-8")
text_buf.write("hostname sw-01\n")
text_buf.flush()
print(raw_buf.getvalue())

The pattern of accepting a file-like object rather than a path string is called the "file-object protocol" and is worth designing into your own APIs. Users can pass a real file, a StringIO for tests, or a compressed stream — and your code doesn't change.

Find one test in your suite that creates a temp file, writes to it, reads it back, and then cleans up. Replace the file with a StringIO or BytesIO. The test body shrinks; the three-step write/read/cleanup collapses; the underlying function still works identically.
