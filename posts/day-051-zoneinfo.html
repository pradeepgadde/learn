zoneinfo: timezones in the stdlib (and pendulum)

IANA timezones in the standard library since 3.9. You can finally retire pytz. pendulum still wins for date arithmetic.

20 Feb 2026 • ~3 min read • Python 3.9+
TL;DR

Use zoneinfo.ZoneInfo("Europe/London") for timezone-aware datetimes — no third-party install. Always store and pass UTC; convert at display time. Reach for pendulum when you need clean date arithmetic, durations, or human-friendly parsing.

Datetime in Python has a long, painful history. Python 3.9 mostly fixed it: zoneinfo ships with the stdlib, gives you the IANA database, and works natively with datetime. If you're still importing pytz in new code, stop.

Aware datetimes, the right way

pythonfrom datetime import datetime, timezone
from zoneinfo import ZoneInfo

# Now, in UTC — the only way to think about "now" in code
now_utc = datetime.now(timezone.utc)

# Same instant, displayed in another zone
now_ldn = now_utc.astimezone(ZoneInfo("Europe/London"))
now_ind = now_utc.astimezone(ZoneInfo("Asia/Kolkata"))

# Construct with a zone explicitly
meeting = datetime(2026, 3, 15, 14, 30,
                   tzinfo=ZoneInfo("America/New_York"))

Two rules that solve 90% of timezone bugs: store and compute in UTC, convert only at display. And never use datetime.now() without a tz= argument — "naive" datetimes are a footgun.

The naive vs aware split

A datetime is "aware" if it has a tzinfo attached, "naive" otherwise. They don't compare or subtract — you'll get TypeError. Pick one (aware) and stick to it. 

pythonnaive = datetime(2026, 2, 20)               # no tzinfo
aware = datetime(2026, 2, 20, tzinfo=timezone.utc)

aware - naive
# TypeError: can't subtract offset-naive and offset-aware datetimes

The DST trap

Adding 24 hours to a datetime across a DST transition gives you the wrong wall-clock time. Use timedelta for elapsed time, not for "tomorrow at the same hour." 

pythonfrom datetime import timedelta

tz = ZoneInfo("Europe/London")
before_dst = datetime(2026, 3, 28, 10, 0, tzinfo=tz)
tomorrow   = before_dst + timedelta(days=1)
tomorrow.strftime("%H:%M")
# '11:00' — DST kicked in, the wall clock moved an hour

For "tomorrow at this same wall-clock time", round-trip through naive and re-attach the tz: 

pythonnaive = before_dst.replace(tzinfo=None) + timedelta(days=1)
same_wallclock_tomorrow = naive.replace(tzinfo=tz)

Parsing ISO 8601

Python 3.11 fixed datetime.fromisoformat to handle the full standard, including the Z suffix for UTC. 

python# 3.11+
datetime.fromisoformat("2026-02-20T14:30:00Z")
datetime.fromisoformat("2026-02-20T14:30:00+05:30")
# Both give aware datetimes with the right offset

For ISO timestamps before 3.11 (or for less-strict formats), use dateutil.parser or pendulum.parse.

When to reach for pendulum

pendulum is a third-party library with the API datetime wishes it had: chained methods, clean arithmetic, durations as first-class objects. 

pythonimport pendulum

now = pendulum.now("Asia/Kolkata")
tomorrow = now.add(days=1)            # DST-aware
last_friday = now.previous(pendulum.FRIDAY)
duration = tomorrow - now
duration.in_hours()                       # 24

# Human-friendly diff
past = pendulum.now().subtract(hours=2, minutes=15)
past.diff_for_humans()                      # '2 hours ago'

For a script that lives entirely in the stdlib, zoneinfo + datetime is enough. For applications that do heavy date math or display human-friendly times, pendulum earns its dependency cost.

On Windows, install tzdata — zoneinfo needs the IANA database, which Windows doesn't ship. pip install tzdata. On Linux and macOS, the system provides it.

Find a script that does datetime.utcnow() (deprecated in 3.12) or datetime.now() without a timezone. Replace with datetime.now(timezone.utc). Convert to local time only when displaying. Time-zone bugs that have been hiding for years suddenly stop happening.
