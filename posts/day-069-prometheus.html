prometheus_client: metrics for the python process

Counter, Gauge, Histogram, Summary — the four metric types and when to use each. Plus the multiprocess gotcha for gunicorn workers.

10 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Use Counter for things that only go up (requests, errors), Gauge for values that change up and down (queue depth, memory), Histogram for distributions (latency, payload size). Expose /metrics and let Prometheus scrape. For multi-process workers, use the PROMETHEUS_MULTIPROC_DIR mode.

OpenTelemetry (Day 068) is the modern, vendor- neutral path. prometheus_client is the older, simpler one — and Prometheus is still the most-deployed metrics backend on Kubernetes. Knowing both is worth it.

The four metric types

pythonfrom prometheus_client import Counter, Gauge, Histogram

# Counter — monotonically increasing
requests_total = Counter(
    "http_requests_total",
    "Total HTTP requests",
    ["method", "status"],
)
requests_total.labels(method="GET", status="200").inc()

# Gauge — value that goes up and down
queue_depth = Gauge("queue_depth", "Items waiting")
queue_depth.set(42)

# Histogram — distribution of values, with quantile rollups
latency = Histogram(
    "http_request_duration_seconds",
    "Request latency",
    ["endpoint"],
    buckets=(0.005, 0.01, 0.025, 0.05, 0.1, 0.25,
             0.5, 1, 2.5, 5, 10),
)
latency.labels(endpoint="/devices").observe(0.042)

Histograms with custom buckets are nearly always better than the default ones — pick buckets that match your actual SLOs. The defaults (powers of 2 over a few seconds) often miss meaningful resolution.

The naming conventions

  • Counters end in _total — requests_total, not just requests.
  • Histograms include the unit — _seconds, _bytes. Helps PromQL stay readable.
  • Labels for dimensions, not for values — method=GET good (low cardinality), user_id=12345 bad (cardinality explosion).

Exposing /metrics

pythonfrom prometheus_client import make_asgi_app

# Mount /metrics on a FastAPI app
metrics_app = make_asgi_app()
app.mount("/metrics", metrics_app)

Configure Prometheus to scrape http://app:8000/metrics on the default 15s interval. That's the whole pipeline.

The multiprocess gotcha

Each worker process has its own metric counters. With --workers 4, hitting /metrics only shows the metrics for whichever worker handled that request. Numbers jump around as you refresh.

The fix: multiprocess mode, which writes metrics to a shared directory and aggregates on read. 

bashexport PROMETHEUS_MULTIPROC_DIR=/tmp/prom_metrics
mkdir -p $PROMETHEUS_MULTIPROC_DIR
gunicorn -k uvicorn.workers.UvicornWorker --workers 4 app:app
pythonfrom prometheus_client import CollectorRegistry, multiprocess, make_asgi_app

def metrics_endpoint():
    registry = CollectorRegistry()
    multiprocess.MultiProcessCollector(registry)
    return make_asgi_app(registry=registry)

app.mount("/metrics", metrics_endpoint())

With multiprocess mode, Gauge behavior changes — check the docs for which modes (min, max, livesum, all) make sense for each of your gauges.

Stay away from Summary

Prometheus has a fourth type, Summary, which computes quantiles client-side. The per-process quantiles can't be aggregated across workers correctly, which makes them misleading in any multi-instance deployment. Use Histogram with histogram_quantile in PromQL instead — aggregates correctly.

For network-engineering use, a Counter for poll attempts and a Histogram for poll latency, labelled by device role, gives you everything you need for "p99 SSH latency by site type" dashboards in Grafana.

Add three metrics to a service: a Counter for requests-by-status, a Histogram for end-to-end latency, and a Gauge for whatever counts you have in flight. Expose /metrics. Curl it. The output is dense but readable; you're now a Grafana dashboard away from observability.
