respx: mocking httpx without breaking httpx

Mock HTTP responses for tests. Stub by URL pattern, return canned bodies, assert which calls were made — without a real server.

22 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

For httpx-based code, use respx to intercept requests and return canned responses. For requests-based code, use responses. For either, prefer URL-pattern stubs over heavy unittest.mock patches — you're testing your code, not the HTTP library.

Tests that hit real HTTP endpoints are slow, flaky, and require credentials. Tests that mock with unittest.mock bypass the HTTP library entirely — you stop testing URL construction, headers, and serialization. respx is the middle path: real httpx, fake network.

Basic stubbing

pythonimport httpx, respx

@respx.mock
def test_fetches_devices():
    respx.get("https://api.example.com/devices").mock(
        return_value=httpx.Response(200, json=[
            {"hostname": "sw-01", "role": "access"},
        ])
    )

    result = my_function_that_calls_the_api()
    assert len(result) == 1

The decorator activates the mock for the test; any httpx call to that URL gets the stubbed response. Other URLs would raise respx.MockResponseNotFoundError — tests fail loud if you forgot to mock a call.

Async, the same way

respx works with both sync and async httpx clients. No code changes needed. 

python@respx.mock
async def test_async_fetch():
    respx.get("https://api.example.com/health").mock(
        return_value=httpx.Response(200, json={"status": "ok"}),
    )
    async with httpx.AsyncClient() as c:
        r = await c.get("https://api.example.com/health")
        assert r.json() == {"status": "ok"}

URL patterns

For paths with IDs you don't want to enumerate, use a regex. 

pythonimport re

respx.get(url__regex=r"https://api\.example\.com/devices/\d+").mock(
    return_value=httpx.Response(200, json={"hostname": "sw-X"}),
)

Side-effect responses

For dynamic responses (different per call, based on request body, etc.), pass a callable. 

pythondef handler(request):
    payload = json.loads(request.content)
    return httpx.Response(201, json={"id": 42, **payload})

respx.post("https://api.example.com/devices").mock(side_effect=handler)

Useful for echoing back submitted data with an ID, or returning different status codes based on input shape.

Asserting calls were made

python@respx.mock
def test_sends_metric():
    route = respx.post("https://metrics.local/v1/ingest").mock(
        return_value=httpx.Response(204),
    )

    my_code_that_should_send_a_metric()

    assert route.called
    assert route.call_count == 1
    request = route.calls.last.request
    assert json.loads(request.content)["value"] == 42

Pass-through for partial mocking

Sometimes you want to mock one endpoint but let others hit the real server. pass_through does that. 

pythonrespx.get("https://api.example.com/v1/special").mock(
    return_value=httpx.Response(200, json={"stub": True}),
)
# Everything else hits the real network
respx.route().pass_through()

Mostly useful in integration-style tests where only one slow/flaky upstream needs stubbing.

The requests-library equivalent: responses

For codebases still using requests, the equivalent library is responses: 

pythonimport responses, requests

@responses.activate
def test_legacy():
    responses.add(responses.GET, "https://api.example.com/x",
                  json={"ok": True})

    assert requests.get("https://api.example.com/x").json() == {"ok": True}

Same idea, slightly different API. Pick by your HTTP library.

VCR: record real responses, replay forever

For tests against APIs you don't control: VCR records the real HTTP call once, plays back the recording on subsequent runs. pytest-vcr handles the integration. 

python@pytest.mark.vcr
def test_real_api():
    # First run: hits real API, records response
    # Subsequent runs: replays recording, no network
    result = my_api_call()
    assert result["status"] == "ok"

Cassette files (YAML) live in your repo. Watch for accidentally recording auth tokens — VCR has filters to scrub sensitive headers.

Don't mock at the unittest.mock.patch("httpx.get") level. You stop testing what you actually send and start testing your mocks. Stub at the URL level (respx) so the HTTP library is still in the loop.

Find a test that hits a real API or that uses heavy mock.patch on httpx internals. Convert to respx. The test becomes faster and clearer about what it's asserting (URL, method, body), and you can assert on outbound headers (Authorization, custom IDs) you couldn't before.
