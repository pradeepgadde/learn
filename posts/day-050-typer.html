typer: argparse without the boilerplate

Type-annotated functions become rich CLIs. Sub-commands, validation, shell completion, colored help — all derived from the function signature.

19 Feb 2026 • ~3 min read • Python 3.7+
TL;DR

Decorate a function with @app.command(), annotate the parameters with types and defaults, run app(). typer handles parsing, validation, and help generation. Sub-commands, options, and flags all come from the signature.

We covered argparse on Day 014 — the standard-library answer. typer (built on the same ideas as click) gives you the same CLI for half the code, plus rich features like shell completion and pretty --help output. Same author as FastAPI; same design philosophy.

The minimal CLI

pythonimport typer

app = typer.Typer()

@app.command()
def probe(host: str, port: int = 22, verbose: bool = False):
    """Connect to HOST and report status."""
    if verbose:
        typer.echo(f"connecting to {host}:{port}...")
    # do the thing

if __name__ == "__main__":
    app()

That gives you probe HOST [--port 22] [--verbose], full --help, type validation that rejects --port abc, and a --install-completion flag for free. The function signature is the CLI.

Sub-commands

Add more @app.command() functions and each becomes a sub-command. Like git add, git commit, git push — the same pattern. 

pythonapp = typer.Typer()

@app.command()
def deploy(env: str, version: str):
    """Deploy VERSION to ENV."""
    ...

@app.command()
def rollback(env: str):
    """Roll back the last deploy in ENV."""
    ...

# Usage:
#   mytool deploy staging v1.2.3
#   mytool rollback staging
#   mytool --help          (lists all sub-commands)

Rich options with typer.Option

For options that need a short alias, a help string, a constraint, or env-var fallback, wrap the default in typer.Option(...). The function signature still reads naturally. 

pythonfrom typing import Annotated

@app.command()
def deploy(
    env: Annotated[str, typer.Argument(help="Target env")],
    version: Annotated[str, typer.Option("--version", "-v",
                                       envvar="DEPLOY_VERSION")],
    dry_run: Annotated[bool, typer.Option("--dry-run",
                                       help="Plan only")] = False,
):
    ...

Annotated[T, ...] is the modern way to attach metadata to types. Both typer and pydantic v2 use it. Old-style defaults (x: str = typer.Option(...)) also work, but mypy is happier with Annotated.

Choices via Enum

Constrain to a fixed set of values? Use a string Enum — typer renders it as a constrained argument with auto-completion. 

pythonfrom enum import StrEnum

class Env(StrEnum):
    DEV     = "dev"
    STAGING = "staging"
    PROD    = "prod"

@app.command()
def deploy(env: Env):
    ...

# mytool deploy banana  -->  Error: Invalid value 'banana'.
#                              Must be one of: dev, staging, prod

Confirmation prompts and password input

Built-in helpers for the things every CLI eventually needs. 

python@app.command()
def delete(name: str):
    typer.confirm(f"Really delete {name}?", abort=True)
    # aborts the program if the user says no

@app.command()
def login(user: str):
    password = typer.prompt("Password", hide_input=True)
    ...

Pairs perfectly with rich

Since typer uses the rich library internally for help formatting, your colored output from Day 031 fits in seamlessly — tables, progress bars, panels — all play well together.

For a single-script with one command, plain argparse (Day 014) is fine and ships with Python — no extra dependency. As soon as you have two sub-commands or want --install-completion, jump to typer. The migration is straightforward enough that starting with argparse and switching later is a valid path too.

Take any argparse script you've written. Convert it to typer. Count the lines you deleted — usually 30–50% reduction with no functionality lost. Add typer install-completion and tab-complete in your shell for the first time.
