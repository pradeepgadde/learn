contextvars: async-safe globals

threading.local broke with asyncio. contextvars fixes it. Per-task storage for request IDs, user context, and trace info — without threading them through every function call.

29 Mar 2026 • ~3 min read • Python 3.7+
TL;DR

Declare a ContextVar at module level. Set it at the entry point of each request/task. Read it anywhere in the call stack without passing it as a parameter. Each asyncio task gets its own copy; changes in one task don't leak to another. Think "thread-local, but for coroutines".

Two patterns appear in every non-trivial async codebase: "I need this request's ID to reach the logger" and "I need the current user to reach the permission check, five calls deep." Passing both through every function signature is noisy. A global is shared across concurrent tasks and is wrong. contextvars is the right answer.

The basics

pythonfrom contextvars import ContextVar

# Declare at module level
request_id: ContextVar[str] = ContextVar("request_id", default="")
current_user: ContextVar[str] = ContextVar("current_user")

# Set a value; returns a Token you can use to reset later
token = request_id.set("abc-123")

# Read it from anywhere in the call chain
rid = request_id.get()          # "abc-123"

# Restore to the previous value (optional)
request_id.reset(token)

Real use: request ID through a FastAPI app

python# vars.py
from contextvars import ContextVar
request_id: ContextVar[str] = ContextVar("request_id", default="")

# middleware.py
from starlette.middleware.base import BaseHTTPMiddleware
import uuid
from .vars import request_id

class RequestIDMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        rid = request.headers.get("X-Request-ID", str(uuid.uuid4()))
        token = request_id.set(rid)
        try:
            response = await call_next(request)
        finally:
            request_id.reset(token)
        response.headers["X-Request-ID"] = rid
        return response

# db.py — deep in the call stack, no rid parameter needed
from .vars import request_id
import structlog

log = structlog.get_logger()

async def fetch_device(device_id: int):
    log.info("fetch_device", rid=request_id.get(), id=device_id)
    ...

Every log line from every function in this request emits the same rid — without it being passed through the router, the service layer, and the repository.

Why not threading.local?

pythonimport threading
_local = threading.local()

async def handler_a():
    _local.rid = "aaa"
    await asyncio.sleep(0)   # yields control
    # _local.rid might be "bbb" now — another coroutine ran on this thread

asyncio runs many coroutines on a single thread. threading.local is per-thread, so all concurrent tasks on the same thread share the same storage — data leaks between requests. ContextVar is per-task: each asyncio task (and its children) has its own context copy.

Child tasks inherit, but don't share

pythonimport asyncio
from contextvars import ContextVar

var: ContextVar[int] = ContextVar("var", default=0)

async def child():
    print(var.get())         # sees parent's value: 42
    var.set(99)             # only changes child's copy

async def parent():
    var.set(42)
    await asyncio.create_task(child())
    print(var.get())         # still 42 — child's write didn't propagate back

A new task inherits a copy of the context at the moment of creation. Mutations in the child don't flow back up. This is exactly the right semantics for request isolation.

Structlog integration

structlog (Day 034) has first-class support for context vars. Bind values to the context once, have them appear on every subsequent log line automatically. 

pythonimport structlog

# In middleware:
structlog.contextvars.clear_contextvars()
structlog.contextvars.bind_contextvars(request_id=rid)

# Anywhere else in the request — no extra parameters:
log.info("processing")
# {"event": "processing", "request_id": "abc-123", ...}

Sync code too

contextvars works fine in sync Python — each thread gets its own context. For thread pools called from async code, the event loop copies the context into the thread automatically when using loop.run_in_executor.

A common footgun: setting a ContextVar in a @pytest.fixture without resetting it. The next test runs with leftover context. Reset in the fixture's teardown, or use contextvars.copy_context().run(...) to sandbox the test in a fresh context.

Add a request-ID context var to one of your FastAPI apps via middleware. Verify that two concurrent requests each log their own ID and never each other's. Then remove all the rid parameters that previously threaded through your service layer. The diff is always satisfying.
