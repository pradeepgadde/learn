JSON and YAML without the footguns

Three short rules for parsing config files: pretty-print on write, validate on read, never use yaml.load.

15 Jan 2026 • ~3 min read • Python 3.11+ for tomllib
TL;DR

For JSON, always pass indent=2, sort_keys=True when writing for humans. For YAML, always use yaml.safe_load — yaml.load can execute arbitrary Python. For config in modern projects, prefer TOML and the built-in tomllib (3.11+).

Three formats handle 99% of config-file work in Python land: JSON for machine-to-machine, YAML for ops-friendly hand editing, TOML for project metadata. Each has a quiet trap that bites people once and never again.

JSON for humans

json.dumps defaults to a single line with no spacing. Fine for the wire, awful for diffs and code review. 

pythonimport json

config = {"port": 8080, "host": "0.0.0.0", "tags": ["prod", "v2"]}

# For humans
print(json.dumps(config, indent=2, sort_keys=True))
{ "host": "0.0.0.0", "port": 8080, "tags": [ "prod", "v2" ] }

sort_keys=True is the secret sauce: any two equivalent configs produce identical output, so git diff shows real changes only.

The default= escape hatch

JSON doesn't know about datetime, Path, set, or your dataclass. The default callback handles whatever the encoder doesn't. 

pythonfrom datetime import datetime
from pathlib import Path

def to_json(obj):
    if isinstance(obj, datetime): return obj.isoformat()
    if isinstance(obj, Path):     return str(obj)
    if isinstance(obj, set):      return sorted(obj)
    raise TypeError(f"can't serialize {type(obj).__name__}")

payload = {"when": datetime.utcnow(), "path": Path("/etc/app")}
print(json.dumps(payload, default=to_json))

YAML: safe_load, always

PyYAML's load can construct arbitrary Python objects, including ones that execute code. A malicious YAML file can run shell commands. safe_load only produces basic types (dict, list, str, int, float, bool, None). 

pythonimport yaml

# NEVER do this on untrusted input:
# yaml.load(open("config.yml"), Loader=yaml.Loader)

# Always:
with open("config.yml") as f:
    config = yaml.safe_load(f)

# And to write:
with open("config.yml", "w") as f:
    yaml.safe_dump(config, f, sort_keys=False, default_flow_style=False)

default_flow_style=False forces block style (one item per line) instead of inline brackets. Much more readable for nested configs.

TOML for project config

Since Python 3.11, parsing TOML is built in. No third-party library, no pip install. This is the format pyproject.toml uses, and increasingly the format for app config. 

pythonimport tomllib     # 3.11+; on older versions: import tomli as tomllib

with open("pyproject.toml", "rb") as f:
    data = tomllib.load(f)

print(data["project"]["name"])

Note the "rb" mode — tomllib wants bytes, not text. It's deliberately read-only; for writing, use tomli-w.

YAML's "Norway problem": the unquoted string NO parses as False. Country code for Norway. ON, OFF, and YES have similar issues. When in doubt, quote your strings, or use TOML, where there are no surprises.

Find a config file in your project. If it's YAML, grep your codebase for yaml.load — replace any with safe_load. If it's JSON written by your code, add indent=2, sort_keys=True. Commit. The next diff will be dramatically clearer.
