sqlite3: the database in your stdlib

A real SQL database, ACID transactions, JSON support, no server — all built into Python. Replaces a dozen flat-file caches the moment your data has structure.

06 Feb 2026 • ~3 min read • Python 3.0+
TL;DR

sqlite3 is a complete SQL database in the standard library — no install, no server. Use parameterized queries (?) always, never f-strings. Use row_factory = sqlite3.Row for named column access. Wrap writes in with conn: for automatic transactions. Good for caches, scratch data, single-user CLIs, and millions of rows.

SQLite is the most-deployed database engine in the world. It runs in your browser, on every phone, in every .app bundle. The Python binding ships with the standard library — you can use it right now with no install.

The minimal usage

pythonimport sqlite3

# File path, or ":memory:" for an in-process database
conn = sqlite3.connect("cache.db")
conn.row_factory = sqlite3.Row     # dict-like row access

conn.execute("""
    CREATE TABLE IF NOT EXISTS devices (
        hostname TEXT PRIMARY KEY,
        ip       TEXT NOT NULL,
        seen_at  TEXT DEFAULT CURRENT_TIMESTAMP
    )
""")

# with-block = automatic transaction (commit on exit, rollback on exception)
with conn:
    conn.execute(
        "INSERT OR REPLACE INTO devices (hostname, ip) VALUES (?, ?)",
        ("sw-01", "10.0.0.1"),
    )

for row in conn.execute("SELECT * FROM devices"):
    print(row["hostname"], row["ip"])

Parameterized queries, always

Never use string formatting to build SQL. The ? placeholder is safe; the f-string is a bug factory. 

python# NEVER — SQL injection waiting to happen
conn.execute(f"SELECT * FROM devices WHERE hostname = '{name}'")

# Always — parameter passed safely, types preserved
conn.execute("SELECT * FROM devices WHERE hostname = ?", (name,))

Note the trailing comma — (name,) is a one-element tuple, (name) is just name in parentheses. SQLite needs a sequence.

Named placeholders for many params

pythonconn.execute(
    "INSERT INTO devices (hostname, ip, role) "
    "VALUES (:host, :ip, :role)",
    {"host": "sw-02", "ip": "10.0.0.2", "role": "core"},
)

Easier to read than counting positional args. Pairs beautifully with dataclass instances: conn.execute(sql, dataclasses.asdict(device)).

Bulk inserts: executemany

For loading many rows, executemany is dramatically faster than calling execute in a loop — and even faster inside a single transaction. 

pythonrows = [(d.hostname, d.ip) for d in devices]

with conn:                                  # single transaction
    conn.executemany(
        "INSERT OR IGNORE INTO devices (hostname, ip) VALUES (?, ?)",
        rows,
    )

100,000 inserts in a single transaction completes in well under a second on commodity hardware. The same inserts auto-committed individually take minutes.

JSON columns

SQLite has built-in JSON functions since 3.9 (every modern Python). Store nested data without an ORM, query it with SQL. 

pythonimport json

conn.execute("CREATE TABLE events (id INTEGER PRIMARY KEY, data TEXT)")

payload = {"type": "login", "user": "pradeep", "tags": ["admin"]}
conn.execute("INSERT INTO events (data) VALUES (?)",
             (json.dumps(payload),))

# Query JSON fields directly with SQL
for row in conn.execute("""
    SELECT id, json_extract(data, '$.user') AS user
    FROM events
    WHERE json_extract(data, '$.type') = 'login'
"""):
    print(row["id"], row["user"])

WAL mode for concurrent readers

For any database touched by more than one process — a CLI plus a service, parallel scripts — switch to Write-Ahead Logging. Readers no longer block writers and vice versa. 

pythonconn.execute("PRAGMA journal_mode=WAL")
conn.execute("PRAGMA synchronous=NORMAL")   # still safe, much faster

When SQLite isn't right. Multiple writers contending heavily; clients on different machines (no network protocol — SQLite is a file); strict per-row access control. For everything else — analytics scratchpads, app caches, embedded data, single-user web apps, even small SaaS tenants — it's the right tool.

Pick a script that maintains a JSON or pickle "cache file" on disk. Replace it with SQLite. You'll get atomic updates, indexed lookups, and the ability to query the cache with sqlite3 cache.db from your terminal — all for the same lines of code.
