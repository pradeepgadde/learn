HTTP with httpx, the requests killer

Same API as requests, plus the things requests never grew up to support: async, HTTP/2, and a sane timeout default.

17 Jan 2026 • ~3 min read • Python 3.8+
TL;DR

httpx is a drop-in replacement for requests with explicit timeouts, real connection pooling via Client(), optional async, and HTTP/2. Use a Client as a context manager for any script that makes more than one request.

requests defined ergonomic Python HTTP for a decade. httpx is what it would look like if it were designed today — same friendly API, none of the legacy gotchas.

The familiar API

pythonimport httpx

r = httpx.get("https://api.github.com/users/pradeepgadde")
r.raise_for_status()
data = r.json()
print(data["public_repos"])

If you know requests, you know httpx. get, post, put, delete, r.json(), r.text, r.status_code — all there, all behave the same.

Use a Client for multiple requests

Each top-level httpx.get() opens a new connection. That's wasteful when you're hitting the same host repeatedly. A Client keeps connections alive between requests. 

pythonwith httpx.Client(base_url="https://api.github.com", timeout=10.0) as client:
    for user in users:
        r = client.get(f"/users/{user}")
        r.raise_for_status()
        print(user, r.json()["public_repos"])

Three things to notice. The Client is a context manager so connections close cleanly. base_url means you write paths, not full URLs. timeout applies to every request automatically.

Timeouts are not optional

Where requests defaults to no timeout (your script can hang forever), httpx defaults to 5 seconds. Override it explicitly so the value is in your code, not in a library default that might change. 

python# Single overall timeout
httpx.get(url, timeout=10.0)

# Granular control: connect, read, write, pool
timeout = httpx.Timeout(10.0, connect=3.0)
httpx.get(url, timeout=timeout)

# Explicitly disable (almost never the right answer)
httpx.get(url, timeout=None)

Going async, when it pays off

For one or two requests, sync is fine. For dozens or hundreds of independent requests — pinging a fleet of devices, scraping pages, fanning out to APIs — the async client is dramatically faster. 

pythonimport asyncio, httpx

async def fetch(client, user):
    r = await client.get(f"/users/{user}")
    return user, r.json()["public_repos"]

async def main(users):
    async with httpx.AsyncClient(base_url="https://api.github.com") as c:
        return await asyncio.gather(*(fetch(c, u) for u in users))

results = asyncio.run(main(["torvalds", "gvanrossum", "pradeepgadde"]))

Same get/post API, just AsyncClient and await. Twenty requests in parallel take roughly as long as one.

The friendly extras

Two things that hide under the surface but are worth knowing:

  • httpx.get(url, params={"q": "python"}) — URL-encodes for you, handles list values correctly.
  • client.headers["Authorization"] = "Bearer ..." on the client applies to every request from that client. Set once.

Already on requests? Don't migrate for the sake of it. Reach for httpx when you start a new project, when you need async, when you need HTTP/2, or when a flaky service makes you wish for proper timeouts.

Find a script that hits a single host more than three times using individual requests.get calls. Wrap them in a Client with base_url and a timeout. Time the before and after — you'll often see a 2–5x speedup from connection reuse alone.
