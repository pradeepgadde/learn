asyncio.gather and the patterns that scale

Fan out a thousand requests on one event loop. Three primitives — gather, TaskGroup, and Semaphore — cover most real async work.

08 Feb 2026 • ~3 min read • Python 3.11+ for TaskGroup
TL;DR

Use asyncio.gather(*coros, return_exceptions=True) for a fan-out where individual failures shouldn't crash the rest. On 3.11+, prefer TaskGroup — it cancels siblings on failure, which is usually safer. Cap concurrency with a Semaphore so you don't overwhelm the upstream.

asyncio gets a reputation for being hard. Most of that difficulty comes from people learning every primitive at once. In practice three constructs cover almost all application code — the rest is for library authors.

The fan-out: gather

pythonimport asyncio, httpx

async def fetch(client, url):
    r = await client.get(url, timeout=5)
    return url, r.status_code

async def main(urls):
    async with httpx.AsyncClient() as c:
        results = await asyncio.gather(
            *(fetch(c, u) for u in urls),
            return_exceptions=True,
        )
    for r in results:
        print(r)

asyncio.run(main(urls))

return_exceptions=True is critical for fan-out. Without it, the first failure cancels the others and you lose all in-flight results. With it, exceptions become return values you can iterate over.

The 3.11 upgrade: TaskGroup

TaskGroup is structured concurrency — the async equivalent of a function returning only after all its work is done. If any task raises, siblings are cancelled and the original exception propagates. 

pythonasync def main(urls):
    async with httpx.AsyncClient() as c:
        async with asyncio.TaskGroup() as tg:
            tasks = [tg.create_task(fetch(c, u)) for u in urls]
        # All tasks done here. Results in t.result() for each task.
        return [t.result() for t in tasks]

When to prefer TaskGroup over gather: if a partial result is useless (you need all of them or none), TaskGroup is safer because it doesn't leave orphan tasks running on failure. gather(return_exceptions=True) is right when you genuinely want partial success.

Bound concurrency: Semaphore

Launching 10,000 concurrent connections will exhaust file descriptors and crash the upstream. Cap concurrent in-flight tasks with a semaphore. 

pythonasync def bounded_fetch(sem, client, url):
    async with sem:
        return await fetch(client, url)

async def main(urls):
    sem = asyncio.Semaphore(50)
    async with httpx.AsyncClient() as c:
        return await asyncio.gather(
            *(bounded_fetch(sem, c, u) for u in urls),
            return_exceptions=True,
        )

At any moment, at most 50 fetches are in-flight; the rest are queued, waiting on the semaphore. Submit 5000 tasks, the system handles them in waves of 50.

Per-task timeouts: asyncio.timeout

Since 3.11, asyncio.timeout is a context manager that cancels its body if it exceeds the limit. Cleaner than wait_for for nested code. 

pythonasync def probe(host):
    async with asyncio.timeout(5):
        async with asyncssh.connect(host) as c:
            return await c.run("show version")

Don't block the loop

The cardinal asyncio rule: never call a synchronous blocking function from a coroutine. time.sleep, requests.get, open(...).read() on a slow disk — they freeze the entire event loop and every other task on it. 

python# Use the async equivalents
await asyncio.sleep(1)              # not time.sleep
await httpx_async_client.get(url)   # not requests.get

# For unavoidably-blocking calls (a sync library), offload to a thread
result = await asyncio.to_thread(slow_sync_func, arg)

When in doubt: I/O-bound and high concurrency? asyncio. I/O-bound and modest concurrency (under 100)? ThreadPoolExecutor is simpler. CPU-bound? Neither — use ProcessPoolExecutor (Day 038).

Build a tiny URL checker. Take a list of URLs, fan out with gather + return_exceptions=True, cap at 25 concurrent with a Semaphore, set a 10-second per-request timeout. Print one line per URL: status, time, or the error class. You've just built a useful production tool.
