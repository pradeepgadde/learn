Generators: lazy by default

Replace one keyword and your function streams instead of accumulating. The difference between OOM-ing on a 10 GB log and processing it in 50 MB.

20 Jan 2026 • ~3 min read • Python 3.0+
TL;DR

Replace return some_list with a loop of yields and your function becomes a generator — items are produced one at a time, on demand. Memory stays flat regardless of input size. Use generator expressions (x for x in ...) for inline cases, and yield from to delegate.

The single biggest win in Python's design is treating iteration as a first-class concept. Generators let any function participate in that protocol, which means you can process arbitrarily large data sources without holding them in memory.

The two-character refactor

python# Eager: builds the whole list before returning
def errors(log):
    out = []
    for line in log:
        if "ERROR" in line:
            out.append(line)
    return out

# Lazy: yields one line at a time, ever-bounded memory
def errors(log):
    for line in log:
        if "ERROR" in line:
            yield line

The caller iterates either the same way: for e in errors(open("app.log")): .... The difference is that the eager version reads the whole 10 GB file before you see a single line; the lazy version starts producing immediately and never holds more than one line at a time.

Generator expressions

For one-off generators, use generator expressions — same syntax as a list comprehension but with parentheses. 

python# List comprehension: builds the full list
squares = [x * x for x in range(10_000_000)]   # ~80 MB

# Generator expression: produces on demand
squares = (x * x for x in range(10_000_000))   # a few hundred bytes

total = sum(squares)        # Streams through, never materializes

When you pass a generator to sum, min, max, any, all, or "".join, you can drop the outer brackets entirely: 

python# No need for [ ] inside sum()
total = sum(x * x for x in values)
has_error = any("ERROR" in line for line in log)

Composing pipelines

Generators chain. The output of one feeds the next, and nothing materializes until you iterate the end of the chain. This is the Python equivalent of Unix pipes. 

pythondef read_lines(path):
    with open(path) as f:
        yield from f

def non_blank(lines):
    return (l.rstrip() for l in lines if l.strip())

def parsed(lines):
    return (json.loads(l) for l in lines)

# Composes top-to-bottom, processes one record at a time
for record in parsed(non_blank(read_lines("events.ndjson"))):
    handle(record)

yield from for delegation

When a generator wants to yield everything from another iterable, write yield from instead of a manual loop. 

pythondef walk_files(root):
    for subdir in root.iterdir():
        if subdir.is_dir():
            yield from walk_files(subdir)
        else:
            yield subdir

Without yield from you'd need a nested for loop with another yield — more typing, same result.

One caveat: generators are single-use. Once you've iterated through one, it's exhausted — iterating again gives you zero items. If you need to consume the same data twice, materialize it into a list first, or wrap the generator in a function so each call starts fresh.

Find a function in your code that builds a list and returns it. If callers only iterate the result once, delete the list, replace append with yield, and remove the return. Run your tests — they should still pass, and your function now scales.
