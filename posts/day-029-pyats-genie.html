pyATS & Genie: structured network testing

A testbed YAML, parsers for thousands of show commands, and a real testing framework for the network itself. Originally Cisco-internal, now open source.

29 Jan 2026 • ~3 min read • Python 3.8+
TL;DR

pyATS is a testing framework for networks; Genie is its library of structured parsers and APIs. Define your devices in a testbed.yaml, connect, run any show command, and get back a normalized dict. Combine with pytest-style assertions for proper network validation.

Where TextFSM gives you parsed records and netmiko gives you connectivity, pyATS gives you the framework around all of it — testbed inventories, structured parsers, and assertion libraries. It's overkill for a quick script, but it's the right scale for "automate validation of every change before and after."

The testbed file

A testbed YAML is the inventory: devices, credentials, connection parameters. One file, version-controlled, the source of truth. 

yamltestbed:
  name: lab1

devices:
  core-sw-01:
    os: iosxe
    type: switch
    connections:
      cli:
        protocol: ssh
        ip: 10.1.0.1
    credentials:
      default:
        username: netadmin
        password: "%ENV{NET_PASS}"
      enable:
        password: "%ENV{NET_ENABLE}"

Note %ENV{...} — pyATS resolves environment variables so you don't commit secrets. Other useful sources: %ASK{} for interactive prompts, %CALLABLE{module.func} for vault integrations.

Connecting and running a command

pythonfrom genie.testbed import load

tb = load("testbed.yaml")
dev = tb.devices["core-sw-01"]
dev.connect(log_stdout=False)

# Raw output, like netmiko
raw      = dev.execute("show version")

# Structured output — a dict, parsed by Genie
parsed   = dev.parse("show interfaces")
print(parsed["GigabitEthernet0/1"]["oper_status"])
# 'up'

dev.disconnect()

parse uses Genie's library — thousands of templates covering Cisco platforms (IOS, IOS-XE, IOS-XR, NX-OS), with growing coverage of Junos and others. The return shape is normalized across versions; an interface is always at parsed[name]["oper_status"] whether the device is a Catalyst or an ASR.

Snapshot diffs: the killer feature

pyATS can take a "before" snapshot of arbitrary commands, run a change, take an "after" snapshot, and tell you exactly what differs in the structured output. This is the change-validation workflow you've always wanted. 

pythonfrom genie.utils.diff import Diff

before = dev.parse("show ip route")
# ... apply change ...
after  = dev.parse("show ip route")

d = Diff(before, after)
d.findDiff()
print(d)
# Shows added, removed, and changed routes — not text diff, structural diff

Structural diff means renames don't matter, key order doesn't matter, and you see only what actually changed. No more squinting at diff -u on raw command output.

The test framework: AEtest

pyATS includes a pytest-shaped test runner specifically designed for network test cases. Triggers, verifications, cleanup — all first-class concepts. 

pythonfrom pyats import aetest

class CheckLinks(aetest.Testcase):

    @aetest.setup
    def connect(self, testbed):
        testbed.devices["core-sw-01"].connect(log_stdout=False)

    @aetest.test
    def all_uplinks_up(self, testbed):
        parsed = testbed.devices["core-sw-01"].parse("show interfaces")
        for name, data in parsed.items():
            if name.startswith("TenGig"):
                assert data["oper_status"] == "up", name

Run with pyats run job ... and you get a report: pass/fail per test, structured logs per device, the whole audit trail.

pyATS is heavy — large install, many dependencies, opinionated structure. Worth it for ongoing validation frameworks; overkill for a one-off script. For one-off parsing, stick to ntc-templates from Day 028.

Set up a testbed YAML for one device. In ten lines, connect, take a snapshot of show ip route, wait 5 seconds, take another snapshot, and run Diff. The output is exactly what your network looks like through a change window.
