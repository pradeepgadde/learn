itertools.batched, finally

Chunking an iterable into fixed-size groups was a recipe for twenty years. In Python 3.12 it became a built-in.

10 May 2026 • ~3 min read • Python 3.12+
TL;DR

itertools.batched(iterable, n) yields tuples of size n, with the final tuple possibly shorter. Works on any iterable (including generators), no slicing, no index math. Add strict=True on Python 3.13+ to require exact-size groups.

For two decades, every Python codebase had its own chunked(), grouper(), or batch() helper. They all did the same thing slightly differently. itertools.batched ends the debate.

The old slicing approach

python# Works only on sequences (lists, tuples, strings)
def chunks(seq, n):
    for i in range(0, len(seq), n):
        yield seq[i:i + n]

# Breaks on a generator: needs len() and indexing

The new built-in

pythonfrom itertools import batched

for group in batched("ABCDEFGHIJ", 3):
    print(group)
('A', 'B', 'C') ('A', 'B', 'C') ('D', 'E', 'F') ('G', 'H', 'I') ('J',)

Note the last tuple is short. That's the right default behavior for almost all uses — you don't lose data.

Where it actually shines: generators

Slicing requires a sequence. batched takes any iterable, which means it works on file iterators, database cursors, and other generators where you couldn't have used the slicing trick. 

pythondef read_records(path):
    with open(path) as f:
        for line in f:
            yield json.loads(line)

# Insert into the DB 500 at a time, never holding all of it
for batch in batched(read_records("events.ndjson"), 500):
    db.insert_many(batch)

The whole pipeline stays lazy. Memory usage is bounded by the batch size, not the file size.

The classic API-paging use case

python# API endpoint accepts up to 100 IDs per request
user_ids = get_all_user_ids()     # could be a generator

for id_batch in batched(user_ids, 100):
    response = api.get_users(ids=list(id_batch))
    handle(response)

Strict mode (3.13+)

Sometimes a short final tuple is a bug, not a feature — you're processing fixed-format records and a partial group means corrupted input. Pass strict=True and batched raises ValueError instead of yielding the runt. 

pythonfrom itertools import batched

# Each line should have exactly 4 RGBA values
for pixel in batched(stream, 4, strict=True):
    paint(*pixel)

On Python 3.11 or earlier, the closest equivalent is the more-itertools third-party library, which has had chunked() for years. Worth pulling in if you can't move to 3.12 yet.

Find your codebase's local chunked / batch / grouper helper — almost every project has one. Replace it with itertools.batched and delete the helper. Update the import, run the tests, ship it.
