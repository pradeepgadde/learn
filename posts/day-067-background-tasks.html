Background tasks: FastAPI vs Celery vs RQ

Three options for "do this work after the response": BackgroundTasks for fire-and-forget, RQ for simple persistent queues, Celery for the heavy stuff.

08 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

FastAPI BackgroundTasks for tasks that fit "fire and forget, in this same process, and we don't really care if it's lost on crash." RQ for "we want a real queue but want it simple." Celery for "many workers, retries, schedules, chains, ten years of stack overflow answers." Pick the smallest tool that fits.

"Send the response now; do this slow thing afterwards." Three patterns, big differences in durability and complexity.

Tier 1: FastAPI BackgroundTasks

pythonfrom fastapi import FastAPI, BackgroundTasks

app = FastAPI()

def send_welcome_email(to: str):
    # slow SMTP call
    ...

@app.post("/signup")
def signup(email: str, tasks: BackgroundTasks):
    create_user(email)
    tasks.add_task(send_welcome_email, email)
    return {"ok": True}        # returns immediately

The task runs after the response is sent, in the same worker process. Use it when:

  • The task is short (seconds, not minutes).
  • It's OK if the task is lost when the process restarts mid-task.
  • You don't need retries, scheduling, or visibility into queue depth.

For "send a webhook notification" or "log to an audit table" — perfect. For "process this upload" — you want a real queue.

Tier 2: RQ (Redis Queue)

RQ runs jobs in separate worker processes, backed by Redis. Crashes don't lose jobs. Failed jobs go to a failure queue you can inspect. 

python# tasks.py
def resize_image(path: str):
    ...

# In your FastAPI app
from redis     import Redis
from rq        import Queue
from tasks     import resize_image

q = Queue(connection=Redis.from_url("redis://localhost"))

@app.post("/upload")
def upload(path: str):
    job = q.enqueue(resize_image, path)
    return {"job_id": job.id}
bash# Run a worker in a separate process
rq worker

That's it. Add scheduling with rq-scheduler, observability with rq-dashboard. The whole stack fits in a few hours. Right call when "Celery is too much" but you need a real queue.

Tier 3: Celery

Celery is the heavyweight: many brokers (Redis, RabbitMQ, SQS), retries, scheduling (beat), chains, groups, chord patterns, time limits, per-task rate limits, priority queues. With it comes complexity — many moving parts to configure, debug, and version. 

python# celery_app.py
from celery import Celery

celery_app = Celery(
    "myapp",
    broker="redis://localhost/0",
    backend="redis://localhost/1",
)

@celery_app.task(bind=True, max_retries=3, default_retry_delay=60)
def sync_inventory(self, device_id: int):
    try:
        ...
    except TransientError as e:
        raise self.retry(exc=e)

# Caller
sync_inventory.delay(device_id=42)         # enqueue
sync_inventory.apply_async(                  # with options
    args=[42], countdown=30, expires=300,
)

Pick Celery when you actually need its features. For most apps starting today, RQ first — graduate to Celery if you outgrow it.

Tier 4: dramatiq, taskiq, arq, etc.

A few worth knowing about: dramatiq is a Celery alternative with a smaller, cleaner API; arq is async-native and Redis-backed; taskiq is a modern async-first option with FastAPI integration. All produce the same shape: enqueue in your web process, run in worker processes.

The decision tree

  • Quick, in-process, fire-and-forget → FastAPI BackgroundTasks.
  • Real queue, no exotic features → RQ or arq.
  • Retries / scheduling / chains / many brokers / decade of recipes → Celery.
  • Periodic batch jobs unrelated to the web app → cron or a systemd timer running a Python script. Don't pull in Celery just for cron.

Idempotency first. Whatever tool you pick, design tasks to be safe to re-run. Workers crash, retries happen, dead- letter queues redrive. A task that double- charges a card on retry is a bug, not a queue choice.

For an existing endpoint where you do a slow thing inline (send email, call a third party, generate a report): try the cheapest tier first. Move it to BackgroundTasks. Time the response. Promote to RQ only when the fire-and-forget semantics start hurting.
