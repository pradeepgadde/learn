hypothesis: property tests find what you didn't think of

Specify what should hold for any input; hypothesis searches for inputs that break it. The library tests no human would write — and finds the bugs no human did.

20 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Decorate with @given(strategy); write a test that holds for any input from that strategy. Hypothesis throws hundreds of inputs at it and shrinks failures to the minimal example. Best for round-trip properties, invariants, and comparison against a reference implementation.

Most tests follow the pattern: "for input X, output should be Y." Property tests flip it: "for any input, this property should hold." Hypothesis is the search engine that finds counter-examples.

The "round-trip" pattern

Best place to start. Anything you can serialize and deserialize: 

pythonfrom hypothesis import given, strategies as st
import json

@given(st.dictionaries(st.text(), st.integers()))
def test_json_round_trip(d):
    assert json.loads(json.dumps(d)) == d

Looks tame. Run it and hypothesis will throw thousands of dicts at the function: empty, nested, with weird unicode, with surrogate characters that aren't valid JSON. Real bugs show up here.

Strategies: the input space

Strategies describe distributions of values. The standard ones cover most cases: 

pythonst.integers(min_value=0, max_value=2**32)         # a port number
st.floats(allow_nan=False, allow_infinity=False)
st.text(min_size=1, max_size=64)
st.lists(st.integers(), unique=True)
st.ip_addresses(v=4)                                # IPv4
st.datetimes()
st.one_of(st.none(), st.integers())              # Optional[int]

# Build a custom strategy from primitives
device_strategy = st.builds(
    Device,
    hostname=st.text(alphabet="abcdefghijklmnop-", min_size=1),
    role    =st.sampled_from(["access", "core", "distribution"]),
    port    =st.integers(min_value=1, max_value=65535),
)

Invariants: things that should always hold

python@given(st.lists(st.integers()))
def test_sort_preserves_length(xs):
    assert len(sorted(xs)) == len(xs)

@given(st.lists(st.integers()))
def test_sort_is_idempotent(xs):
    assert sorted(sorted(xs)) == sorted(xs)

Properties that hold for any input are gold. Each one is essentially infinite unit tests in two lines.

Comparison: oracle pattern

When you have a known-correct slow implementation and you're optimizing it — check the fast one against the slow one for random inputs. 

pythondef slow_unique(xs):
    return [x for i, x in enumerate(xs) if x not in xs[:i]]

@given(st.lists(st.integers(min_value=0, max_value=100)))
def test_fast_unique_matches(xs):
    assert fast_unique(xs) == slow_unique(xs)

Shrinking: the killer feature

When hypothesis finds a failing input, it doesn't show you the random 47-element list that triggered the bug. It shrinks the input to the minimal failing case — usually a 2 or 3 element example. 

textFalsifying example: test_round_trip(d={'\udcff': 0})
# Hypothesis found the issue with a single key/value pair —
# the failure trigger is a surrogate character that's not valid JSON.

That's worth gold during debugging. The minimal example is almost always the bug, not the elaborate input.

Replaying past failures

Hypothesis remembers failing examples in .hypothesis/ and re-runs them on every test invocation. So once a bug is found and you check in the fix, the test stays in your suite forever — with the original failing example baked in.

Settings: speed vs thoroughness

pythonfrom hypothesis import settings, HealthCheck

@settings(max_examples=1000, deadline=200)
@given(...)
def test_thorough(...): ...

Default is 100 examples per test — adequate for CI. Bump for nightly runs; decrease in the fast loop if you're iterating.

What it's NOT for

  • Specific behavior tests — "this exact input should produce this exact output." Use a regular pytest test.
  • Tests that depend on external state — databases, networks. Hypothesis runs each test many times; side effects compound.
  • Tests of code that rejects most inputs — if 99% of generated inputs hit the validation path and return early, you're not testing what you think.

For network-engineering tasks, hypothesis shines on serializers/parsers. Round-trip test your config-render → config-parse pipeline. The first run of a property test on a parser nearly always finds something.

Pick any function in your code that takes structured input (dict, list, string with a format). Write one round-trip or invariant test with hypothesis. Run it once. If it passes, your code's better than you think. If it fails — you found the bug nobody else was going to.
