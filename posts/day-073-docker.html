Docker for Python apps: the right base image

python:slim vs python:alpine vs distroless. The five Dockerfile lines that decide whether your image is 80MB or 800MB. Plus PID 1 and signals.

14 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Use python:3.12-slim as your base unless you have a strong reason. Avoid alpine — musl breaks too many packages. Pin a specific tag, run as a non-root user, set PYTHONDONTWRITEBYTECODE=1 and PYTHONUNBUFFERED=1. Tomorrow's post covers multi-stage builds for production.

Bad Python Dockerfiles are everywhere on the internet. They're fat, slow to build, run as root, and break SIGTERM. Five small choices avoid all of that.

Pick the right base

  • python:3.12 — the full Debian image. ~1GB. Way too much for most apps.
  • python:3.12-slim — Debian without the kitchen sink. ~120MB base. Use this.
  • python:3.12-alpine — tiny (~50MB) but uses musl libc. Many wheels (numpy, cryptography, lxml, psycopg) don't have alpine prebuilds, so pip falls back to compiling from source — slow builds and a fragile result. Skip unless you've measured a real reason.
  • Distroless — no shell, no package manager, just Python and your code. Smallest attack surface. Great for production once you've got the build pinned down.

The minimum-viable Dockerfile

dockerfile# Pin both the Python version AND the base OS for reproducibility
FROM python:3.12.7-slim-bookworm

# Don't write .pyc files; flush stdout/stderr immediately
ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PIP_NO_CACHE_DIR=1 \
    PIP_DISABLE_PIP_VERSION_CHECK=1

# Create non-root user
RUN useradd --create-home --shell /bin/bash app
WORKDIR /app
USER app

# Layer-cache deps separately from source
COPY --chown=app:app requirements.txt .
RUN pip install --user -r requirements.txt
ENV PATH=/home/app/.local/bin:$PATH

COPY --chown=app:app . .

EXPOSE 8000
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]

Six things this gets right that the typical copy-pasted Dockerfile gets wrong: pinned base, env vars set, non-root user, deps cached separately from source, exec-form CMD (so signals work), and explicit port.

Dependencies in their own layer

The COPY requirements.txt + pip install + COPY . ordering means changing your source code doesn't bust the dependency-install layer. Builds go from minutes to seconds on the common path.

The PID 1 problem (and the signal fix)

When you write CMD uvicorn app:app ... in shell form, Docker wraps it in /bin/sh -c, which becomes PID 1. Shell doesn't forward signals. Your container ignores SIGTERM and gets SIGKILL'd 30 seconds later (Day 072).

The fix: exec form — JSON array, not a string. 

dockerfile# BAD — sh swallows signals
CMD uvicorn app:app --host 0.0.0.0

# GOOD — uvicorn becomes PID 1, gets SIGTERM directly
CMD ["uvicorn", "app:app", "--host", "0.0.0.0"]

For complex command lines that need shell features (env-var expansion, pipes), wrap with tini or dumb-init: 

dockerfileRUN apt-get update && apt-get install -y --no-install-recommends tini && rm -rf /var/lib/apt/lists/*
ENTRYPOINT ["/usr/bin/tini", "--"]
CMD ["sh", "-c", "exec uvicorn app:app --workers ${WORKERS:-1}"]

tini as PID 1 forwards signals properly and reaps zombie children. Many production setups standardize on this.

uv-based images

For uv (Day 019), the official Astral image gets you all the speed without the manual setup: 

dockerfileFROM ghcr.io/astral-sh/uv:python3.12-bookworm-slim

WORKDIR /app
COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-install-project

COPY . .
RUN uv sync --frozen

CMD ["uv", "run", "uvicorn", "app:app", "--host", "0.0.0.0"]

The .dockerignore is mandatory

COPY . . happily copies your .git directory, virtualenvs, build artifacts, and node_modules into the image. Add a .dockerignore: 

text.git
.venv
venv
__pycache__
*.pyc
.pytest_cache
.mypy_cache
.ruff_cache
.coverage
node_modules
*.log
.env
.env.*
.DS_Store

Without this, your image gets big, builds get slow (because the build context transfer is fat), and secrets in .env end up baked into the image.

Never bake secrets into images. Use Docker secrets, K8s secrets, or env vars set at runtime. ENV DATABASE_PASSWORD=... in a Dockerfile is permanently visible in docker history and image layers, even if you "remove" it later.

Build the minimum-viable Dockerfile above for a small project. Run docker images and note the size. Then send a SIGTERM to a running container with docker stop and watch the logs — with the exec-form CMD, your shutdown handler runs. With shell form, it doesn't.
