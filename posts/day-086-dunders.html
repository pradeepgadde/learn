Dunders worth knowing: __call__, __getattr__, __init_subclass__

Beyond __init__ and __repr__. Three dunders that quietly power a lot of magic in libraries you already use.

27 Mar 2026 • ~3 min read • Python 3.6+
TL;DR

__call__ makes instances callable like functions — great for stateful callables. __getattr__ intercepts missing-attribute access — how API client SDKs feel like first-class objects. __init_subclass__ runs when a class is subclassed — how plugin registries auto-register without decorators.

Python's data model lets you define what operators, attribute access, and even construction mean for your objects. Most code only ever defines __init__, __repr__, maybe __eq__. The three below are the next tier — explain a lot of "how does that library do that?"

__call__ — instances that act like functions

pythonclass RateLimiter:
    def __init__(self, rps: float):
        self.interval = 1 / rps
        self._last    = 0.0

    def __call__(self):
        now  = time.monotonic()
        wait = self._last + self.interval - now
        if wait > 0:
            time.sleep(wait)
        self._last = time.monotonic()

limit = RateLimiter(rps=10)

for device in devices:
    limit()                          # called like a function
    poll(device)

Why not just a function with a closure? Because the instance carries state you can inspect, reset, or pickle. Pytorch's nn.Module, scikit-learn's estimators, FastAPI's Depends(callable) — all built on __call__.

Bonus: any callable works as a decorator. Class-based decorators are just classes with __call__: 

pythonclass retry:
    def __init__(self, attempts=3):
        self.attempts = attempts

    def __call__(self, fn):
        def wrapper(*a, **kw):
            for _ in range(self.attempts):
                try: return fn(*a, **kw)
                except Exception: pass
            raise
        return wrapper

@retry(attempts=5)
def poll(): ...

__getattr__ — the fallback for missing attributes

Triggered only when normal attribute lookup fails. Lets you forward, lazy-load, or synthesize attributes on demand. 

pythonclass DeviceProxy:
    def __init__(self, connection):
        self._conn = connection

    def __getattr__(self, name):
        # Called only if `name` isn't a real attribute
        if name.startswith("show_"):
            cmd = name.replace("_", " ")
            return lambda: self._conn.send(cmd)
        raise AttributeError(name)

d = DeviceProxy(conn)
d.show_version()                # sends "show version"
d.show_interfaces_brief()       # sends "show interfaces brief"

That's how SDKs feel "magic". Two rules to be a good citizen:

  • Raise AttributeError for unknown names — don't return None. hasattr() relies on the exception.
  • Don't shadow real attributes accidentally. __getattr__ only runs when the normal path fails, but a typo (self.con instead of self._conn) silently falls into the dynamic path.

For every attribute access (including existing ones), use __getattribute__ instead. Almost always the wrong tool — easy to break recursion, easy to break introspection.

__init_subclass__ — hook into subclass creation

Runs once when someone subclasses you. Great for plugin registries, validation, or auto-wiring. 

pythonclass Driver:
    _registry: dict[str, type] = {}

    def __init_subclass__(cls, vendor: str, **kw):
        super().__init_subclass__(**kw)
        Driver._registry[vendor] = cls

class CiscoDriver(Driver, vendor="cisco"): ...
class AristaDriver(Driver, vendor="arista"): ...

Driver._registry
# {'cisco': <class CiscoDriver>, 'arista': <class AristaDriver>}

Each subclass passes vendor= as a keyword, which becomes a parameter to __init_subclass__. The base class keeps a registry. No decorator, no metaclass, no manual registration.

SQLAlchemy, pydantic, attrs — all use this hook to wire up subclass behavior at definition time.

The other dunders worth a mention

  • __enter__ / __exit__ — context managers (Day 010 covered the function-based form; the class form is the real one).
  • __iter__ / __next__ — make your object work in a for loop. For most cases __iter__ returning a generator is enough.
  • __hash__ / __eq__ — make instances usable as dict keys or set members. Override both together; the rules are subtle.
  • __post_init__ — not a Python dunder strictly, but a dataclass extension hook (Day 007) that runs after auto-generated __init__.
  • __set_name__ — runs when a descriptor is assigned in a class body, with the name it was bound to. Powers field-name introspection in dataclass-style libraries.

Restraint. Dunders are powerful, which means they make your code hard for the next person to read. The library author defining __getattr__ in pydantic is earning the magic; the application programmer using it should rarely need to. Reach for these when you're building framework-y code, not for everyday objects.

Find a small bit of code where you've built a registry of subclasses with a decorator (@register). Try replacing it with __init_subclass__. The decorator vanishes; subclassing is the registration. It's a satisfying refactor.
