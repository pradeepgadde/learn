uvicorn and ASGI: serving Python in production

uvicorn is the standard ASGI server. Workers, reload, proxies, and the gunicorn pairing — plus what ASGI actually is.

02 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

uvicorn is the standard ASGI server for FastAPI and Starlette. --reload in dev, --workers N in prod. For multi-worker management with graceful shutdown and zero-downtime reloads, run uvicorn workers under gunicorn (gunicorn -k uvicorn.workers.UvicornWorker app:app).

ASGI is the async successor to WSGI. Where WSGI was synchronous and request/response only, ASGI handles async, websockets, and long-lived connections. uvicorn is the reference server — small, fast, and the default way to run anything ASGI-native.

The dev command

bashuvicorn app:app --reload --host 0.0.0.0 --port 8000
  • app:app — module path and the variable holding the ASGI app instance.
  • --reload — restarts on file changes. Dev-only.
  • --host 0.0.0.0 — bind to all interfaces (default is just localhost).
  • --port — default 8000.

The prod command

bashuvicorn app:app \
    --host 0.0.0.0 \
    --port 8000 \
    --workers 4 \
    --proxy-headers \
    --forwarded-allow-ips '*'

--workers spawns N processes. A reasonable default is 2× the CPU core count for I/O-bound apps, equal to core count for CPU-bound. --proxy-headers tells uvicorn to trust X-Forwarded-For and X-Forwarded-Proto from your load balancer.

Gunicorn for production process management

For graceful worker restarts, better signal handling, and the option to swap workers without dropping connections, run uvicorn workers under gunicorn. The pattern most teams settle on. 

bashuv add gunicorn uvicorn

gunicorn app:app \
    --workers 4 \
    --worker-class uvicorn.workers.UvicornWorker \
    --bind 0.0.0.0:8000 \
    --timeout 60 \
    --keep-alive 5 \
    --access-logfile -

Gunicorn handles process supervision; uvicorn workers handle the actual ASGI traffic. Best of both.

Behind a reverse proxy

Don't expose uvicorn directly to the internet in most cases. Put nginx, Caddy, or Traefik in front for TLS termination, request buffering, and connection limits. uvicorn behind nginx behind a load balancer is the standard topology. 

nginxupstream app {
    server 127.0.0.1:8000;
}

server {
    listen 443 ssl http2;
    location / {
        proxy_pass         http://app;
        proxy_set_header   Host             $host;
        proxy_set_header   X-Real-IP        $remote_addr;
        proxy_set_header   X-Forwarded-For  $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
    }
}

What ASGI looks like underneath

An ASGI app is a callable accepting three arguments per connection: scope (connection info), receive (await for incoming messages), send (await to push outgoing). FastAPI wraps all this; you almost never write it by hand. But knowing the shape helps when middleware or low-level code surprises you. 

python# The simplest possible ASGI app — for context only
async def app(scope, receive, send):
    assert scope["type"] == "http"
    await send({"type": "http.response.start", "status": 200,
                 "headers": [(b"content-type", b"text/plain")]})
    await send({"type": "http.response.body", "body": b"hi"})

Lifespan: startup and shutdown hooks

ASGI also defines a "lifespan" protocol — one message at startup, one at shutdown. FastAPI exposes this via the lifespan context manager. 

pythonfrom contextlib import asynccontextmanager
from fastapi import FastAPI

@asynccontextmanager
async def lifespan(app):
    # Startup
    app.state.db = await connect_db()
    yield
    # Shutdown
    await app.state.db.close()

app = FastAPI(lifespan=lifespan)

This is where DB pools, message-queue clients, and background workers should be wired up — not at module import time.

For extremely performance-sensitive workloads, granian (Rust-based ASGI server) is faster than uvicorn for some patterns. For 99% of services, uvicorn is fast enough that the bottleneck is your code, not the server.

Take a FastAPI app and run it under gunicorn -k uvicorn.workers.UvicornWorker --workers 4. Send 100 concurrent requests (use hey or wrk). Watch the throughput. Try with and without --workers. The difference is real.
