testcontainers: real services for integration tests

Spin up a real Postgres, Redis, or Kafka in Docker for one test. Better than mocks at the integration boundary; faster than docker-compose.

23 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

PostgresContainer("postgres:16") starts a fresh Postgres in Docker, gives you the connection string, and stops it after the test. Use a session-scoped fixture so the container starts once for the whole test run, with each test wrapped in a transaction.

Mocks are great for unit tests; for the layer between your code and the database, fake databases (sqlite, mocks of psycopg) miss real bugs — type quirks, locking, JSON operators, dialect differences. testcontainers gives you the real thing, ephemerally.

Postgres in three lines

pythonfrom testcontainers.postgres import PostgresContainer

with PostgresContainer("postgres:16") as pg:
    conn_str = pg.get_connection_url()
    # conn_str: postgresql+psycopg2://test:test@localhost:32787/test
    # ...run your code against it...

Container starts, exposes a random port, you connect, you do your thing, the container stops. Total cost: ~2 seconds per test session for Postgres. Good return on the seconds spent.

The pytest fixture pattern

For real test suites, hoist the container into a session-scoped fixture (Day 077). Each test gets a transaction-wrapped session that rolls back at the end — isolation without a full restart. 

python# conftest.py
import pytest
from testcontainers.postgres import PostgresContainer
from sqlalchemy import create_engine
from myapp.models import Base

@pytest.fixture(scope="session")
def pg_engine():
    with PostgresContainer("postgres:16") as pg:
        engine = create_engine(pg.get_connection_url())
        Base.metadata.create_all(engine)
        yield engine
        engine.dispose()

@pytest.fixture
def db_session(pg_engine):
    conn    = pg_engine.connect()
    trans   = conn.begin()
    session = Session(bind=conn)
    yield session
    session.close()
    trans.rollback()
    conn.close()

Redis, Kafka, and friends

testcontainers has officially-supported modules for many services: 

pythonfrom testcontainers.redis     import RedisContainer
from testcontainers.mongodb   import MongoDbContainer
from testcontainers.kafka     import KafkaContainer
from testcontainers.mysql     import MySqlContainer
from testcontainers.localstack import LocalStackContainer   # AWS

with RedisContainer() as r:
    redis_url = f"redis://{r.get_container_host_ip()}:{r.get_exposed_port(6379)}"

Generic containers

For services without a dedicated module, the generic DockerContainer base class works for anything in Docker. 

pythonfrom testcontainers.core.container import DockerContainer
from testcontainers.core.waiting_utils import wait_for_logs

container = (
    DockerContainer("hashicorp/vault:latest")
    .with_exposed_ports(8200)
    .with_env("VAULT_DEV_ROOT_TOKEN_ID", "root")
)
with container as vault:
    wait_for_logs(vault, "Vault server started")
    port = vault.get_exposed_port(8200)
    # test against http://localhost:port

Speed: don't restart per test

Container start is the expensive part — seconds, not ms. Three patterns to keep tests fast:

  • Session-scoped container fixture. Start once, use everywhere.
  • Transactions for isolation. Each test in a transaction that rolls back.
  • Reusable containers across local test runs — testcontainers supports this with the TESTCONTAINERS_REUSE_ENABLE=true env var. Container survives the Python process exit; next run reuses it. Saves the start cost on every iteration.

CI considerations

On GitHub Actions and most CI providers, Docker is available out of the box. testcontainers Just Works. Two gotchas:

  • Your CI runner needs enough RAM — Postgres + Redis + your app together can hit 1GB easily.
  • Pull a specific image tag, not latest. Otherwise the test suite changes behavior whenever upstream pushes.

When NOT to use testcontainers

  • Pure unit tests. Mock the database call, don't spin up a database.
  • Multi-service workflows where the orchestration matters — use docker-compose for the suite, run tests against the running stack.
  • Embedded environments without Docker. Some CI systems and dev VMs don't have it; sqlite-in-memory is the Postgres-compatible-enough fallback.

The big win is shifting your "integration" tests left — closer to unit tests in terms of speed and reliability, but with the real database in the loop. Bugs you'd otherwise only catch in staging now show up on the developer's laptop in three seconds.

For one of your repository or DAO classes that talks to Postgres: write an integration test against a testcontainers Postgres. Compare confidence with your previous "mock the cursor" tests — testcontainers catches the SQL syntax errors and JSONB operators that mocks happily ignored.
