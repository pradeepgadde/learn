WebSockets with FastAPI

Long-lived two-way connections. FastAPI handles the protocol; you write the loop. The patterns for chat, telemetry streaming, and live updates.

07 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Decorate with @app.websocket("/ws"), accept the connection, then loop with receive_text / receive_json / send_text / send_json. Catch WebSocketDisconnect for clean teardown. For broadcast, keep a registry of connected sockets.

WebSockets keep a TCP connection open after the HTTP upgrade. Both sides can send messages whenever — no polling, low latency, ideal for live data. FastAPI runs them on the same event loop as your HTTP routes.

The echo example

pythonfrom fastapi import FastAPI, WebSocket, WebSocketDisconnect

app = FastAPI()

@app.websocket("/ws/echo")
async def echo(ws: WebSocket):
    await ws.accept()
    try:
        while True:
            data = await ws.receive_text()
            await ws.send_text(f"echo: {data}")
    except WebSocketDisconnect:
        pass

Three things to internalize: always await ws.accept() first; loop until disconnect; the disconnect arrives as an exception, not a return value.

JSON messages

For structured messages, receive_json / send_json handle the marshalling. Combine with pydantic for validation. 

pythonfrom pydantic import BaseModel

class Cmd(BaseModel):
    action: str
    target: str

@app.websocket("/ws/control")
async def control(ws: WebSocket):
    await ws.accept()
    try:
        while True:
            cmd    = Cmd.model_validate(await ws.receive_json())
            result = await execute(cmd)
            await ws.send_json({"ok": True, "result": result})
    except WebSocketDisconnect:
        pass

Broadcast: a connection registry

For chat or live telemetry, you need to push messages to multiple clients. Track them in a set and iterate when broadcasting. 

pythonclass Hub:
    def __init__(self):
        self.conns: set[WebSocket] = set()

    async def connect(self, ws: WebSocket):
        await ws.accept()
        self.conns.add(ws)

    def disconnect(self, ws: WebSocket):
        self.conns.discard(ws)

    async def broadcast(self, msg):
        dead = []
        for ws in self.conns:
            try:
                await ws.send_json(msg)
            except Exception:
                dead.append(ws)
        for ws in dead:
            self.disconnect(ws)

hub = Hub()

@app.websocket("/ws/feed")
async def feed(ws: WebSocket):
    await hub.connect(ws)
    try:
        while True:
            # Keep the connection open; ignore inbound messages
            await ws.receive_text()
    except WebSocketDisconnect:
        hub.disconnect(ws)

Other parts of the app push events with await hub.broadcast({...}). The dead- socket cleanup pattern matters — clients disappear without warning all the time.

The multi-worker problem

With --workers 4, each worker has its own Hub instance. A message from one worker won't reach clients on the others. For small deployments, run a single worker. For real scale, use a pub/sub backplane (Redis, NATS) so each worker subscribes and forwards to its connected clients. 

python# With Redis as a pub/sub backplane (sketch)
async def redis_subscriber():

Heartbeats and ping/pong

WebSocket connections silently die behind NAT, proxies, and firewalls. uvicorn sends WS-level pings (controlled by --ws-ping-interval); for application-level liveness, send your own ping JSON every N seconds and expect a reply.

Testing WebSockets

pythonfrom fastapi.testclient import TestClient

def test_echo():
    client = TestClient(app)
    with client.websocket_connect("/ws/echo") as ws:

        assert ws.receive_text() == "echo: hello"

For Server-Sent Events (one-way streaming) the setup is much simpler than WebSockets. Use StreamingResponse with a generator that yields data: ...\n\n blocks. Right tool when you only need server → client.
