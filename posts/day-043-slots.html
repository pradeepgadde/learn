__slots__: free memory savings

One class attribute drops per-instance memory by 50%+ on small classes. The cost is no dynamic attribute assignment — usually a feature, not a bug.

12 Feb 2026 • ~3 min read • Python 3.0+
TL;DR

Add __slots__ = ("a", "b", "c") to a class to fix its set of attributes. Per-instance memory drops dramatically because Python skips the per-instance __dict__. For dataclasses, @dataclass(slots=True) does it for you. Use only on classes instantiated in volume.

By default, every Python instance has a __dict__ — a hash table for arbitrary attributes. Flexible, but expensive: ~280 bytes per empty instance just for the dict, before any data. For a class you instantiate millions of times, that adds up to gigabytes.

The minimal change

python# Before — each instance carries a __dict__
class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

# After — fixed slots, no per-instance dict
class Point:
    __slots__ = ("x", "y")

    def __init__(self, x, y):
        self.x = x
        self.y = y

The instance behaves identically — p.x still works, you can still pass instances around. The difference is invisible until you measure memory.

Measuring the win

pythonimport sys

class FatPoint:
    def __init__(self, x, y):
        self.x, self.y = x, y

class SlimPoint:
    __slots__ = ("x", "y")
    def __init__(self, x, y):
        self.x, self.y = x, y

a, b = FatPoint(1, 2), SlimPoint(1, 2)
print(sys.getsizeof(a) + sys.getsizeof(a.__dict__))   # ~344 B
print(sys.getsizeof(b))                                 # ~56 B

Roughly 6× smaller for a two-attribute object. For a list of a million instances, that's the difference between 350 MB and 60 MB.

The trade-off: no new attributes

With slots, you can only assign attributes that are listed. Adding new ones at runtime fails. This is usually what you want — typos become errors. 

pythonp = SlimPoint(1, 2)
p.z = 3
# AttributeError: 'SlimPoint' object has no attribute 'z'

# Common typo bug, caught at the assignment line:
p.xx = 99
# AttributeError: 'SlimPoint' object has no attribute 'xx'

For dataclasses, just pass slots=True

Since 3.10, dataclasses (Day 007) take a slots=True parameter that adds __slots__ automatically. 

pythonfrom dataclasses import dataclass

@dataclass(slots=True, frozen=True)
class Coord:
    lat: float
    lon: float

For value objects you instantiate at scale — coordinates, events, log records, packets — this is essentially free memory savings.

Subclassing rules

Two subtleties to watch:

  • If a subclass needs additional attributes, declare them in its __slots__ — not in the parent's.
  • If a subclass omits __slots__, it gets a __dict__ back, defeating the purpose. Always declare in subclasses too (use an empty tuple __slots__ = () if no new attributes).
pythonclass ColoredPoint(SlimPoint):
    __slots__ = ("color",)

    def __init__(self, x, y, color):
        super().__init__(x, y)
        self.color = color

When NOT to use slots

Skip __slots__ for:

  • Classes you instantiate a few times — the savings don't matter.
  • Classes that benefit from dynamic attributes — mock objects, plugin registries, debugging helpers.
  • Classes with multiple inheritance from non-slot classes — the rules get tricky and you usually lose the savings anyway.

A second small benefit: attribute access on slotted instances is slightly faster than dict-based access, on the order of 20–30%. Rarely the reason to use slots, but a nice side effect.

Find a class in your code that you instantiate at scale (a record, a row, a coordinate). Add __slots__ matching its __init__ assignments — or, if it's a dataclass, add slots=True. Measure with sys.getsizeof. The number of MB you just saved is real.
