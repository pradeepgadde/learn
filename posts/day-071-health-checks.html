Health checks: liveness, readiness, and startup

Three different probes, three different jobs. Liveness for restart-me. Readiness for stop-routing-traffic. Startup for don't-give-up-yet.

12 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Three endpoints with different jobs: /healthz (liveness — am I dead? keep it dumb), /readyz (readiness — should traffic come to me? check downstream deps), /startupz (startup — am I still booting? slow-start grace period). Mixing them is a common cause of cascading outages.

The single /health endpoint that everyone writes works until the day a downstream Postgres flickers and your orchestrator restarts every replica simultaneously. Then you need three checks with three semantics.

Liveness: am I dead?

Liveness answers one question: should you kill and restart this process? It must not check downstream services. If the database is down, you don't want every replica restarting — you want them to wait for the database to come back. Restarting fixes nothing here. 

python@app.get("/healthz")
def healthz():
    # Just prove the process is responding. That's it.
    return {"status": "ok"}

That's the right amount of complexity. The only time this should fail is if the event loop is wedged or memory is so blown that even returning a JSON literal fails — both legitimate restart triggers.

Readiness: should traffic come to me?

Readiness answers: should the load balancer send requests to me right now? Here you do check downstream — if the DB is unreachable, this replica can't serve useful work, so the LB should stop sending it traffic. The process stays up; the routing backs off. 

python@app.get("/readyz")
async def readyz():
    checks = {}

    try:
        async with db.connect() as c:
            await c.execute("SELECT 1")
        checks["db"] = "ok"
    except Exception as e:
        checks["db"] = f"failed: {e}"

    try:
        await redis.ping()
        checks["redis"] = "ok"
    except Exception as e:
        checks["redis"] = f"failed: {e}"

    if all(v == "ok" for v in checks.values()):
        return checks
    raise HTTPException(status_code=503, detail=checks)

Two things to do here that aren't obvious: timeout each downstream check (don't let readyz hang for 30 seconds), and check only deps you actually need to serve the next request — not every dep your app could ever talk to.

Startup: don't give up yet

Startup probes give a process more time to boot before liveness/readiness even begin. Useful for apps with slow startup — warming caches, loading models, running migrations. Without this, K8s may kill a slow-booting pod before it finishes starting. 

python_app_started = False

@asynccontextmanager
async def lifespan(app):
    global _app_started
    await warm_caches()
    await connect_pools()
    _app_started = True
    yield

@app.get("/startupz")
def startupz():
    if not _app_started:
        raise HTTPException(status_code=503, detail="warming up")
    return {"status": "ready"}

Wiring it up in Kubernetes

yamllivenessProbe:
  httpGet: { path: /healthz, port: 8000 }
  initialDelaySeconds: 30
  periodSeconds: 10
  timeoutSeconds: 2
  failureThreshold: 3

readinessProbe:
  httpGet: { path: /readyz, port: 8000 }
  periodSeconds: 5
  timeoutSeconds: 2
  failureThreshold: 1   # one fail = stop traffic immediately

startupProbe:
  httpGet: { path: /startupz, port: 8000 }
  periodSeconds: 5
  failureThreshold: 60  # ~5 minutes to start

Note readinessProbe.failureThreshold: 1 — you want immediate traffic backoff when readiness fails. livenessProbe is more conservative because the consequence of false-positive is a restart.

The cascading-failure trap

If liveness checks the database, and the database flickers for 30 seconds, every replica fails liveness, every replica is restarted, and you've turned a database blip into a full outage with cold caches and thundering reconnect storms. Keep liveness independent of downstream — this is non-negotiable in production.

For deeper diagnostics, expose a separate /debug endpoint behind auth that dumps connection pool stats, queue depths, recent error counts. Probes stay simple; humans get rich info on demand.

Audit your service's health endpoint. If it checks the database, split it into /healthz (no checks) and /readyz (checks). Update your K8s manifest. The next time the database twitches, your pods stay up and recover gracefully instead of restart-stampeding.
