dataclasses + JSON: the round-trip

Three ways to round-trip dataclasses through JSON. Pick by how strict you need the validation.

27 Feb 2026 • ~3 min read • Python 3.10+
TL;DR

For simple cases use asdict + Cls(**data). For nested objects use a default= hook on json.dumps. For real validation with type coercion, use pydantic (Day 032). Don't write your own custom serializer when one of these will do.

Dataclasses (Day 007) and JSON (Day 015) come up together constantly: load config, ship a payload, cache to disk, send through a queue. Three approaches, ordered by complexity.

The simplest: asdict + dict-construct

pythonfrom dataclasses import dataclass, asdict
import json

@dataclass
class Device:
    hostname: str
    role:     str
    port:     int = 22

d    = Device("sw-01", "access")
data = json.dumps(asdict(d))

# Round-trip back
restored = Device(**json.loads(data))

Works perfectly for flat dataclasses with JSON-native types. Two limitations: Device(**data) doesn't validate types — it'll happily build a Device with port="22" instead of an int. And nested non-JSON types (datetimes, paths, enums) need explicit handling.

Nested types: the default= hook

For values that aren't JSON-native, pass a default function that converts them to something that is. 

pythonfrom datetime import datetime
from pathlib import Path
from enum import Enum

def to_json(obj):
    if isinstance(obj, datetime): return obj.isoformat()
    if isinstance(obj, Path):     return str(obj)
    if isinstance(obj, Enum):     return obj.value
    if isinstance(obj, set):      return sorted(obj)
    raise TypeError(obj)

json.dumps(asdict(device), default=to_json, indent=2)

The reverse direction (parsing those strings back into typed objects on load) gets manual. That's where the next approach earns its keep.

Real round-trip with pydantic

For payloads that come from outside or need validation, pydantic gives you both directions for free. 

pythonfrom pydantic import BaseModel
from datetime import datetime
from pathlib import Path

class Device(BaseModel):
    hostname:   str
    role:       str
    port:       int      = 22
    last_seen:  datetime = None
    config_dir: Path     = Path("/etc")

# Out
payload  = device.model_dump_json()

# In, with full validation and type coercion
restored = Device.model_validate_json(payload)
# last_seen comes back as a real datetime
# config_dir comes back as a real Path
# port="22" would still be coerced to int
# missing required fields raise ValidationError

For data crossing a process or network boundary, the validation alone is worth the dependency.

One pitfall: dataclass field order on load

Device(**data) uses keyword arguments, so JSON key order doesn't matter. But if any dataclass field has no default and is missing from the JSON, you'll get a confusing TypeError from __init__, not the kind of nice "missing field" error pydantic gives. For defensive parsing, default everything to None and validate after.

Storing dataclasses in SQLite

Pairing today's pattern with Day 037: use asdict to insert, parse the result row back into the dataclass.

Tidy and dependency-free. For richer types (datetimes, JSON-encoded sub-fields), upgrade to pydantic.

A small middle-ground: msgspec and attrs + cattrs both give you faster, type-validated serialization than pydantic with smaller dependencies. Worth a look if pydantic feels heavy for your use case.

Pick a dataclass you serialize today. Make a tiny round-trip test: assert Cls(**asdict(obj)) == obj. If it fails because of a non-JSON-native field, add a default= hook. If you find yourself wanting validation on load, port to pydantic. The right tool for the job becomes obvious very quickly.
