Type hints, the useful 20%

You don't need to type everything. You need to type the public edges — and use the four or five constructs that catch real bugs.

12 Jan 2026 • ~3 min read • Python 3.10+
TL;DR

Use the modern X | Y union syntax, the built-in generics (list[int], not List[int]), and reach for TypedDict when a function takes or returns a structured dict. Skip the rest until you need it.

Type hints became culturally Pythonic somewhere around 3.9. The mistake most teams make is going all-in — annotating every loop variable — and burning out on the busywork. The 20% that actually matters is small.

The modern union syntax

Since Python 3.10, X | Y replaces Union[X, Y], and X | None replaces Optional[X]. Read like English, no imports needed. 

python# Old, still works
from typing import Optional, Union
def find_user(id: int) -> Optional[User]: ...

# Modern, no import
def find_user(id: int) -> User | None: ...
def parse(value: str | int) -> bool: ...

Built-in generics

Use the lowercase, built-in versions. The capitalized List, Dict, Tuple, Set from typing are deprecated for new code. 

pythondef tally(words: list[str]) -> dict[str, int]:
    counts: dict[str, int] = {}
    for w in words:
        counts[w] = counts.get(w, 0) + 1
    return counts

The 80/20 rule for where to annotate

Three places earn their keep, the rest are optional:

  • Function signatures — especially the public API of a module.
  • Class attributes — particularly on dataclasses, where they're already required.
  • Containers whose contents aren't obvious — users: dict[str, User] tells the next reader something they couldn't infer.

Local variables in a short function rarely need annotation — the assignment usually makes the type obvious.

TypedDict for structured dicts

When a function takes or returns a dict with a known schema (config, JSON payloads, API responses), TypedDict documents the keys without forcing you to make a class. 

pythonfrom typing import TypedDict

class DeviceConfig(TypedDict):
    hostname: str
    port: int
    vlans: list[int]

def apply_config(cfg: DeviceConfig) -> None:
    port = cfg["port"]      # type-checker knows this is int

One escape hatch: Any

Sometimes you genuinely don't know the type — the value comes from json.loads, or a third-party library with no stubs. Use Any. It's not a defeat; it's a clear signal to readers and to the type checker. 

pythonfrom typing import Any

def parse_response(raw: str) -> dict[str, Any]:
    return json.loads(raw)

Type hints don't run. Python ignores them at runtime. To actually catch errors, run mypy or pyright in CI. Without a checker, type hints are documentation — useful, but not enforced.

Pick a single module in your project — the one with the cleanest public API. Add type hints to only its top-level functions. Run mypy on it. Note how many real bugs the type checker flags before you've touched a single internal function.
