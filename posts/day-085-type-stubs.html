Type stubs and py.typed: ship type info with your package

A py.typed marker tells type checkers your package is ready. .pyi stubs type third-party code you don't own. Conventions and gotchas.

26 Mar 2026 • ~3 min read • Python 3.7+
TL;DR

If your package has type hints, drop an empty py.typed file in the package root (PEP 561) and include it in the wheel. Otherwise mypy ignores your types when imported from outside. For libraries you don't own that lack types, write .pyi stubs — they shadow the real module for type-checking purposes only.

You annotated your library beautifully (Day 012, Day 042). You publish it. Someone installs it, runs mypy, and sees... no type information. Why? Because you forgot the marker.

The PEP 561 marker

textsrc/my_tool/
├── __init__.py
├── core.py
├── cli.py
└── py.typed              # empty file — its existence is the signal

Just create the file. Empty. Its mere presence tells mypy, pyright, pytype, and friends: "this package's .py files contain canonical type information — trust them."

Without it, every type checker treats your package as opaque (any imported symbol becomes Any). Adding annotations to your code without adding py.typed is invisible to consumers.

Include it in the wheel

The catch: build backends don't always include non-.py files automatically. With hatchling (Day 083) it usually works, but verify: 

bashhatch build
unzip -l dist/my_tool-*.whl | grep py.typed
# my_tool/py.typed   should appear

If it's missing, force inclusion in pyproject.toml: 

toml[tool.hatch.build.targets.wheel]
packages = ["src/my_tool"]

[tool.hatch.build.targets.wheel.force-include]
"src/my_tool/py.typed" = "my_tool/py.typed"

For setuptools, add py.typed to package_data. For poetry, set include = ["my_tool/py.typed"].

Stub files (.pyi): types without code

A .pyi file mirrors a module's public surface with signatures only. Type checkers prefer it over the matching .py when both exist. 

python# networking.py — runtime code
def parse_cidr(s):
    return _complicated_logic(s)
python# networking.pyi — types only, no implementation
from ipaddress import IPv4Network

def parse_cidr(s: str) -> IPv4Network: ...

Three uses of .pyi:

  • Inline a type override when the runtime form is too dynamic to annotate clearly.
  • Type a C extension — the .so file has no Python source, so the .pyi is the only annotation source.
  • Type a third-party library that lacks types — ship stubs separately (see below).

Stub-only packages

For libraries you don't own, you can publish a stub package alongside. The convention: types-FOO shadows FOO's types. Many examples: types-requests, types-PyYAML, types-paramiko. 

texttypes-mylib/
├── pyproject.toml
└── mylib-stubs/             # note the -stubs suffix
    ├── __init__.pyi
    └── client.pyi
toml[project]
name = "types-mylib"

[tool.hatch.build.targets.wheel]
packages = ["mylib-stubs"]

Type checkers find the FOO-stubs directory automatically; runtime imports of FOO still go to the real package. The stubs are a parallel, type-only mirror.

Generating stubs automatically

For a library that does have type hints, you can generate .pyi files mechanically with stubgen (ships with mypy): 

bashstubgen -p my_tool -o stubs/
# Outputs stubs/my_tool/*.pyi mirroring the public surface

Useful when you want stubs separate from the runtime package, or when you're starting a types-FOO stub for someone else's library.

partial vs complete

Some files in your package may be fully annotated; others not. mypy's default is all-or-nothing per package. To opt in to partial typing, add a marker: 

textmy_tool/
├── py.typed                 # with content "partial\n"

With partial, type checkers fall back to typeshed stubs (or Any) for unannotated bits, instead of refusing the whole package. Useful as you migrate large codebases.

Verify it works end-to-end

bash# In a fresh venv:
pip install ./my-tool
pip install mypy
echo "from my_tool import parse_cidr; reveal_type(parse_cidr('10/24'))" > t.py
mypy t.py
# note: Revealed type is "ipaddress.IPv4Network"

If reveal_type shows the actual return type instead of Any, the py.typed marker is doing its job.

One thing to remember: type stubs lie. The .pyi says one thing, runtime does another, and nobody catches the drift. Whenever possible, annotate the real .py file directly — the types stay honest because they live next to the code they describe.

Pick one of your published packages (or any local one). Verify py.typed is included in the wheel. Install in a clean venv, run mypy against a small file that imports from it. If the symbols come back typed: you're done. If not: fix one of the build-config gotchas above and ship a patch release.
