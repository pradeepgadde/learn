concurrent.futures: parallelism without the threads tax

ThreadPoolExecutor for I/O, ProcessPoolExecutor for CPU. The same map and submit API.

07 Feb 2026 • ~3 min read • Python 3.2+
TL;DR

For I/O-bound work (HTTP, SSH, disk), use ThreadPoolExecutor. For CPU-bound work (number crunching, parsing, image processing), use ProcessPoolExecutor. Both share an API: executor.map(fn, iterable) for the simple case, executor.submit(fn, arg) + as_completed for results in finishing order.

Before reaching for asyncio (Day 039) or threads directly, try concurrent.futures. The API is small, the parallelism model is explicit, and switching between threads and processes is a one-class swap.

The map pattern: same operation on every input

pythonfrom concurrent.futures import ThreadPoolExecutor
import httpx

def probe(host):
    r = httpx.get(f"https://{host}", timeout=5)
    return host, r.status_code

hosts = ["github.com", "google.com", "python.org"]

with ThreadPoolExecutor(max_workers=10) as ex:
    for host, status in ex.map(probe, hosts):
        print(host, status)

map preserves input order in the output. The with block waits for all tasks to finish before exiting, even if one raised — tasks aren't cancelled, just collected.

The submit pattern: results as they complete

When tasks have wildly varying durations (slow probes, fast probes), don't wait in submission order. Use submit + as_completed. 

pythonfrom concurrent.futures import ThreadPoolExecutor, as_completed

with ThreadPoolExecutor(max_workers=25) as ex:
    # Submit returns a Future immediately, no blocking
    futures = {ex.submit(probe, h): h for h in hosts}

    for fut in as_completed(futures):
        host = futures[fut]
        try:
            _, status = fut.result()
            print(f"{host}: {status}")
        except Exception as e:
            print(f"{host}: failed ({e})")

The {future: input} dict pattern is essential — as_completed yields futures, not original inputs, so you need a way to map back.

Threads vs processes: pick by workload

The Python GIL means CPU-bound work in threads doesn't actually run in parallel — threads take turns. For I/O-bound work, this doesn't matter (threads release the GIL while waiting on the network or disk). 

pythonfrom concurrent.futures import ProcessPoolExecutor

def crunch(n):
    # Pure CPU work — sieve, image filter, regex on huge text
    return sum(i * i for i in range(n))

with ProcessPoolExecutor() as ex:
    results = list(ex.map(crunch, [10_000_000] * 8))

ProcessPoolExecutor defaults to os.cpu_count() workers. Each worker is a separate Python process with its own memory and GIL, so CPU-bound work scales to your core count.

The process-pool gotcha: pickling

Process pools pickle arguments and return values to ship them between processes. That means lambdas, local functions, and unpicklable objects (file handles, sockets, locks) don't work. Top-level functions only. 

python# DON'T — lambda can't be pickled
with ProcessPoolExecutor() as ex:
    ex.map(lambda x: x * 2, items)   # PicklingError

# DO — module-level function
def double(x): return x * 2

with ProcessPoolExecutor() as ex:
    ex.map(double, items)

Sizing the pool

Two rules of thumb that work most of the time:

  • I/O-bound (threads): 10–50 workers per pool. The bottleneck is the upstream service, not your CPU.
  • CPU-bound (processes): os.cpu_count() — the default. More processes than cores = thrashing.

For a busy SSH fleet, max_workers=25 is a good starting point. Your auth server or the target devices will tell you when you've gone too high.

For very large numbers of I/O-bound tasks (thousands of HTTP requests, hundreds of SSH connections), asyncio (Day 039) scales further than threads. The crossover is around 100–500 concurrent tasks — below that, threads are simpler.

Take a script that makes 10+ HTTP requests in a loop. Wrap it in ThreadPoolExecutor with max_workers=10. Time the before and after. You'll usually see a 5–10x speedup with no other changes.
