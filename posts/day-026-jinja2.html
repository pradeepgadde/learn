Jinja2 for config templating

Separate the config you want from the data that drives it. One template, any number of devices, fewer copy-paste sins.

26 Jan 2026 • ~3 min read • Python 3.7+
TL;DR

Jinja2 turns a text template plus a Python dict into a rendered string. Three constructs cover 90% of network config work: {{ var }} for values, {% for %} for loops, {% if %} for conditionals. Use FileSystemLoader for templates on disk; use the | filter for formatting.

Network configs are repetitive in structure but vary in details — the same VLAN block on every access switch, with different VLAN IDs and descriptions. Jinja2 was built in the web world but it's the de-facto templating engine for Ansible, Salt, and most network-automation toolchains for exactly this reason.

The minimal template render

pythonfrom jinja2 import Template

tmpl = Template("""
hostname {{ hostname }}
!
interface Loopback0
 ip address {{ loopback }} 255.255.255.255
""")

print(tmpl.render(hostname="core-sw-01", loopback="10.255.0.1"))
hostname core-sw-01 ! interface Loopback0 ip address 10.255.0.1 255.255.255.255

For one-off renders that's all you need. Template(...) takes a string, render(...) takes the variables.

Loops and conditionals

jinja!
{% for v in vlans %}
vlan {{ v.id }}
 name {{ v.name }}
{% endfor %}
!
{% for iface in interfaces %}
interface {{ iface.name }}
 description {{ iface.desc }}
{% if iface.trunk %}
 switchport mode trunk
 switchport trunk allowed vlan {{ iface.allowed | join(',') }}
{% else %}
 switchport mode access
 switchport access vlan {{ iface.vlan }}
{% endif %}
{% endfor %}
pythondata = {
    "vlans": [
        {"id": 10, "name": "users"},
        {"id": 20, "name": "voice"},
    ],
    "interfaces": [
        {"name": "Gi1/0/1", "desc": "uplink",
         "trunk": True,  "allowed": [10, 20]},
        {"name": "Gi1/0/2", "desc": "phone",
         "trunk": False, "vlan": 20},
    ],
}

The | join(',') is a Jinja filter — it runs the value through a function before rendering. There are dozens built in: upper, lower, default('x'), length, sort. They chain: {{ name | lower | replace(' ', '-') }}.

Templates on disk

For real projects, templates live as .j2 files next to your Python code. Use a FileSystemLoader and an Environment. 

pythonfrom jinja2 import Environment, FileSystemLoader, StrictUndefined

env = Environment(
    loader=FileSystemLoader("templates"),
    undefined=StrictUndefined,         # raise on missing vars
    trim_blocks=True,                  # cleaner output
    lstrip_blocks=True,
    keep_trailing_newline=True,
)

tmpl = env.get_template("access-switch.j2")
config = tmpl.render(**device_data)

StrictUndefined is the unsung hero. By default, Jinja silently renders missing variables as the empty string — perfect for sneaking vlan with nothing after it into a production switch. StrictUndefined raises instead.

One template, many devices

pythonimport yaml
from pathlib import Path

tmpl = env.get_template("access-switch.j2")

for device_file in Path("vars/devices").glob("*.yml"):
    data = yaml.safe_load(device_file.read_text())
    out = Path("build") / f"{device_file.stem}.cfg"
    out.write_text(tmpl.render(**data))

Per-device YAML in, per-device config out. The template is the source of truth for structure; the YAML is the source of truth for facts. Diff build/ against your deployed configs and you have a basic config-drift check.

Two filters worth knowing for network work: {{ ip | ipaddr('network') }} from the netaddr ecosystem (also available in Ansible's ansible.utils) and the built-in {{ value | regex_replace('^x', 'y') }}. When the built-ins aren't enough, you can register your own with env.filters['my_filter'] = my_func.

Pick three devices with similar configs (e.g. three access switches). Extract the parts that differ into a YAML file per device. Build a single Jinja template that renders any of them. The first time you change a setting across the fleet by editing one template, you'll be sold.
