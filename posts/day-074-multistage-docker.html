Multi-stage Dockerfiles: the build/runtime split

Build heavy in one stage; copy the result into a slim runtime stage. Smaller images, faster pulls, no C compiler in production.

15 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Use FROM ... AS builder for the compile-heavy stage (gcc, build deps, pip install). Use a fresh FROM python:slim for runtime. COPY --from=builder brings only the installed site-packages and your code across. The runtime image has no compilers, no build artifacts, no surprises.

Production images should ship the smallest attack surface possible. The build tooling (gcc, make, dev headers, pip cache) is irrelevant once installation is done. Multi- stage builds let you discard all of it.

The pattern

dockerfile# ============================================================
# STAGE 1: builder — has compilers, builds wheels
# ============================================================
FROM python:3.12.7-slim-bookworm AS builder

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PIP_NO_CACHE_DIR=1

# Build-time system deps (compilers, headers)
RUN apt-get update && apt-get install -y --no-install-recommends \
        build-essential \
        libpq-dev \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY requirements.txt .

# Install into a virtualenv so we can copy a clean tree
RUN python -m venv /opt/venv
ENV PATH=/opt/venv/bin:$PATH
RUN pip install -r requirements.txt

# ============================================================
# STAGE 2: runtime — slim, no compilers
# ============================================================
FROM python:3.12.7-slim-bookworm AS runtime

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    PATH=/opt/venv/bin:$PATH

# Only the runtime libs — not -dev versions
RUN apt-get update && apt-get install -y --no-install-recommends \
        libpq5 \
    && rm -rf /var/lib/apt/lists/*

RUN useradd --create-home --shell /bin/bash app
WORKDIR /app
USER app

# Copy the built virtualenv from the builder stage
COPY --from=builder /opt/venv /opt/venv

COPY --chown=app:app . .

CMD ["uvicorn", "app:app", "--host", "0.0.0.0"]

The runtime image has the compiled site-packages but none of the tools used to compile them. For a typical FastAPI app this is the difference between a 500MB image and a 180MB one.

The virtualenv trick

Why install into a venv if we're already in a container? Because COPY --from works at the filesystem level — it's much cleaner to copy /opt/venv as a single directory than to track down site-packages + scripts + data files spread across the system Python install.

uv version

dockerfile# Builder
FROM ghcr.io/astral-sh/uv:python3.12-bookworm-slim AS builder
WORKDIR /app
COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-install-project --no-dev
COPY . .
RUN uv sync --frozen --no-dev

# Runtime
FROM python:3.12-slim
COPY --from=builder --chown=app:app /app /app
WORKDIR /app
ENV PATH=/app/.venv/bin:$PATH
CMD ["uvicorn", "app:app", "--host", "0.0.0.0"]

uv keeps its venv at .venv/ in the project, so you copy the whole project. The --no-dev flag skips dev-only packages (pytest, ruff, mypy) that have no place in production.

BuildKit cache mounts

Docker BuildKit (default in modern Docker) supports cache mounts that persist between builds. For pip and uv this is huge. 

dockerfile# syntax=docker/dockerfile:1.6

FROM python:3.12-slim AS builder
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

Re-running the build hits the warm cache instead of re-downloading wheels. CI builds often go from 90 seconds to 15.

Distroless for the truly small

For the smallest production image, copy your venv into a distroless base. No shell, no apt, no curl — just enough Python to run. 

dockerfileFROM python:3.12-slim AS builder
# ...build virtualenv...

FROM gcr.io/distroless/python3-debian12
COPY --from=builder /opt/venv /opt/venv
COPY . /app
WORKDIR /app
ENV PATH=/opt/venv/bin:$PATH PYTHONPATH=/opt/venv/lib/python3.12/site-packages
CMD ["/opt/venv/bin/uvicorn", "app:app", "--host", "0.0.0.0"]

Pros: ~80MB total, minimal CVE surface. Cons: no shell to debug. kubectl exec and docker exec won't get you a shell — debugging means rebuilding with the slim base. Worth it for clearly-defined services you don't poke at.

Image scanning

Once you've shrunk the image, scan it for known CVEs. trivy image myapp:tag takes a few seconds and surfaces vulnerable packages. Wire it into your build pipeline (Day 075). Smaller image = fewer CVEs by simple counting.

For local development, you may want a single- stage image with the build tools included (faster iteration, easier debugging). Use a --target builder flag to stop the multi-stage build at the builder stage: docker build --target builder -t myapp:dev .

Take a working single-stage Dockerfile, convert to two stages. Compare image sizes with docker images. For a typical FastAPI app the cut is dramatic. Then run trivy against both — the slim version usually has half the vulnerabilities just because it's smaller.
