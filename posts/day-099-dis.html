dis: peek at Python bytecode

dis.dis() shows the bytecode CPython compiles to. Not for everyday use — but invaluable for understanding why one idiom is faster than another.

9 Apr 2026 • ~2 min read • Python 3.8+
TL;DR

import dis; dis.dis(fn) prints the bytecode instructions CPython generates for a function. Useful for understanding performance differences, verifying that constants are folded, and curiosity-driven learning about how Python actually executes your code.

Python compiles your source to bytecode before running it. CPython's bytecode is a simple stack-based instruction set. dis lets you read it.

Basic usage

pythonimport dis

def add(a, b):
    return a + b

dis.dis(add)
text  2           RESUME          0

  3           LOAD_FAST           0 (a)
              LOAD_FAST           1 (b)
              BINARY_OP           0 (+)
              RETURN_VALUE

Column 1 is the source line number. Then the offset in bytes, the opcode name, and the argument. The stack grows left-to-right as values are pushed; operators pop their arguments and push results.

Why one idiom is faster: local vs global

pythonimport dis

x = 10

def uses_global():
    return x + 1

def uses_local():
    y = 10
    return y + 1

dis.dis(uses_global)
# LOAD_GLOBAL 0 (x)   <— dict lookup

dis.dis(uses_local)
# LOAD_FAST 0 (y)     <— array index lookup, faster

LOAD_GLOBAL does a dict lookup in the module's namespace. LOAD_FAST reads directly from a pre-allocated slot in the frame's local variables. That's why "cache globals as locals in tight loops" is a real Python optimisation.

Constant folding

pythondef f():
    return 2 * 3 + 1

dis.dis(f)
# LOAD_CONST 7    ← CPython folded the whole expression at compile time
# RETURN_VALUE

CPython folds constant arithmetic at compile time. 2 * 3 + 1 never executes at runtime; it becomes the literal 7 in the bytecode.

List vs generator comprehension

pythondef with_list():
    return [x*2 for x in range(10)]

def with_gen():
    return (x*2 for x in range(10))

dis.dis(with_list)
# LIST_APPEND in a loop — builds list immediately

dis.dis(with_gen)
# RETURN_GENERATOR + a nested code object — lazy

dis on a class or module

python# Disassemble an entire class (all methods)
dis.dis(MyClass)

# Disassemble a string of source
dis.dis("x = [i**2 for i in range(5)]")

# Get instruction objects programmatically
instrs = list(dis.get_instructions(add))
for i in instrs:
    print(i.opname, i.argval)

When does it actually matter?

Most code: it doesn't. Profile first (Day 035); optimize later. But when you're in a tight inner loop that runs millions of times, understanding LOAD_GLOBAL vs LOAD_FAST, or seeing that a function call compiles to CALL_FUNCTION + setup overhead, explains why micro-optimisations sometimes work. It also demystifies magic: comprehensions, generators, with blocks, and walrus operators all have visible bytecode.

The bytecode format changes across Python versions. 3.12 introduced specialising adaptive interpreter (COLD_CALL, BINARY_OP_ADD_INT) that specialises opcodes at runtime based on observed types. If you see unfamiliar opcode names, check the version-specific docs.

Pick any two equivalent ways to write the same thing — a for loop vs a comprehension, dict(a=1) vs {"a": 1}, or appending to a list vs concatenating. Run dis.dis on both. Count the instructions. The one with fewer instructions is usually (not always) faster. Then verify with %timeit.
