tenacity: retries done right

Exponential backoff, jitter, retry-on-exception filters, and max attempts — all declarative, all in one decorator. Stop writing while-loops with sleeps.

02 Feb 2026 • ~3 min read • Python 3.7+
TL;DR

@retry wraps any function with retry logic. Combine stop_after_attempt, wait_exponential, and retry_if_exception_type to express almost any retry policy. Add reraise=True so the original exception bubbles up, not RetryError.

Every codebase that talks to a network has a hand-rolled retry loop somewhere. They all have subtle bugs — no backoff, no jitter, retrying on exceptions that won't succeed on retry, swallowing the real error. tenacity turns the policy into a declaration.

The boilerplate it deletes

python# Before — a thousand small bugs waiting
def fetch(url):
    for i in range(3):
        try:
            return httpx.get(url)
        except Exception:
            time.sleep(2)
    raise RuntimeError("gave up")

# After
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3),
       wait=wait_exponential(multiplier=1, min=1, max=10),
       reraise=True)
def fetch(url):
    return httpx.get(url)

Same intent, dramatically clearer. reraise=True is important — without it, the failure becomes a RetryError wrapping the real exception, which breaks every except SpecificError upstream.

Retry only on the right exceptions

Retrying on any exception is almost always wrong. A ValueError in your URL parsing won't fix itself. Retry on the specific transient failures. 

pythonfrom tenacity import retry, retry_if_exception_type

@retry(
    stop=stop_after_attempt(5),
    wait=wait_exponential(multiplier=1, max=30),
    retry=retry_if_exception_type((httpx.TimeoutException,
                                       httpx.ConnectError)),
    reraise=True,
)
def fetch(url):
    r = httpx.get(url, timeout=5)
    r.raise_for_status()
    return r

Notice this retries on connection problems but not on HTTPStatusError from raise_for_status() — a 404 isn't a transient failure, retrying it is useless and rude to the upstream. Add 5xx-only retry by using retry_if_result with a status check.

Jitter prevents thundering herds

If a service comes back online after a brief outage and every retrying client hits it at exactly the same time, you just DDoSed it. Add jitter so retries spread out. 

pythonfrom tenacity import wait_random_exponential

@retry(wait=wait_random_exponential(multiplier=1, max=60),
       stop=stop_after_attempt(8),
       reraise=True)
def probe_device(host):
    ...

wait_random_exponential is exponential backoff with built-in randomness — usually what you want for anything talking to a shared service.

Combining stop conditions

Stop after N attempts or M seconds, whichever comes first. Useful for "try hard but don't block forever." 

pythonfrom tenacity import stop_after_attempt, stop_after_delay

@retry(stop=(stop_after_attempt(10) | stop_after_delay(30)),
       wait=wait_exponential(),
       reraise=True)
def poll_until_ready():
    ...

Logging each retry

For debugging or audit, hook into each retry attempt. 

pythonfrom tenacity import retry, before_sleep_log
import logging

log = logging.getLogger(__name__)

@retry(stop=stop_after_attempt(3),
       wait=wait_exponential(),
       before_sleep=before_sleep_log(log, logging.WARNING),
       reraise=True)
def fetch(url):
    ...

Each retry now emits "Retrying fetch in 2.0s as it raised ConnectError" at WARNING level. Combined with structured logging (Day 016), you get a complete audit trail of retries.

Don't retry non-idempotent operations blindly. A retried POST /charge-card can charge the card twice. Either make the operation idempotent (with an idempotency key) or only retry on errors that prove the request didn't reach the server (connect errors, not read timeouts).

Find a for i in range(N): try/except retry loop in your code. Replace it with @retry. Pick the right exception types to retry on (not "any exception"). Add reraise=True. Delete the loop. Sleep better.
