mypy: turn type hints into a real check

Type hints are documentation until a checker enforces them. mypy is the original. Adopt one module at a time.

11 Feb 2026 • ~3 min read • Python 3.8+
TL;DR

Run mypy your_package/ to catch type errors. Don't try to type-check everything on day one — adopt module-by-module with strict = true per-module overrides. Three flags carry most of the value: --strict, --disallow-untyped-defs, --warn-unused-ignores.

Type hints (Day 012) are documentation. mypy turns them into enforced checks: a function that returns str | None can't be passed to one that expects str without you handling the None case explicitly. The errors mypy catches are real bugs, not style nits.

The minimal run

bashuv add --dev mypy

# Check a package or module
mypy src/myapp/

# Output looks like:
# src/myapp/network.py:42: error: Argument 1 to "connect" has
#   incompatible type "str | None"; expected "str"  [arg-type]

Each error includes the file, line, the type mismatch, and a rule code in brackets you can ignore selectively.

Configuration in pyproject.toml

toml[tool.mypy]
python_version = "3.12"

# Reasonable starting strictness
warn_return_any = true
warn_unused_ignores = true
warn_redundant_casts = true
no_implicit_optional = true

# Per-module overrides for incremental adoption
[[tool.mypy.overrides]]
module = "myapp.core.*"
strict = true

[[tool.mypy.overrides]]
module = "myapp.legacy.*"
ignore_errors = true

The per-module override pattern is how you adopt mypy without rewriting your codebase. New code goes under strict = true; legacy code is excluded entirely. Migrate modules one at a time, removing them from the ignore list as they get cleaned up.

The flags that matter

  • --strict — turns on all the strict-mode flags at once. The default is too lenient; you almost certainly want this on the modules you care about.
  • --disallow-untyped-defs — refuses to check functions that have no annotations at all (otherwise they pass silently). Forces you to annotate before mypy will help.
  • --warn-unused-ignores — surfaces stale type: ignore comments. As code evolves, suppressions become unnecessary; this catches them.

Suppressing errors locally

Sometimes mypy is wrong, or you genuinely need to do something it can't model. Use # type: ignore[code] with the rule code — never just # type: ignore. 

python# Bad: silences ALL errors on this line, including future ones
result = spooky(x)  # type: ignore

# Good: silences only the specific error you intended
result = spooky(x)  # type: ignore[arg-type]

Stub files for typeless dependencies

Some libraries don't ship type information. mypy will complain that "module has no type stubs". Two fixes: 

bash# Look for an official stub package — they're conventionally named types-X
pip install types-requests types-PyYAML
toml# If no stubs exist, silence the missing-import warning per-module
[[tool.mypy.overrides]]
module = ["paramiko.*", "netmiko.*"]
ignore_missing_imports = true

Generic types, briefly

For containers whose contents matter, type the contents. Most cases are list[X], dict[K, V], tuple[X, ...] for variable-length tuples or tuple[X, Y, Z] for fixed-length. 

pythondef parse_devices(paths: list[Path]) -> dict[str, DeviceConfig]:
    return {p.stem: DeviceConfig.model_validate_json(p.read_text())
            for p in paths}

mypy vs pyright

pyright (Microsoft) is the other major type-checker, with first-class editor support via Pylance in VS Code. It's faster than mypy and slightly stricter by default. For interactive editing, pyright is the de-facto standard. For CI, both work; pick by which gives clearer errors on your codebase.

Don't try to type-check everything on day one. Pick one new module, add types, run mypy, fix the errors, ship. Repeat next week with another module. Inertia is the killer of type-checker adoption; momentum is the fix.

Add mypy to your most-tested module. Set strict = true for that module only. Run. Fix what comes up. Most "errors" turn out to be either missed None handling or genuine documentation gaps. Both are wins.
