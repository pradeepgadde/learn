NETCONF with ncclient: configs as XML

Transactional, schema-validated config changes — no screen-scraping, no half-applied configs. The protocol the CLI should have been.

30 Jan 2026 • ~3 min read • Python 3.7+
TL;DR

NETCONF is the standards-track replacement for CLI scraping: structured XML over SSH, with separate candidate and running datastores, atomic commits, and rollback. ncclient is the Python client. Lock the datastore, edit, validate, commit — or roll back cleanly.

CLI automation has fundamental problems: prompt parsing, device-specific quirks, and worst of all, no transactions. A config push that fails halfway through leaves the device in a half-broken state. NETCONF (RFC 6241) was designed to fix all of this.

Connecting

pythonfrom ncclient import manager

conn = manager.connect(
    host="router-01",
    port=830,                          # NETCONF default port
    username="netadmin",
    key_filename="~/.ssh/id_ed25519",
    hostkey_verify=True,
    device_params={"name": "iosxe"},  # or 'junos', 'nexus', ...
)

# See what the device offers
for cap in conn.server_capabilities:
    if "yang" in cap:
        print(cap)

Capabilities are the device's contract: which YANG models it supports, whether it has writable candidate and startup datastores, whether it supports :rollback-on-error. Worth printing once per device type.

Reading config and state

python# Filter to just the interfaces tree
filt = """
<filter>
  <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces"/>
</filter>
"""

reply = conn.get_config(source="running", filter=filt)
print(reply.data_xml[:300])

get returns operational state (counters, link status); get_config returns persisted config. Different things, different endpoints — an important NETCONF distinction the CLI doesn't make.

The atomic edit-validate-commit cycle

Here's the workflow you can't safely do over CLI: lock the candidate datastore, apply changes, validate, commit if good, discard if bad. 

pythonpayload = """
<config>
  <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces">
    <interface>
      <name>GigabitEthernet1/0/24</name>
      <description>uplink to dc-router-02</description>
      <enabled>true</enabled>
    </interface>
  </interfaces>
</config>
"""

with conn.locked("candidate"):
    conn.edit_config(target="candidate", config=payload)
    conn.validate(source="candidate")
    conn.commit()
# Lock auto-released. If anything raised, candidate is dropped.

Three guarantees you don't get from CLI: nobody else can edit while you hold the lock, the device validates the full config before applying, and the change is atomic — all of it or none of it.

Confirmed commits: rollback if you don't confirm

The networking equivalent of "don't lock yourself out": apply a change, but tell the device to roll it back automatically if you don't confirm within N seconds. 

pythonconn.edit_config(target="candidate", config=payload)
conn.commit(confirmed=True, timeout="60")

# Verify the change worked, then:
conn.commit()       # confirms

# If you can't reach the device after the change, it auto-rolls back.

This is the safety net for ACL changes, OSPF reconfigs, anything where the change could sever the management plane. Junos has had it for years; modern IOS-XE supports it too.

Building XML in Python instead of strings

Hand-writing XML strings is fragile. lxml or xmltodict let you build the payload from Python data structures. 

pythonimport xmltodict

data = {
    "config": {
        "interfaces": {
            "@xmlns": "urn:ietf:params:xml:ns:yang:ietf-interfaces",
            "interface": {
                "name":        "GigabitEthernet1/0/24",
                "description": "uplink",
                "enabled":     "true",
            },
        },
    },
}
payload = xmltodict.unparse(data, full_document=False)

NETCONF is one of three structured-management protocols you'll encounter: NETCONF (XML, mature), RESTCONF (HTTP+JSON, newer), and gNMI (gRPC, streaming telemetry). All three are based on YANG models. Pick by what your devices support; the data shapes are similar.

On a lab device that supports NETCONF, write a script that locks the candidate, sets a description on a loopback interface, validates, and commits — with a confirmed-commit timeout of 30 seconds. Then deliberately skip the confirm and watch the device roll back the change on its own.
