Pre-commit beyond ruff: the hooks worth wiring up

Catch trailing whitespace, accidental secrets, broken YAML, and merge conflict markers before the commit goes out.

17 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Beyond ruff and ruff-format: add pre-commit-hooks for whitespace/EOL/YAML basics, detect-secrets for credential leak prevention, shellcheck for any shell scripts, and actionlint for GitHub Actions YAML. All driven from one .pre-commit-config.yaml.

Ruff (Day 041) covers the Python-specific cleanup — lint and format. Pre-commit wraps a lot more than that: file hygiene, secret detection, config-file validation, cross-language linters. Set up once, runs on every git commit.

The starter config

yaml# .pre-commit-config.yaml
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v5.0.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-toml
      - id: check-json
      - id: check-merge-conflict
      - id: check-added-large-files
        args: ['--maxkb=500']
      - id: check-case-conflict
      - id: mixed-line-ending
      - id: detect-private-key

  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.7.4
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format

  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.5.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']

  - repo: https://github.com/rhysd/actionlint
    rev: v1.7.4
    hooks:
      - id: actionlint

Install once, then it runs automatically: 

bashuv add --dev pre-commit
uv run pre-commit install                     # installs git hook
uv run pre-commit run --all-files             # run on the whole repo first time

What each one catches

  • trailing-whitespace — the most pointless source of "fix whitespace" commits in history.
  • end-of-file-fixer — ensures files end in a newline. Some tools complain when they don't.
  • check-yaml/json/toml — parses the file. Catches the typo that breaks CI before you push.
  • check-merge-conflict — rejects files with <<<<<<< markers in them. You'd be surprised.
  • check-added-large-files — stops you committing the 200MB tarball you forgot to .gitignore.
  • check-case-conflict — README and readme on the same branch breaks Windows and macOS clones.
  • detect-private-key — catches accidental SSH/PGP key commits.
  • detect-secrets — the more thorough version. AWS keys, API tokens, high-entropy strings.
  • actionlint — validates your GitHub Actions YAML. Catches typos that would fail CI.

detect-secrets baseline

First run scans your existing repo and writes a baseline of "known false positives" so that legitimate strings (like example API keys in documentation) don't trigger every commit. 

bashuv run detect-secrets scan > .secrets.baseline
git add .secrets.baseline

New high-entropy strings outside the baseline block the commit until you audit and either fix the leak or update the baseline.

For repos with shell scripts

yaml  - repo: https://github.com/koalaman/shellcheck-precommit
    rev: v0.10.0
    hooks:
      - id: shellcheck

ShellCheck flags common bash mistakes — unquoted variables, broken globbing, non-portable syntax. Catches a lot of Saturday-morning incidents.

For repos with Dockerfiles

yaml  - repo: https://github.com/hadolint/hadolint
    rev: v2.13.1-beta
    hooks:
      - id: hadolint-docker

Hadolint catches the Dockerfile anti-patterns from Days 073 and 074: shell-form CMD, missing version pins, apt-get update without cleanup, etc.

Run pre-commit in CI too

Hooks installed locally are easy to bypass with git commit --no-verify. Run the same suite in CI as a backstop: 

yaml# In your GitHub Actions workflow
- uses: pre-commit/action@v3.0.1

Now there's no escape. Local pre-commit makes the loop fast; CI pre-commit makes it non-negotiable.

Updating hook versions

bashuv run pre-commit autoupdate

Updates all the rev: pins to the latest tags. Run monthly, commit, watch CI to verify nothing broke.

One nuance: pre-commit hooks only run on staged files by default. If you ever need to re-run on everything (after adding a new hook, or after a big refactor): pre-commit run --all-files.

Drop the config above into a repo, run pre-commit install, then pre-commit run --all-files. Watch how much it cleans up on first run. The friction of catching small problems is much smaller than the friction of getting them fixed in code review three days later.
