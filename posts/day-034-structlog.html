structlog for JSON logs you can grep

Logs as key-value events, not pre-formatted strings. Bind context once. Ship JSON to your aggregator. Stop parsing your own log lines with regex.

03 Feb 2026 • ~3 min read • Python 3.8+
TL;DR

structlog treats logs as structured events, not strings. Pass key=value pairs instead of f-strings. Bind contextual fields like request_id once with .bind() and they appear in every subsequent log line. Configure JSON output for production, pretty terminal output for dev.

The standard logging module from Day 016 is fine for human eyes. The moment you ship logs to Loki, ELK, Splunk, Datadog — anywhere logs become queryable structured data — you want JSON output with named fields. Building that on top of logging is awkward; structlog is built for it.

The minimal setup

pythonimport structlog

structlog.configure(
    processors=[
        structlog.processors.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ],
)

log = structlog.get_logger()

log.info("connected", device="sw-01", port=22, latency_ms=42)
{"event": "connected", "device": "sw-01", "port": 22, "latency_ms": 42, "level": "info", "timestamp": "2026-02-03T14:03:21.182Z"}

That's a JSON document per log line. Pipe it to jq, ship it to Loki, query by field. The contrast with "connected to sw-01:22 in 42ms" is enormous — nobody has to write a regex to extract the latency.

Pretty output for dev, JSON for prod

pythonimport os, structlog

renderer = (structlog.processors.JSONRenderer()
            if os.environ.get("ENV") == "production"
            else structlog.dev.ConsoleRenderer())

structlog.configure(processors=[
    structlog.processors.add_log_level,
    structlog.processors.TimeStamper(fmt="iso"),
    renderer,
])

ConsoleRenderer gives you colored, key-value output that's instantly readable in a terminal. Same code, same logger calls, different presentation per environment.

The killer feature: bound context

Bind a value once and every subsequent log call from that logger includes it. No more passing request_id through five function signatures just to log it. 

pythondef handle_request(req):
    log = structlog.get_logger().bind(
        request_id=req.id,
        user=req.user.name,
    )
    log.info("received", path=req.path)
    result = process(req, log)
    log.info("completed", status=result.status)
{"event": "received", "path": "/api/v1/devices", "request_id": "abc-123", "user": "pradeep", "level": "info", ...} {"event": "completed", "status": 200, "request_id": "abc-123", "user": "pradeep", "level": "info", ...}

Both lines carry request_id and user because they came from the same bound logger. Pass log down to nested functions and the context follows.

Context vars for ambient binding

For web frameworks where you want bound context per-request without threading the logger through everything, contextvars integration handles it. 

pythonfrom structlog.contextvars import bind_contextvars, clear_contextvars

def middleware(request):
    clear_contextvars()
    bind_contextvars(request_id=request.id)
    # every log.info() in this request now includes request_id
    # automatically — no logger threading needed

Logging exceptions with context

pythontry:
    connect(host)
except ConnectionError:
    log.exception("connect_failed", host=host, attempt=3)

log.exception includes the full traceback as a structured field. With JSON output, your aggregator gets the stack trace as searchable text alongside the contextual fields.

You can route stdlib logging through structlog using structlog.stdlib.LoggerFactory() — useful when third-party libraries log via stdlib but you want everything to come out in your structured pipeline. The configuration takes 5 minutes to set up; the docs have a complete recipe.

Take a script with a few log.info(f"...") calls. Convert them to structlog: pass each variable as a keyword instead of formatting it in. Configure JSONRenderer. Pipe the output through jq '.' and watch how much easier it is to query.
