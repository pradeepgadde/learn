importlib: dynamic imports and plugin patterns

Load modules by string name at runtime. The right way to build plugin systems, driver registries, and lazy loaders.

6 Apr 2026 • ~3 min read • Python 3.8+
TL;DR

importlib.import_module("pkg.mod") imports a module by string name. Use it for plugin systems where the module name comes from config. Use importlib.metadata.entry_points() to discover installed plugins from third-party packages. Both are the approved alternatives to __import__ and dynamic exec.

import_module: the clean dynamic import

pythonimport importlib

# Equivalent to: from myapp.drivers import cisco
mod = importlib.import_module("myapp.drivers.cisco")

# Relative import within a package
mod = importlib.import_module(".cisco", package="myapp.drivers")

# Load by name from config
driver_name = "cisco"                     # could come from TOML/env
mod = importlib.import_module(f"myapp.drivers.{driver_name}")
driver = getattr(mod, "CiscoDriver")()

Plugin registry from config

python# drivers/
# ├── __init__.py
# ├── cisco.py      → class CiscoDriver
# ├── arista.py     → class AristaDriver
# └── juniper.py    → class JuniperDriver

def load_driver(vendor: str):
    try:
        mod = importlib.import_module(f".{vendor}", "myapp.drivers")
    except ModuleNotFoundError:
        raise ValueError(f"No driver for vendor '{vendor}'")
    class_name = f"{vendor.capitalize()}Driver"
    return getattr(mod, class_name)

DriverClass = load_driver("cisco")
driver      = DriverClass(host="10.0.0.1")

Entry points: discover installed plugins

The modern pattern for third-party plugins: the plugin registers an entry point in its pyproject.toml; your app discovers them at runtime via importlib.metadata. 

toml# In the plugin's pyproject.toml:
[project.entry-points."myapp.drivers"]
vendor-x = "vendor_x_driver:XDriver"
pythonfrom importlib.metadata import entry_points

# Discover all installed drivers
drivers = {}
for ep in entry_points(group="myapp.drivers"):
    drivers[ep.name] = ep.load()    # imports and returns the class

# Any package that declares the entry point is auto-discovered
# — no config change required in the host app

This is how pytest discovers plugins, how Celery finds task modules, how click finds extensions. Install a package; it just works.

importlib.resources: read package data files

pythonfrom importlib.resources import files

# Read a file bundled inside a package (works in wheels too)
text = (files("myapp") / "data" / "templates.json").read_text()
# Replaces the old pkg_resources.resource_string() pattern

importlib.util: introspect and spec-load

pythonimport importlib.util

# Check if a module is available without importing it
def is_installed(name: str) -> bool:
    return importlib.util.find_spec(name) is not None

is_installed("httpx")    # True if httpx is installed
is_installed("numpy")    # True or False
python# Load a .py file from an arbitrary path
spec   = importlib.util.spec_from_file_location("plugin", "/tmp/myplugin.py")
module = importlib.util.module_from_spec(spec)
spec.loader.exec_module(module)

Loading arbitrary .py files from disk with spec_from_file_location executes them. Only do this for files you trust. Plugin systems that load third-party packages via entry points are safer — the user consciously installed the package.

Add an entry point to a small package you own. In the consuming app, use entry_points(group="your.group") to discover and load it. Install the package in dev mode (uv pip install -e .), verify discovery works, then uninstall it and verify it disappears from the list. That's the full plugin lifecycle in five minutes.
