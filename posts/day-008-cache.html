@cache: memoization in one line

One decorator turns a slow pure function into a fast one. The cache lives for the lifetime of the process.

08 Jan 2026 • ~3 min read • Python 3.9+
TL;DR

Decorate any pure function with @functools.cache and identical calls return instantly from a dict. Use @lru_cache(maxsize=N) when memory matters. Both require hashable arguments — that's the only catch.

Memoization — remembering a function's output so you don't recompute it — used to be a paragraph of bookkeeping code. Since Python 3.9, it's one line.

The textbook example, made fast

pythonfrom functools import cache

@cache
def fib(n):
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)

print(fib(100))     # instant; without @cache, effectively forever

Without the decorator, fib(100) would make astronomically many redundant calls. With it, each n is computed once and looked up forever after.

Real-world use: expensive lookups

The textbook case is recursion, but the everyday case is anything that's slow and called repeatedly with the same input — a network probe, a regex compile, a config parse, a DB lookup of stable reference data. 

pythonfrom functools import cache
import socket

@cache
def resolve(hostname):
    return socket.gethostbyname(hostname)

# Hits DNS once per hostname, even if called a thousand times
for host in log_lines:
    ip = resolve(host)

cache vs lru_cache

@cache grows without bound. Fine for a process that sees a finite set of inputs. For long-lived processes with unbounded inputs, use @lru_cache(maxsize=N) — once full, it evicts the least-recently-used entry. 

pythonfrom functools import lru_cache

@lru_cache(maxsize=1024)
def render_thumbnail(image_id, size):
    ...

Inspecting and clearing

Both decorators expose cache_info() and cache_clear() on the wrapped function. Useful for benchmarking and for tests. 

pythonfib(100)
print(fib.cache_info())
# CacheInfo(hits=98, misses=101, maxsize=None, currsize=101)

fib.cache_clear()

Two real gotchas. First, arguments must be hashable — pass a tuple, not a list. Second, the cache is per-process and lives until you clear it; don't use @cache on a method whose result depends on mutable state, or you'll cache stale answers and lose hours debugging.

Profile a script you have. Find the function that gets called most often with the same inputs. Add @cache. Re-run. The speedup is often dramatic and the diff is one line.
