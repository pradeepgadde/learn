dataclasses advanced: field(), __post_init__, InitVar

Beyond the basics. field() for defaults and metadata, __post_init__ for derived values, InitVar for init-only arguments, frozen=True for immutable value objects.

3 Apr 2026 • ~3 min read • Python 3.10+
TL;DR

Use field(default_factory=list) for mutable defaults. Use __post_init__ to derive fields or validate after construction. Use InitVar[T] for constructor arguments that don't become fields. Use @dataclass(frozen=True) for hashable, immutable objects.

field(): the right way for mutable defaults

pythonfrom dataclasses import dataclass, field

@dataclass
class Site:
    name:    str
    devices: list[str] = field(default_factory=list)
    tags:    dict[str, str] = field(default_factory=dict)
    active:  bool = True

# This is the WRONG way — all instances share one list:
# devices: list[str] = []   # raises ValueError at class definition

Dataclasses catch the mutable-default mistake at class-definition time and raise ValueError. The fix is always field(default_factory=...).

field() options

pythonfrom dataclasses import field

x: int = field(default=0)
x: int = field(default_factory=int)   # same
x: int = field(repr=False)            # hide from __repr__
x: int = field(compare=False)          # exclude from __eq__/__lt__
x: int = field(hash=False)             # exclude from __hash__
x: int = field(init=False)             # set in __post_init__ only
x: int = field(metadata={"unit": "ms"}) # arbitrary metadata dict

__post_init__: derived values and validation

pythonfrom ipaddress import IPv4Network

@dataclass
class Subnet:
    cidr:    str
    network: IPv4Network = field(init=False, repr=False)

    def __post_init__(self):
        # Runs after __init__, so all init fields are set
        self.network = IPv4Network(self.cidr, strict=False)
        if self.network.prefixlen < 8:
            raise ValueError(f"Prefix too broad: {self.cidr}")

s = Subnet("10.0.0.0/24")
s.network.num_addresses   # 256

InitVar: constructor-only arguments

Sometimes you want a parameter in __init__ that drives some logic but shouldn't become a field. 

pythonfrom dataclasses import dataclass, field, InitVar

@dataclass
class HashedSecret:
    password: InitVar[str]              # in __init__ but not a field
    salt:     str
    digest:   str = field(init=False)

    def __post_init__(self, password: str):
        import hashlib
        self.digest = hashlib.sha256(
            (self.salt + password).encode()
        ).hexdigest()

s = HashedSecret(password="secret", salt="abc")
# s has .salt and .digest — password is NOT stored

frozen=True: hashable, immutable

python@dataclass(frozen=True)
class Point:
    x: float
    y: float

p = Point(1.0, 2.0)
hash(p)                  # works — can use as dict key or set member
p.x = 3.0                 # FrozenInstanceError

order=True: sortable objects

python@dataclass(order=True)
class Version:
    major: int
    minor: int
    patch: int

sorted([Version(1,10,0), Version(2,0,0), Version(1,9,5)])
# [Version(1,9,5), Version(1,10,0), Version(2,0,0)]

dataclasses.replace(): immutable updates

pythonfrom dataclasses import replace

v1 = Version(1, 0, 0)
v2 = replace(v1, minor=1)   # Version(1, 1, 0) — v1 unchanged

For very complex data models, pydantic (Day 032, Day 062) gives you all of this plus JSON serialisation, custom validators, and strict type coercion. Reach for plain dataclasses when you want zero dependencies; reach for pydantic when you need the full machinery.

Find a dataclass in your code that uses a list or dict as a field default without field(default_factory=...). Fix it. Then add a __post_init__ that validates one invariant — something like "port must be between 1 and 65535". These two changes alone prevent a class of subtle bugs.
