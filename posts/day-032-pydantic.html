pydantic: config validation that pays off

Turn dicts into validated models with type coercion, defaults, and clear errors. The right tool for every place data enters your program.

01 Feb 2026 • ~3 min read • Python 3.8+
TL;DR

Subclass BaseModel, declare fields with type annotations, and pydantic validates on construction. Use BaseSettings for config that pulls from env vars. Use Field(...) for constraints. Validation errors come pre-formatted and pinpoint exactly which field is wrong.

Dataclasses (Day 007) hold data; pydantic validates it. Anywhere data enters your program from outside — config files, API responses, env vars, request bodies — pydantic catches malformed input at the boundary instead of letting it propagate as a confusing crash three layers in.

The minimal model

pythonfrom pydantic import BaseModel

class DeviceConfig(BaseModel):
    hostname: str
    port:     int = 22
    vlans:    list[int] = []

cfg = DeviceConfig(hostname="sw-01", port="2222", vlans=["10", "20"])
print(cfg.port)    # 2222 (int — coerced from str)
print(cfg.vlans)   # [10, 20] (each str coerced to int)

The "2222" string became an int automatically. So did each VLAN. Pydantic v2 is strict by default about types but accepts safe coercions. Pass strict=True per-field if you want to disable even safe coercions.

Validation errors that point at the problem

pythonfrom pydantic import ValidationError

try:
    DeviceConfig(hostname="sw-01", port="http", vlans="all")
except ValidationError as e:
    print(e)
2 validation errors for DeviceConfig port Input should be a valid integer, unable to parse string as an integer [type=int_parsing, input_value='http'] vlans Input should be a valid list [type=list_type, input_value='all']

Both problems caught in one pass, both pointing at the exact field and value. Compare to the typical "TypeError: int() argument must be..." with no field name.

Constraints with Field

pythonfrom pydantic import BaseModel, Field, IPvAnyAddress

class RetryConfig(BaseModel):
    max_attempts: int   = Field(default=3, ge=1, le=10)
    backoff:      float = Field(default=0.5, gt=0)
    target:       IPvAnyAddress

ge/le/gt/lt for numeric ranges, min_length/max_length for collections, pattern for regex on strings. Built-in types like IPvAnyAddress, EmailStr, HttpUrl, SecretStr validate semantically.

Settings from env vars

For config that comes from environment variables — the 12-factor app pattern — pydantic-settings (a separate package since v2) reads them automatically. 

pythonfrom pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_prefix="APP_", env_file=".env")

    db_url:      str
    debug:       bool = False
    workers:     int  = 4

# Reads APP_DB_URL, APP_DEBUG, APP_WORKERS from env or .env
settings = Settings()

Booleans accept "true", "1", "yes", "on" (case-insensitive) for true. Lists accept JSON-encoded strings or comma-separated values depending on configuration. The fallback chain is explicit and predictable.

Loading from JSON or dicts

pythonimport json

# From dict
cfg = DeviceConfig.model_validate({"hostname": "sw-01", "port": 22})

# Directly from a JSON string
cfg = DeviceConfig.model_validate_json('{"hostname":"sw-01"}')

# Round-trip back to JSON or dict
cfg.model_dump_json(indent=2)
cfg.model_dump()    # dict

Pydantic v2 (released 2023) is dramatically faster than v1 — the core is now in Rust. Method names changed too (parse_obj → model_validate, dict() → model_dump()). If you're starting fresh, use v2. If you're on v1, don't migrate unless you need the speed.

Pick a JSON or YAML config file your scripts read today. Define a BaseModel for it. Replace the manual config["foo"] lookups with attribute access on the model. The first time pydantic catches a typo in a coworker's config edit before deploy, you'll send me a thank-you note.
