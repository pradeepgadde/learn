Alembic: schema migrations that work

Generate and apply SQL migrations from your SQLAlchemy models. Autogenerate, upgrade, downgrade — and the conventions that keep migrations sane.

05 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

alembic init alembic sets up the directory. Point env.py at your Base.metadata. Then alembic revision --autogenerate -m "..." creates a migration from model diffs. alembic upgrade head applies them. Always review autogenerated migrations before committing.

Models change. Tables don't change themselves. Alembic is the standard SQLAlchemy companion for producing and tracking schema changes — the Django migrations app, but for the rest of us.

Initial setup

bashuv add alembic
alembic init alembic
# Creates alembic/, alembic.ini, env.py

Two edits to wire it up: 

python# alembic/env.py
from myapp.models import Base     # your DeclarativeBase

target_metadata = Base.metadata      # enables autogenerate
ini# alembic.ini
sqlalchemy.url = postgresql+psycopg://user:pw@host/dbname
# Better: read from env var in env.py with os.environ

Generate a migration

bashalembic revision --autogenerate -m "add devices table"
# Creates alembic/versions/abc123_add_devices_table.py

Open the file. It contains upgrade() and downgrade() functions with operations like op.create_table, op.add_column, op.create_index. Read it like a code review before commit. Autogenerate is good but not infallible — it misses some changes (renames, server defaults, certain index modifications) and occasionally generates wrong diffs.

Apply and roll back

bash# Apply all pending migrations
alembic upgrade head

# Roll back one
alembic downgrade -1

# Apply or roll back to a specific revision
alembic upgrade abc123
alembic downgrade abc123

# See where the DB is right now
alembic current

# See history
alembic history --verbose

What autogenerate misses

Some changes need manual edits:

  • Renames — autogenerate sees a column drop and an add, not a rename. Edit to use op.alter_column(... new_column_name=...).
  • Data migrations — if you split a column, you also need to migrate the data. Add op.execute("UPDATE ...") between schema operations.
  • Default value changes — especially server-side defaults. Always check.

Manual migration template

pythonfrom alembic import op
import sqlalchemy as sa

# revision identifiers
revision = "abc123"
down_revision = "def456"

def upgrade():
    op.add_column("devices", sa.Column("site_id", sa.Integer(),
                                       nullable=True))
    op.create_foreign_key("fk_devices_site", "devices", "sites",
                          ["site_id"], ["id"])

def downgrade():
    op.drop_constraint("fk_devices_site", "devices", type_="foreignkey")
    op.drop_column("devices", "site_id")

Naming conventions: name your constraints

Without explicit names, Alembic generates op.f("ix_blah_blah") calls that depend on database conventions. Set a naming convention on your MetaData and autogenerate produces stable names. 

pythonfrom sqlalchemy import MetaData

naming_convention = {
    "ix":  "ix_%(column_0_label)s",
    "uq":  "uq_%(table_name)s_%(column_0_name)s",
    "ck":  "ck_%(table_name)s_%(constraint_name)s",
    "fk":  "fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s",
    "pk":  "pk_%(table_name)s",
}

class Base(DeclarativeBase):
    metadata = MetaData(naming_convention=naming_convention)

Now constraint and index names match across environments and don't change between Postgres and SQLite. Sets up future you for a happy life.

Production discipline

  • One logical change per migration — smaller migrations roll back cleanly.
  • Test downgrade — in dev, alembic downgrade -1 && alembic upgrade +1 ensures both directions work.
  • Run before deploy — not on app start. App start should fail loudly if the schema is behind, not silently mutate.
  • Backwards-compatible migrations for zero-downtime deploys: split column renames, add-then-deprecate-then-drop. The "expand and contract" pattern.

Be very careful with autogenerated drops. If a column moved to a different file Alembic doesn't see, autogenerate will generate a drop. Double-check every drop in the generated file before you commit.

On any small project: install alembic, point it at your models, generate an initial migration with alembic revision --autogenerate -m "init", apply it. Add a column, generate a migration, apply, downgrade, upgrade again. The full cycle takes ten minutes and removes the fear of schema changes.
