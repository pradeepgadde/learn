Graceful shutdown: SIGTERM the right way

When the orchestrator says "stop", you have 30 seconds. Drain in-flight requests, close connections, exit cleanly — in sync and async.

13 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

SIGTERM is the polite "please stop"; SIGKILL is the "you had your chance". On SIGTERM: stop accepting new work, finish what's in flight, close clients and pools, then exit. uvicorn and gunicorn do the HTTP-server side correctly — you handle cleanup in a lifespan context or signal handler.

Every container orchestrator (Kubernetes, ECS, Nomad, systemd) sends SIGTERM, waits a grace period, then SIGKILL. Code that doesn't handle SIGTERM gets killed mid-request, drops connections, leaves transactions half-committed. Code that does gets a clean roll.

FastAPI: the lifespan context

The right place for shutdown work in FastAPI is the same place we set up startup work (Day 061): the lifespan context manager. Code after the yield runs on shutdown. 

pythonfrom contextlib import asynccontextmanager
import httpx, structlog

log = structlog.get_logger()

@asynccontextmanager
async def lifespan(app):
    # Startup
    app.state.http = httpx.AsyncClient()
    app.state.db   = await connect_db()
    log.info("startup_complete")
    yield
    # Shutdown — runs when SIGTERM is received
    log.info("shutdown_starting")
    await app.state.http.aclose()
    await app.state.db.close()
    log.info("shutdown_complete")

uvicorn handles the HTTP layer for you: stops accepting new connections, waits for in-flight requests to finish (up to --timeout-graceful- shutdown seconds), then triggers your lifespan shutdown.

Plain scripts: the signal handler

Long-running scripts — pollers, consumers, periodic syncs — need to register a signal handler explicitly. 

pythonimport signal, time

_should_stop = False

def _handle_sigterm(signum, frame):
    global _should_stop
    _should_stop = True

signal.signal(signal.SIGTERM, _handle_sigterm)
signal.signal(signal.SIGINT,  _handle_sigterm)

while not _should_stop:
    poll_once()
    time.sleep(5)

# Cleanup runs after the loop exits
close_connections()

The flag pattern is the safe way — signal handlers run between bytecode instructions and can interrupt almost anything, so keep them trivial.

Async scripts: cancel the main task

pythonimport asyncio, signal

async def main():
    try:
        while True:
            await poll_once()
            await asyncio.sleep(5)
    except asyncio.CancelledError:
        # SIGTERM arrived — clean up
        await close_connections()
        raise

async def _run():
    task = asyncio.create_task(main())
    loop = asyncio.get_running_loop()
    for sig in (signal.SIGTERM, signal.SIGINT):
        loop.add_signal_handler(sig, task.cancel)
    try:
        await task
    except asyncio.CancelledError:
        pass

asyncio.run(_run())

SIGTERM cancels the main task, the CancelledError propagates to your cleanup code, then the program exits. Use add_signal_handler instead of signal.signal in async code — plays nicely with the loop.

Don't suppress CancelledError

A surprisingly common mistake: 

python# BAD — swallows shutdown signals
try:
    await do_work()
except Exception:
    log.exception("oops")
    # CancelledError matches Exception in 3.7; in 3.8+ it doesn't,
    # but watch out for `except BaseException` patterns

Catch the specific exceptions you want to handle. CancelledError should propagate so the loop can actually wind down. Same goes for KeyboardInterrupt in sync code.

The grace period budget

Kubernetes default terminationGracePeriodSeconds is 30. Within that window: drain HTTP, finish the in-flight DB transaction, push final metrics, flush logs. If you need more time (long-running jobs), bump the grace period; don't try to skip the cleanup. 

yamlspec:
  terminationGracePeriodSeconds: 60
  containers:
    - name: app
      lifecycle:
        preStop:
          exec:
            command: ["/bin/sh", "-c", "sleep 5"]   # let LBs notice

The preStop sleep gives the load balancer time to remove the pod from rotation before SIGTERM arrives — a few seconds of sleep here saves a lot of in-flight failed requests.

Workers and queues

For RQ, Celery, and similar: most have built-in SIGTERM handling that finishes the current task and exits. The thing to verify is that your tasks are short-ish, or you've configured a long enough grace period to let them complete. Tasks that always exceed grace period need to be checkpoint-able and idempotent.

SIGKILL is non-negotiable — you can't catch it. The kernel just ends the process. Anything that must be done at shutdown should also be safe to recover from on next startup. Treat the grace period as best-effort, not a guarantee.

Run kill -TERM <pid> against a local copy of your service and watch the shutdown logs. If you see in-flight requests getting cut off, or pools left unclosed, you have shutdown work to do. The same script adapts trivially to test docker SIGTERM behaviour.
