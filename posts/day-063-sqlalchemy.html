SQLAlchemy 2.0: the modern Python ORM

Unified API, proper type hints, and the select() statement that replaces the old query. The default ORM choice for Python in 2026.

04 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Declare models with Mapped[X] annotations on a class inheriting from DeclarativeBase. Open a Session per request or operation. Build queries with select(Model).where(...) and execute with session.execute(). Commit transactions inside with session.begin():.

SQLAlchemy 2.0 (released 2023) was a major cleanup. The legacy Query API still exists, but new code should use the unified select() API and the typed Mapped annotation style. Editor support and mypy integration finally feel right.

Setup

pythonfrom sqlalchemy import create_engine, String, select
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, Session

engine = create_engine("sqlite:///app.db", echo=False)

class Base(DeclarativeBase):
    pass

class Device(Base):
    __tablename__ = "devices"

    id:       Mapped[int] = mapped_column(primary_key=True)
    hostname: Mapped[str] = mapped_column(String(64), unique=True)
    role:     Mapped[str] = mapped_column(String(32))

Base.metadata.create_all(engine)

Mapped[str] tells the editor and mypy that device.hostname is a string. mapped_column attaches DB constraints. The two annotations split — Python types vs storage configuration — is cleaner than the old single-call style.

Insert and query

pythonwith Session(engine) as session:
    with session.begin():               # transaction
        session.add_all([
            Device(hostname="sw-01", role="access"),
            Device(hostname="sw-02", role="core"),
        ])

    # Read
    stmt     = select(Device).where(Device.role == "access")
    access   = session.scalars(stmt).all()

    # By primary key
    d        = session.get(Device, 1)

session.scalars() returns the model instances; session.execute() returns rows of columns. Pick by what you actually want.

Update and delete

pythonwith Session(engine) as session, session.begin():
    # Mutation through the ORM (loads first, then writes on commit)
    d = session.get(Device, 1)
    d.role = "distribution"

    # Bulk update without loading rows
    from sqlalchemy import update
    session.execute(
        update(Device).where(Device.role == "old").values(role="new")
    )

Relationships

pythonfrom sqlalchemy import ForeignKey
from sqlalchemy.orm import relationship

class Site(Base):
    __tablename__ = "sites"
    id:      Mapped[int] = mapped_column(primary_key=True)
    name:    Mapped[str]
    devices: Mapped[list["Device"]] = relationship(back_populates="site")

class Device(Base):
    __tablename__ = "devices"
    id:       Mapped[int] = mapped_column(primary_key=True)
    hostname: Mapped[str]
    site_id:  Mapped[int] = mapped_column(ForeignKey("sites.id"))
    site:     Mapped["Site"]    = relationship(back_populates="devices")

back_populates wires the two sides together. site.devices and device.site stay in sync automatically when you assign either side.

Eager loading: avoid N+1

Accessing device.site in a loop triggers a separate query per device — the classic N+1. selectinload fixes it. 

pythonfrom sqlalchemy.orm import selectinload

stmt = select(Device).options(selectinload(Device.site))
for d in session.scalars(stmt):
    print(d.hostname, d.site.name)   # no extra queries

Async, in one swap

Replace create_engine with create_async_engine and Session with AsyncSession. The query API stays almost identical. 

pythonfrom sqlalchemy.ext.asyncio import create_async_engine, AsyncSession

engine = create_async_engine("sqlite+aiosqlite:///app.db")

async with AsyncSession(engine) as session:
    result = await session.scalars(select(Device))
    devices = result.all()

Pairs naturally with FastAPI: open an AsyncSession in a Depends dependency that yields to the path function and commits on exit. The standard "session per request" pattern.

Take a script that does manual SQL with sqlite3 or psycopg. Re-write the read path using SQLAlchemy 2.0 with Mapped annotations. The first query is more code; the tenth one is far less.
