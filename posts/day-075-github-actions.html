GitHub Actions for Python CI

A solid Python CI pipeline in 30 lines: matrix testing, uv caching, ruff, mypy, pytest. Plus the patterns for building and pushing Docker images.

16 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Use actions/setup-python with caching (or the official astral-sh/setup-uv for uv). Matrix-test across the Python versions you support. Run ruff, mypy, pytest as separate steps so failures are obvious. Build and push Docker images with docker/build-push-action on tags or main.

Boring CI is good CI. It runs the same things you run locally, with the same tools, on every push. No magic. Here's a pipeline that covers the essentials.

The pipeline

yaml# .github/workflows/ci.yml
name: ci

on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        python-version: ["3.11", "3.12", "3.13"]
    steps:
      - uses: actions/checkout@v4

      - name: Install uv
        uses: astral-sh/setup-uv@v3
        with:
          enable-cache: true

      - name: Set up Python
        run: uv python install ${{ matrix.python-version }}

      - name: Install dependencies
        run: uv sync --frozen

      - name: Lint (ruff)
        run: uv run ruff check .

      - name: Format check (ruff)
        run: uv run ruff format --check .

      - name: Type check (mypy)
        run: uv run mypy src

      - name: Test
        run: uv run pytest --cov=src --cov-report=xml

      - name: Upload coverage
        if: matrix.python-version == '3.12'
        uses: codecov/codecov-action@v4
        with:
          files: ./coverage.xml

Notes worth taking:

  • fail-fast: false — one Python version failing shouldn't cancel the others. You want to see all the failures at once, not whack-a-mole.
  • Each tool is a separate step — the GitHub UI shows red ✗ on the specific failure, not buried in a "lint and test" megastep.
  • Coverage uploaded once — only from one matrix entry, otherwise codecov merges multiple reports awkwardly.
  • Pin action versions to majors (@v4) for security and stability. Pin to SHAs (@a1b2c3d) for stricter supply-chain control.

The pip/Poetry version

If you're not on uv yet, the pip version is just a few lines different: 

yaml      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
          cache: 'pip'

      - run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt -r requirements-dev.txt

The cache: pip shortcut handles wheel caching out of the box. Builds drop from ~60 seconds to ~10 once warm.

Concurrency: cancel old runs

When someone pushes three commits in two minutes, you don't need three CI runs. Cancel earlier runs of the same workflow on the same branch. 

yamlconcurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

Building and pushing the Docker image

yaml  build-image:
    needs: test
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4

      - uses: docker/setup-buildx-action@v3

      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - uses: docker/metadata-action@v5
        id: meta
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=sha
            type=raw,value=latest

      - uses: docker/build-push-action@v6
        with:
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

cache-from: type=gha + cache- to persists the BuildKit layer cache in GitHub's actions cache. Image rebuilds on small changes are fast.

Secrets and OIDC

Don't paste long-lived AWS keys into Actions secrets. Use OIDC trust between GitHub and your cloud, request short-lived credentials at runtime with aws-actions/configure-aws-credentials or the GCP/Azure equivalents. The setup is a bit fiddly the first time and the result is much safer.

Branch protection

CI is only as useful as the rules around it. In repo Settings → Branches: require the test job to pass before merge to main, require PR review, optionally require linear history. CI becomes the gate, not a suggestion.

For periodic jobs (daily dependency updates, weekly security scans), use schedule: triggers. Pair with Dependabot for automatic dependency PRs — let it open them, let CI test them, you only have to look when something breaks.

Drop the workflow above into .github/workflows/ci.yml on a test repo. Push something. Watch it run. Then break a test on purpose, push again, and watch the red ✗ light up. Once green-on-main is normal in your repo, every PR has a built-in second pair of eyes.
