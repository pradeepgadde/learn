Exception groups: many failures, one raise

Concurrent code can fail in multiple ways at once. ExceptionGroup and except* let you report and handle all of them — not just the first one.

09 Feb 2026 • ~3 min read • Python 3.11+
TL;DR

ExceptionGroup wraps multiple exceptions into one. except* matches members by type and runs the handler on the matched subset. The standard way to surface failures from TaskGroup and any operation that can produce multiple errors at once.

Until 3.11, raising in Python was an all-or-nothing thing — one exception, one place. Async code broke this model: ten tasks could fail simultaneously and you'd only see one. PEP 654 introduced exception groups to handle the rest properly.

The grouping

pythonerrors = []
for device in devices:
    try:
        push_config(device)
    except Exception as e:
        errors.append(e)

if errors:
    raise ExceptionGroup("some pushes failed", errors)

One raise carries every individual failure. The traceback shows all of them, not just the first. Callers can decide what to do with the whole bundle.

The new syntax: except*

Inside an exception group, you might want to handle connection errors one way and validation errors another. except* matches members by type and runs the handler on just the matched subset. 

pythontry:
    push_to_fleet(devices)
except* ConnectionError as eg:
    for err in eg.exceptions:
        log.warning("unreachable: %s", err)
except* ValueError as eg:
    for err in eg.exceptions:
        log.error("bad config: %s", err)

Both except* blocks can run in the same try statement — that's the difference from regular except, which only matches one clause. Each handler receives an ExceptionGroup containing only the matching exceptions, in their original tree positions.

Where you'll meet them: TaskGroup

asyncio.TaskGroup (Day 039) raises an ExceptionGroup when one or more tasks fail. The pattern is symmetric. 

pythontry:
    async with asyncio.TaskGroup() as tg:
        for host in hosts:
            tg.create_task(probe(host))
except* TimeoutError as eg:
    log.warning("%d devices timed out", len(eg.exceptions))
except* ConnectionRefusedError as eg:
    log.error("%d devices refused", len(eg.exceptions))

Filtering and splitting

For programmatic handling outside except*, an ExceptionGroup has a .split(predicate) method that partitions it into matched and unmatched groups. 

pythontry:
    do_batch()
except ExceptionGroup as eg:
    retryable, fatal = eg.split(TimeoutError)
    if retryable:
        log.info("will retry %d failures", len(retryable.exceptions))
    if fatal:
        raise fatal          # re-raise non-retryable ones

split returns a (matched, rest) tuple, either of which may be None if empty.

Backward compatibility

except* X only works on 3.11+. If you need to run on older Pythons, you can still raise ExceptionGroup using the exceptiongroup backport package, but the syntax stays except ExceptionGroup as eg: with manual iteration.

Don't use a regular except Exception to catch an exception group. It does match (because ExceptionGroup inherits from Exception) but you lose the per-type handling. Use except* when you want per-type behavior, or except ExceptionGroup as eg when you want to inspect the whole bundle.

Convert the URL checker from yesterday's challenge to use TaskGroup instead of gather(return_exceptions=True). Catch the resulting ExceptionGroup with two except* clauses — one for timeout errors, one for everything else. Watch the failure summary become much more useful.
