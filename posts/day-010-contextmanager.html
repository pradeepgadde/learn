Custom with-blocks in 5 lines

@contextmanager turns any generator into a real with block. Setup before yield, teardown after. That's it.

11 May 2026 • ~3 min read • Python 3.2+
TL;DR

Decorate a generator with @contextlib.contextmanager, put your setup code before yield, your cleanup code after. You now have a context manager — no __enter__ / __exit__ class boilerplate required.

The with statement is one of Python's quietest wins. Files, locks, transactions, timers — anything that needs paired setup and cleanup belongs inside a with block. The contextlib module makes writing your own embarrassingly easy.

The decorator pattern

pythonfrom contextlib import contextmanager
import time

@contextmanager
def timer(label):
    start = time.perf_counter()
    yield
    elapsed = time.perf_counter() - start
    print(f"{label}: {elapsed * 1000:.1f}ms")

with timer("data load"):
    rows = load_million_rows()
data load: 482.3ms

The structure to internalize: everything before yield runs on enter, everything after runs on exit. That's the whole API.

Yielding a value to the as clause

Whatever you yield becomes the value bound by with ... as x. 

python@contextmanager
def cd(path):
    prev = Path.cwd()
    os.chdir(path)
    try:
        yield path
    finally:
        os.chdir(prev)

with cd("/tmp") as here:
    print(f"Now in {here}")
# Back in the original directory, even if the block raised

The try/finally is essential when cleanup must happen even on exception. Without it, an error inside the with would leave you in the wrong directory.

Suppressing exceptions cleanly

For the "ignore this specific exception" pattern, skip the custom context manager entirely — contextlib.suppress is built in. 

pythonfrom contextlib import suppress
from pathlib import Path

# Idempotent delete
with suppress(FileNotFoundError):
    Path("cache.tmp").unlink()

Reads better than try / except / pass, and the intent is unmistakable.

ExitStack for dynamic with-blocks

Need to manage a variable number of context managers? You can't write with a, b, c: when the count is only known at runtime. ExitStack solves this elegantly. 

pythonfrom contextlib import ExitStack

with ExitStack() as stack:
    files = [stack.enter_context(open(p)) for p in paths]
    # All files closed in reverse order on exit
    for f in files:
        process(f)

You can still write a context manager as a class with __enter__ and __exit__ — and it's the right call when you need to attach state or subclass. For one-off setup/teardown, the decorator wins on line count and clarity every time.
