bisect: binary search on sorted lists

insort keeps a list sorted as you add. bisect_left finds where something belongs in O(log n). The textbook way to look up by range.

15 Feb 2026 • ~3 min read • Python 3.0+
TL;DR

bisect.insort(lst, x) inserts x into a sorted list, keeping it sorted. bisect_left(lst, x) tells you the index where it would go. Combined, they're the right primitive for sorted-list maintenance, range queries, and bucket lookups.

Sorting a list is O(n log n). Maintaining a sorted list as items arrive doesn't have to be — bisect.insort finds the right position in O(log n) and inserts. The list stays sorted at all times.

Insertion that keeps order

pythonimport bisect

scores = []
for s in [42, 17, 88, 23, 91, 5]:
    bisect.insort(scores, s)

print(scores)
# [5, 17, 23, 42, 88, 91]

Compare to scores.append(s); scores.sort() on every insert — that's O(n log n) per insertion, so O(n² log n) for n inserts. insort is O(n) per insertion (the list shift dominates the binary search), so O(n²) overall. Better, especially when the list grows.

Range queries with bisect_left and bisect_right

Given a sorted list of timestamps, find all events between two times. Two binary searches replace a full scan. 

pythontimestamps = [1700, 1720, 1730, 1745, 1800, 1815, 1900]

lo = bisect.bisect_left(timestamps,  1730)
hi = bisect.bisect_right(timestamps, 1815)

in_range = timestamps[lo:hi]
# [1730, 1745, 1800, 1815]

The difference between the two: bisect_left returns the leftmost position where x could go (so for 1730, it returns the index of 1730). bisect_right returns the rightmost (so for 1815, it returns the index after 1815). Together they bracket the inclusive range cleanly.

The bucket-lookup pattern

Convert numeric grades to letter grades, latencies to SLO buckets, packet sizes to histogram bins — all the same shape: which interval does this value fall into? 

pythoncutoffs = [60, 70, 80, 90]
grades  = ["F", "D", "C", "B", "A"]

def grade(score):
    return grades[bisect.bisect_right(cutoffs, score)]

grade(85)   # 'B'
grade(90)   # 'A'
grade(59)   # 'F'

Three lines, no if/elif/else chain. Add a new bucket by editing the cutoffs list.

Searching with a key (3.10+)

On Python 3.10, bisect learned a key= parameter, matching the rest of the sorting API. Sorted by some attribute? Search by it too. 

python# sessions sorted by .start_time
i = bisect.bisect_left(
    sessions,
    target_time,
    key=lambda s: s.start_time,
)

Membership in a sorted set: bisect vs set

For "is x in the collection?", a set is O(1) and clearer. Use bisect when you also need order — range queries, bucket lookups, "what's the largest item ≤ x?".

For a more polished sorted-collection API, the third-party sortedcontainers package gives you SortedList, SortedDict, and SortedSet with O(log n) inserts and deletes (no list shift). Pure Python, faster than what you'd write on top of bisect for big collections.

Find a place in your code where you do list.append(x); list.sort() in a loop. Replace with bisect.insort(list, x). Faster, clearer, and the list is always in a valid sorted state — useful if anyone else reads it mid-update.
