weakref: caches that don't leak

Reference an object without keeping it alive. WeakValueDictionary is the cache that auto-evicts as soon as the rest of the program is done with the value.

13 Feb 2026 • ~3 min read • Python 3.0+
TL;DR

A weakref doesn't count toward the garbage collector's reference count, so the referenced object can still be freed. WeakValueDictionary is a dict whose entries vanish when nothing else holds the value. The right cache when you want "remember this if we still need it, otherwise forget it."

functools.cache (Day 008) keeps every value forever — great until the cache eats your memory. Weak references give you a different shape: cache hits are fast, but as soon as the rest of the program drops its reference to a value, the cache lets it go too.

The basic primitive

pythonimport weakref

class Connection:
    def __init__(self, host):
        self.host = host

conn = Connection("sw-01")
ref  = weakref.ref(conn)

ref()       # <Connection 'sw-01'> — call to dereference
del conn
ref()       # None — referent was freed

Two things to internalize: weakrefs are callable (call them to get the object), and they return None once the referent is gone. Always check before using.

WeakValueDictionary: the auto-evicting cache

pythonimport weakref

cache = weakref.WeakValueDictionary()

def get_device(name):
    if name in cache:
        return cache[name]
    d = load_device_from_disk(name)
    cache[name] = d
    return d

# If nothing else holds 'sw-01', the cache entry is silently dropped.
# If something does hold it, subsequent get_device calls return the same object.

This is the deduplication pattern: same name, same instance — for as long as anyone is using it. As soon as the rest of the program is done, memory comes back. Compare to @cache, which would hold every device ever loaded for the entire process lifetime.

WeakSet for "membership without ownership"

Useful for observer/subscriber patterns — the set knows who's listening, but doesn't keep them alive on its own. 

pythonclass EventBus:
    def __init__(self):
        self._subscribers = weakref.WeakSet()

    def subscribe(self, handler):
        self._subscribers.add(handler)

    def emit(self, event):
        for h in self._subscribers:
            h(event)
# Subscribers that go out of scope are removed automatically — no
# memory leak from forgetting to unsubscribe.

Cleanup callbacks: weakref.finalize

Run a function when an object is about to be collected. Useful for closing external resources you can't tie to a normal context manager. 

pythonclass SessionPool:
    def __init__(self):
        self.handle = open_native_pool()
        weakref.finalize(self, close_native_pool, self.handle)
# When the SessionPool instance is collected, close_native_pool(handle)
# runs automatically — even if the program forgot to close it.

Note the second argument: don't reference self from the callback (that creates a cycle). Pass the resource handle directly.

What can be weakly referenced

Most user-defined classes, yes. Built-in types like int, str, list, tuple, dict, no — you'll get TypeError: cannot create weak reference. If you need weakrefs to a built-in, wrap it in a class.

Weakrefs feel magical at first — "I want a reference that doesn't count" — but they're rarely the right answer. The two situations where they shine: deduplication caches where you want to share instances without keeping them alive, and observer patterns where forgetting to unsubscribe is a known leak source. Outside those, prefer explicit ownership.

If you have a class that you instantiate by some natural key (filename, hostname, ID) and want identity to be preserved, swap your dict for a WeakValueDictionary. Watch your process RSS stop growing without bound.
