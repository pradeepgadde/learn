subprocess.run, done right

Four flags turn subprocess.run from a footgun into a tool you'd actually want to use.

13 Jan 2026 • ~3 min read • Python 3.5+
TL;DR

Always pass arguments as a list, not a string. Always pass check=True, capture_output=True, and text=True. Add timeout= for anything touching the network. Five lines, one safe call.

For shelling out to kubectl, git, ssh, or any CLI tool, subprocess.run is the right answer. But the default behavior is unforgiving: no error on failure, no captured output, bytes everywhere. Four flags fix that.

The boilerplate-free template

pythonimport subprocess

result = subprocess.run(
    ["git", "rev-parse", "--short", "HEAD"],
    check=True,
    capture_output=True,
    text=True,
)

commit = result.stdout.strip()
print(f"Building commit {commit}")

What each flag does:

  • list args — not a string. No shell, no quoting bugs, no injection risk.
  • check=True — raises CalledProcessError on a non-zero exit code instead of silently succeeding.
  • capture_output=True — collects stdout and stderr instead of letting them go to your terminal.
  • text=True — decodes output as a string, not bytes.

Why shell=True is almost always wrong

It's tempting to write subprocess.run("ls -la", shell=True). Then someone passes a filename containing a semicolon and your cron job runs rm -rf /. Pass a list instead and the shell never sees your arguments. 

python# Don't — shell injection waiting to happen
subprocess.run(f"grep {pattern} {file}", shell=True)

# Do — pattern and file are passed as discrete arguments
subprocess.run(["grep", pattern, file], check=True)

The only legitimate reason to use shell=True is when you genuinely need a shell feature: pipes, globs, redirection. And usually you can do those in Python instead.

Network calls need a timeout

Anything that hits the network or an unreliable host should have a timeout. Without one, your script can hang forever. 

pythontry:
    result = subprocess.run(
        ["ssh", "router1", "show", "version"],
        check=True, capture_output=True, text=True,
        timeout=10,
    )
except subprocess.TimeoutExpired:
    log.warning("router1 not responding")
except subprocess.CalledProcessError as e:
    log.error(f"command failed: {e.stderr}")

Streaming output for long-running commands

capture_output=True waits for the command to finish. For a long task where you want to see progress (a build, a deploy), let stdout flow through to the terminal by simply not capturing it. 

python# Output streams to your terminal in real time, exit code is checked
subprocess.run(
    ["docker", "build", "-t", "app:latest", "."],
    check=True,
)

Don't capture stdout manually with stdout=subprocess.PIPE unless you also stdin=subprocess.DEVNULL or read the output as it streams. A child producing more than ~64KB without anyone reading the pipe will deadlock. The capture_output=True shortcut handles this correctly.

Find any place in your code where you call os.system, subprocess.call, or subprocess.run(..., shell=True). Convert it to the four-flag template. If it talks to the network, add a timeout.
