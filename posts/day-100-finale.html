One hundred days of Python

100 days. 100 ideas. A map of the ground covered — and what to build with it.

10 Apr 2026 • ~3 min read • Python 3.11+
TL;DR

100 posts, one concept each day. Not exhaustive — every post was a door, not a wall. The measure isn't whether you remember all 100 tools; it's whether you now reach for the right one faster than before.

In January we started with the language itself: f-strings, pathlib, generators, type hints. By February we were hardening production code — pydantic, tenacity, mypy, structlog, concurrent.futures. March brought the web layer: FastAPI, WebSockets, OpenTelemetry, Docker, GitHub Actions, and the full test stack. April closed the loop with stdlib internals: the AST, importlib, the operator module, bytecode.

The map: 100 ideas by track

Language core (Days 1–21): f-strings, pathlib, Counter, dict.get, enumerate, walrus, dataclasses, @cache, batched, contextmanager, match, type hints, subprocess, argparse, JSON/YAML, logging, httpx, pytest, uv, generators, comprehensions.

Networking (Days 22–31): ipaddress, socket, paramiko/netmiko, asyncssh, Jinja2, Scapy, TextFSM, pyATS/Genie, NETCONF/ncclient, rich.

Production hardening (Days 32–59): pydantic, tenacity, structlog, cProfile, csv, sqlite3, concurrent.futures, asyncio.gather, exception groups, ruff, mypy, __slots__, weakref, heapq, bisect, deque, itertools, functools, typer, zoneinfo, hashlib, secrets, tempfile, shutil, glob, re, dataclasses+JSON, advanced pathlib.

Web & observability (Days 60–90): FastAPI, ASGI, pydantic deep cuts, SQLAlchemy 2, Alembic, middleware, WebSockets, background tasks, OpenTelemetry, prometheus_client, Sentry, health checks, graceful shutdown, Docker, multi-stage builds, GitHub Actions CI, pre-commit, pytest fixtures, parametrize, hypothesis, factory_boy, respx, testcontainers, hatch, PyPI publishing, type stubs, dunders, ABCs/Protocols, contextvars, StringIO/BytesIO, IPython.

Stdlib internals & advanced patterns (Days 91–99): tomllib, ast, dataclasses advanced, operator, inspect, importlib, multiprocessing, textwrap/difflib, dis.

What this vocabulary enables

These aren't isolated tricks. They compose. A realistic task: "poll 200 network devices, parse the output, diff against last state, raise an alert if drift exceeds a threshold, publish metrics, and ship a report." A hundred days ago you'd reach for bash and hope. Now you have vocabulary for every part: asyncssh + TextFSM + difflib + pydantic + prometheus_client + structlog + hatch to package it.

What to do next

Reading is the slowest way to learn programming. The fastest: pick one real problem, build it, break it, fix it. Some concrete ideas:

  • Build a config drift detector. Poll devices with asyncssh, parse with TextFSM or Genie, diff against a golden config with difflib, push alerts to a webhook. Every skill from this series applies.
  • Publish a small package. Pick the most useful script you've written, wrap it with typer, package it with hatch, publish to PyPI with trusted publishing. The full workflow in one afternoon.
  • Write a FastAPI service you actually use. Add pydantic models, SQLAlchemy persistence, OpenTelemetry tracing, a Dockerfile, and a GitHub Actions pipeline. Keep it small; keep it running.
  • Contribute to an open-source project. Find a bug or missing test in a library you use. The ast, inspect, and importlib days give you the vocabulary to read someone else's framework internals.

The tools that came up most

If you could only remember ten from this series: pydantic, httpx, structlog, pytest, ruff, pathlib, asyncio, FastAPI, mypy, and uv. Those nine cover 80% of real Python work. Everything else makes that 80% better.

Ninety nine of these posts introduced a tool. This one doesn't. The habit of picking up a new stdlib module or third-party library, spending thirty minutes with the docs and a REPL, and deciding whether it belongs in your toolkit — that's the real skill. Not the 100 names, but the confidence to add number 101 whenever you need it.
