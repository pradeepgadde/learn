itertools: chain, takewhile, groupby

Three itertools functions that earn their keep: chain to flatten, takewhile for streaming filters, groupby for consecutive-run grouping.

17 Feb 2026 • ~3 min read • Python 3.0+
TL;DR

chain(*iterables) stitches sequences together lazily. takewhile(pred, it) stops at the first false. groupby(it, key) groups consecutive items with the same key — sort first if you want SQL-style grouping.

itertools has 20+ functions; most are niche. Three of them come up over and over and replace loops with one-liners that read better.

chain: flatten without copying

pythonfrom itertools import chain

errors   = ["timeout", "refused"]
warnings = ["slow"]
notes    = ["first run", "new device"]

for item in chain(errors, warnings, notes):
    process(item)

Lazy. No new list created, just iteration that flows from one input to the next. For nested-list flattening (one level deep), use chain.from_iterable: 

pythonbatches = [[1, 2], [3], [4, 5, 6]]
list(chain.from_iterable(batches))
# [1, 2, 3, 4, 5, 6]

takewhile / dropwhile: streaming prefixes

Stop iterating at the first item that fails a predicate — useful for sorted streams where once the condition fails, it stays failed. 

pythonfrom itertools import takewhile, dropwhile

# sorted (newest first) log lines, take all the recent ones
recent = takewhile(lambda l: l.timestamp > cutoff, log_iter)

# skip blank lines at the start of a file
non_blank_start = dropwhile(lambda l: not l.strip(), f)

Compare to filter which checks every item. takewhile stops as soon as the predicate is false — never reads more than it needs. For a 10 GB log file, that's the difference between done-in-a-second and minutes.

groupby: consecutive runs only

The biggest groupby trap: it groups consecutive equal items, not all equal items. Sort first if you want SQL GROUP BY semantics. 

pythonfrom itertools import groupby

events = [
    {"host": "sw-01", "level": "info"},
    {"host": "sw-01", "level": "info"},
    {"host": "sw-02", "level": "warn"},
    {"host": "sw-01", "level": "info"},   # reappears!
]

# groupby alone gets it WRONG — splits 'sw-01' into two groups
for host, items in groupby(events, key=lambda e: e["host"]):
    print(host, list(items))

# Sort first for SQL-style grouping
events.sort(key=lambda e: e["host"])
for host, items in groupby(events, key=lambda e: e["host"]):
    print(host, list(items))

Where consecutive-grouping shines: log files where repeated lines collapse, run-length encoding, detecting change points in time series — any case where you want to compress runs of equal values.

Run-length encoding in two lines

pythondata = "AAAABBBCCDAA"
rle  = [(char, sum(1 for _ in grp)) for char, grp in groupby(data)]
# [('A', 4), ('B', 3), ('C', 2), ('D', 1), ('A', 2)]

The trailing ('A', 2) is the point: same character, but a separate run because B/C/D came in between.

The groupby gotcha: groups are iterators

Each items from groupby is a fresh iterator that's only valid until the outer loop advances. Materialize it (with list()) if you need to iterate twice or store it. 

python# BAD: by the time we do anything with grp later, it's exhausted
collected = {k: grp for k, grp in groupby(data, key=f)}
list(collected["x"])  # [] — already consumed!

# GOOD: materialize each group as you go
collected = {k: list(grp) for k, grp in groupby(data, key=f)}

The other itertools functions worth knowing for completeness: islice for slicing generators, tee for forking an iterator, product / combinations / permutations for combinatorics. The docs page is short and worth bookmarking.

Find a place where you concatenate lists with a + b + c just to iterate them. Replace with chain(a, b, c) — same iteration, no temporary list. Small win, scales well as the lists grow.
