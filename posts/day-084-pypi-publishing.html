Publishing to PyPI: trusted publishing from CI

API tokens are out. Trusted publishing via OIDC from GitHub Actions is in. Plus TestPyPI for dry runs and the version-tag workflow.

25 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Configure trusted publishing on PyPI — tell it which GitHub repo and workflow can publish. From CI, use pypa/gh-action-pypi-publish@release/v1 with no token argument; OIDC handles auth. Test on TestPyPI first. Trigger publishes from version tags so you don't ship every push to main.

Long-lived API tokens leak. Stolen ones get used to push malicious updates to popular packages. Trusted publishing fixes this by using GitHub's OIDC tokens — PyPI verifies the token came from the specific workflow you whitelisted, no secret stored anywhere.

One-time PyPI setup

On pypi.org:

  1. For an existing project: go to "Publishing", add a new "Trusted publisher".
  2. For a new project: add a "Pending trusted publisher" with the project name. The first publish creates the project.
  3. Fill in: GitHub owner, repo, workflow filename, optional environment name. PyPI matches these against the OIDC token claims.

After this, no API tokens, no ~/.pypirc, no secrets in CI.

The release workflow

yaml# .github/workflows/publish.yml
name: publish

on:
  push:
    tags: ["v*"]              # trigger only on version tags

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: astral-sh/setup-uv@v3
      - run: uv tool install hatch
      - run: hatch build
      - uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/

  publish:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: pypi
      url: https://pypi.org/p/my-tool
    permissions:
      id-token: write           # required for trusted publishing
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: dist
          path: dist/
      - uses: pypa/gh-action-pypi-publish@release/v1

Three things make this work: the id-token: write permission (lets GitHub mint an OIDC token), the environment: pypi declaration (matches what you set on PyPI), and the official action that handles the upload. No secrets, no tokens.

The release flow, step by step

bash# 1. Bump version
hatch version minor

# 2. Commit and tag
git commit -am "release v0.2.0"
git tag v0.2.0
git push && git push --tags

# 3. Workflow runs automatically:
#    build -> publish to PyPI

Tag-driven releases mean no accidental publishes. Pushing to main does normal CI; pushing a v* tag publishes.

TestPyPI: dry-run your packaging

TestPyPI is a separate index for testing the publish workflow without affecting real users. Same trusted-publishing setup; different action input. 

yaml      - uses: pypa/gh-action-pypi-publish@release/v1
        with:
          repository-url: https://test.pypi.org/legacy/
bash# Verify the test publish works as expected
pip install --index-url https://test.pypi.org/simple/ \
            --extra-index-url https://pypi.org/simple/ \
            my-tool

Once a TestPyPI publish round-trips correctly (build, upload, install, import), you've derisked the real publish.

Manual fallback: API tokens

If you can't use trusted publishing for some reason — private CI without OIDC, or a registry that doesn't support it — create a project-scoped API token on PyPI, store it as a GitHub secret (PYPI_TOKEN), and pass it to the action: 

yaml      - uses: pypa/gh-action-pypi-publish@release/v1
        with:
          password: ${{ secrets.PYPI_TOKEN }}

Less safe than trusted publishing — the token is a secret that could leak — but the workflow shape is the same.

Verify the package after publishing

bashpip install my-tool
my-tool --help                  # console script works
python -c "import my_tool; print(my_tool.__version__)"

# Inspect the wheel before relying on it
python -m wheel unpack dist/my_tool-0.2.0-py3-none-any.whl
ls my_tool-0.2.0/my_tool/        # verify file list looks right

Common publish failures

  • HTTP 403: not authorized — trusted publishing config doesn't match (workflow filename, environment name, etc.). Check both sides exactly.
  • Filename has already been used — PyPI doesn't allow re-uploading the same version. Bump and retry.
  • Invalid distribution metadata — broken README rendering. Run twine check dist/* locally before tagging.

For private packages, the same flow works against private PyPI servers (devpi, Cloudsmith, AWS CodeArtifact, etc.). Many support OIDC trusted publishing too. For internal-only packages, this is much safer than long-lived deploy tokens.

Pick a small library you've written, set up trusted publishing for a fresh PyPI project name, run the workflow against TestPyPI. Once the round-trip works, do the real publish. Then watch pip install your-package succeed from a clean environment. The first one is a milestone.
