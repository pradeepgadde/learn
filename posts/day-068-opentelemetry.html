OpenTelemetry: traces, metrics, logs in one SDK

The vendor-neutral observability standard. Auto-instrument FastAPI, requests, and DB calls in five minutes. Bolt on custom spans where they matter.

09 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Install opentelemetry-distro and the instrumentations you need (opentelemetry-instrumentation-fastapi, ...-sqlalchemy, etc.). Run with opentelemetry-instrument python -m uvicorn. Configure the OTLP exporter via env vars to ship to your backend (Tempo, Jaeger, Honeycomb, Grafana, Datadog).

OpenTelemetry (OTel) is the standard observability pipeline — one SDK, one wire protocol (OTLP), every backend you'd want. The days of Datadog SDK + New Relic agent + custom StatsD client are gone.

Auto-instrumentation: the easy mode

bashuv add opentelemetry-distro \
       opentelemetry-exporter-otlp \
       opentelemetry-instrumentation-fastapi \
       opentelemetry-instrumentation-sqlalchemy \
       opentelemetry-instrumentation-httpx

opentelemetry-bootstrap -a install   # auto-detects more

Then run your app under opentelemetry-instrument: 

bashexport OTEL_SERVICE_NAME=my-api
export OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4317

opentelemetry-instrument uvicorn app:app --host 0.0.0.0

Every HTTP request, every DB query, every outbound HTTP call now generates a span with timing and metadata. Open your backend; the traces show up.

Custom spans for business operations

Auto-instrumentation tells you "request took 800ms"; custom spans tell you which 200ms was your config-rendering loop versus the database call. 

pythonfrom opentelemetry import trace

tracer = trace.get_tracer(__name__)

def render_configs(devices):
    with tracer.start_as_current_span("render_configs") as span:
        span.set_attribute("device_count", len(devices))
        for d in devices:
            with tracer.start_as_current_span("render_one") as child:
                child.set_attribute("hostname", d.hostname)
                render(d)

Spans nest naturally. The trace view in your backend now shows render_configs as a parent with one child per device, each timed.

Recording errors on a span

pythonfrom opentelemetry.trace import Status, StatusCode

with tracer.start_as_current_span("push_config") as span:
    try:
        push(device)
    except Exception as e:
        span.record_exception(e)
        span.set_status(Status(StatusCode.ERROR, str(e)))
        raise

Metrics, briefly

OTel metrics are similar in shape to traces. Counters for things you increment (requests, errors), histograms for distributions (latency), up-down counters for gauges (queue depth). 

pythonfrom opentelemetry import metrics

meter     = metrics.get_meter(__name__)
requests  = meter.create_counter("requests", unit="1")
latency   = meter.create_histogram("latency", unit="ms")

requests.add(1, {"method": "GET", "status": "200"})
latency.record(42.5, {"endpoint": "/devices"})

Trace propagation across services

OTel auto-injects a traceparent header on outbound requests when httpx is instrumented. Other OTel-instrumented services pick it up — one trace ID across the whole call chain. This is what makes distributed tracing actually useful.

Where the data goes

The standard pattern: ship to an OpenTelemetry Collector sidecar, which routes to your actual backends. Decouples the app from the vendor — switch backends by editing the collector config, not the app.

For local development, run Jaeger in Docker with one command: docker run -p 16686:16686 -p 4317:4317 jaegertracing/all-in-one. Set OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4317 and traces show up at http://localhost:16686. Five minutes from zero to a real trace view.

On any FastAPI service: install the OTel distro, add the FastAPI and httpx instrumentations, point at a local Jaeger. Make a request that triggers an outbound call. Watch the parent and child spans show up in Jaeger with the same trace ID. The "ah-ha" moment is real.
