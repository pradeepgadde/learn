csv module: the gotchas they don't tell you

The basics work fine. Three traps — newlines, dialects, and quoting — account for most "my CSV is broken" bugs in production.

05 Feb 2026 • ~3 min read • Python 3.0+
TL;DR

Always open CSV files with newline="" — otherwise embedded newlines in fields corrupt the output on Windows. Use DictReader / DictWriter for named columns. For unknown dialects, sniff with csv.Sniffer. For real data work, jump to polars or pandas.

The standard csv module is the right tool for config files, exports, and small data interchange. It has three sharp edges that have wasted a million developer hours.

The newline trap

On Windows, line endings are \r\n. If you open a CSV file without newline="", Python's universal-newlines machinery transforms them — and any embedded \n inside a quoted field gets misread as a row break. Always pass newline="". 

pythonimport csv

# Always — read AND write
with open("data.csv", newline="") as f:
    reader = csv.reader(f)
    for row in reader:
        print(row)

with open("out.csv", "w", newline="") as f:
    w = csv.writer(f)
    w.writerow(["name", "value"])

The Python docs even say so explicitly — and ignoring it produces files that look fine on your Mac and break for the customer on Windows.

DictReader for named columns

Indexing rows by integer is fragile. Use DictReader and reference columns by header name — the script keeps working when columns are reordered. 

pythonwith open("devices.csv", newline="") as f:
    for row in csv.DictReader(f):
        print(row["hostname"], row["role"])

Three things to know: header detection is automatic from the first row. Every value comes back as a string — "42", not 42. Missing values are empty strings, not None.

Writing with DictWriter

pythonrows = [
    {"hostname": "sw-01", "role": "access"},
    {"hostname": "sw-02", "role": "core", "notes": "new"},
]

with open("out.csv", "w", newline="") as f:
    w = csv.DictWriter(
        f,
        fieldnames=["hostname", "role", "notes"],
        extrasaction="ignore",    # tolerate unknown keys
    )
    w.writeheader()
    w.writerows(rows)

extrasaction="ignore" is your friend — without it, an extra key in any row raises ValueError. extrasaction="raise" is the default and surprisingly strict.

Sniff the dialect for unknown files

Customer-supplied CSVs have a different separator (comma, semicolon, tab), different quote style, and different line endings every time. csv.Sniffer guesses. 

pythonwith open("weird.csv", newline="") as f:
    sample  = f.read(8192)
    f.seek(0)
    dialect = csv.Sniffer().sniff(sample)
    has_header = csv.Sniffer().has_header(sample)

    reader = csv.reader(f, dialect=dialect)
    if has_header:
        headers = next(reader)
    for row in reader:
        ...

The encoding gotcha

Excel exports CSVs in cp1252 on Windows and UTF-8 on macOS — with no header to indicate which. If you get UnicodeDecodeError, try encoding="utf-8-sig" first (handles BOM), then cp1252, then latin-1 (which accepts any byte, never raises). 

pythonwith open("excel-export.csv", newline="", encoding="utf-8-sig") as f:
    ...

When to leave csv behind

For datasets with hundreds of thousands of rows, type coercion needs, joins, group-bys, or aggregations — reach for polars (fast, memory-efficient) or pandas (ubiquitous). The stdlib csv module is for I/O, not analysis. 

pythonimport polars as pl

df = pl.read_csv("events.csv")
df.filter(pl.col("level") == "ERROR").group_by("service").len()

For "is this CSV well-formed?" validation before processing, the csv module won't help much. Run it through pandas.read_csv with on_bad_lines="warn" first — it surfaces row-by-row issues much more clearly.

Open a CSV-handling script you have. If it doesn't pass newline="", add it. If it indexes rows by integer position, switch to DictReader. Two changes, much more robust code.
