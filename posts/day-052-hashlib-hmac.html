hashlib + hmac: hashing and signatures done right

hashlib for content fingerprints, hmac for verified signatures, compare_digest to defeat timing attacks. The crypto basics every backend touches.

21 Feb 2026 • ~3 min read • Python 3.0+
TL;DR

Use SHA-256 (or BLAKE2) from hashlib for content fingerprints — never MD5 or SHA-1. Use hmac.new(key, msg, "sha256") for signed payloads. Always compare digests with hmac.compare_digest, never ==. For password hashing, use passlib or argon2-cffi, not raw hashing.

Hashing comes up in three contexts: integrity (does this file match what I expected?), keying (deduplication, content-addressable storage), and signatures (did this come from someone who knows the secret?). Each has its right tool.

Content fingerprints with hashlib

pythonimport hashlib

# For small in-memory data
digest = hashlib.sha256(b"hello").hexdigest()

# For files, stream in chunks — never read the whole thing
def file_sha256(path):
    h = hashlib.sha256()
    with open(path, "rb") as f:
        while chunk := f.read(65536):
            h.update(chunk)
    return h.hexdigest()

On 3.11+, there's a one-liner for the file case: hashlib.file_digest(open(path, "rb"), "sha256").hexdigest(). Same chunking, less code.

What to use, what to avoid

  • SHA-256 — the safe default. Universal support, fast on modern CPUs (most have hardware acceleration).
  • BLAKE2 / BLAKE3 — faster than SHA-256 in pure software, equally secure. Use when speed matters.
  • SHA-1, MD5 — broken for security purposes. Fine as non-crypto checksums (caching keys, deduplication where collisions are tolerable). Never for signatures.

HMAC: signatures with a shared secret

Webhook payloads, session tokens, signed URLs — anywhere "did this come from a party that knows the secret?" matters. Hashing alone is not enough — an attacker can compute a hash of anything. HMAC mixes in a key. 

pythonimport hmac, hashlib

secret = b"shared-with-the-other-side"

def sign(payload: bytes) -> str:
    return hmac.new(secret, payload, "sha256").hexdigest()

def verify(payload: bytes, signature: str) -> bool:
    expected = sign(payload)
    return hmac.compare_digest(expected, signature)

That's the GitHub-webhook pattern (and Stripe, and Slack, and most others) in three lines.

The timing-attack defense

compare_digest takes the same time regardless of where two strings differ. == short-circuits at the first mismatched byte, which leaks information about the secret one byte at a time over many requests. 

python# BAD — timing reveals how many bytes the attacker got right
if submitted_token == expected_token:
    ...

# GOOD — constant time
if hmac.compare_digest(submitted_token, expected_token):
    ...

Use compare_digest for any string comparison involving secrets — tokens, API keys, signatures. The cost is essentially zero; the safety is real.

Don't roll your own password hashing

hashlib.sha256(password) is wrong for passwords. It's too fast — an attacker with a leaked database can try billions of passwords per second. Use a function deliberately built for password hashing. 

python# Use argon2-cffi (current best practice) or passlib
from argon2 import PasswordHasher

ph     = PasswordHasher()
stored = ph.hash("correct horse battery staple")
ph.verify(stored, "correct horse battery staple")   # True / raises

Argon2 is intentionally slow (tens of milliseconds per check) and memory-hard, defeating GPU and ASIC brute-forcing. The standard library has hashlib.scrypt and hashlib.pbkdf2_hmac, both better than plain SHA-256 but worse than Argon2 for new code.

For full-blown encryption (AES, RSA, asymmetric keys), reach for cryptography. The standard library doesn't include encryption primitives directly — deliberately, because getting them right is hard and outsourcing to a well-maintained library is the correct call.

Find a script that compares secrets with == — a webhook verifier, a token check, an API-key compare. Switch to hmac.compare_digest. Two-second fix for a real class of attack.
