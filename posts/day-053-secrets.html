secrets: not random, secure

random is for dice rolls. secrets is for tokens, passwords, and anything an attacker shouldn't be able to guess.

22 Feb 2026 • ~2 min read • Python 3.6+
TL;DR

For anything security-sensitive — tokens, password resets, session IDs, API keys — use the secrets module, never random. secrets.token_urlsafe(32) gives you a URL-safe base64 token; secrets.token_hex(32) gives you hex. Both are cryptographically random.

random uses a deterministic PRNG (Mersenne Twister). Given enough output, an attacker can reconstruct the internal state and predict future values. Fine for a coin flip; catastrophic for a password reset token. The secrets module was added in Python 3.6 specifically because too many web apps were using random for security.

The token generators

pythonimport secrets

# 32 bytes of entropy, URL-safe base64 (43 chars)
secrets.token_urlsafe(32)
# 'q3lUSvR7XO9aMpw_3KNs1tHF8b-jJYzVe5gGm6dRcAk'

# 32 bytes, hex-encoded (64 chars)
secrets.token_hex(32)
# 'a4f8c9d2...'

# Raw bytes, when you'll encode it yourself
secrets.token_bytes(32)

32 bytes (256 bits) is the right default for sensitive tokens. Smaller tokens like 16 bytes (128 bits) are still strong; smaller than that starts being guessable.

Picking a random element, the safe way

Need a random PIN, a temporary password from a wordlist, a random API endpoint to probe? Use secrets.choice, not random.choice. 

pythonimport secrets, string

# 12-char alphanumeric password
alphabet = string.ascii_letters + string.digits
password = "".join(secrets.choice(alphabet) for _ in range(12))

# 6-digit PIN (numbers below have 10**6 possibilities — small!)
pin = "".join(secrets.choice(string.digits) for _ in range(6))

Same API as random, different underlying source — in fact, on most systems it reads from os.urandom, which pulls from the OS's CSPRNG.

Bounded random integers

python# Random integer in range(0, n) — uniform, no modulo bias
secrets.randbelow(100)

# Random N-bit integer
secrets.randbits(128)

Unlike random.randrange, these are free of the subtle modulo-bias bugs that affect naive random_int % n patterns.

The decision rule

It's mostly black-and-white:

  • Anything an attacker shouldn't predict — tokens, password reset links, session IDs, CSRF tokens, generated passwords, API keys, nonces — secrets.
  • Everything else — sampling, simulation, picking a random message for testing, shuffling a deck for a game — random is fine and faster.

Comparing tokens safely

Echoing yesterday's lesson: when comparing a submitted token to a stored one, use hmac.compare_digest. Otherwise, the token's strength is wasted on a side-channel. 

pythonimport hmac, secrets

stored    = secrets.token_urlsafe(32)
submitted = request.params["token"]

if hmac.compare_digest(stored, submitted):
    ...

For URL-safe tokens that double as primary keys (without leaking row counts the way auto-increment IDs do), secrets.token_urlsafe(16) is a great choice. Roughly 22 characters, fits in a URL, looks tidy, no collision worries up to billions of rows.

Search your codebase for random.choice or random.randint. For each one, ask: "would I care if an attacker could predict this?" If yes — even if it's a long shot — switch to the corresponding secrets function. The change is one import.
