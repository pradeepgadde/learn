pytest parametrize: tables of test cases

One test function, many test cases. Tables of input/output, readable failure IDs, and stacked decorators for combinatorial coverage.

19 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Use @pytest.mark.parametrize to replace 4 nearly-identical test functions with one test and a table of inputs. Provide ids= for readable test names. Stack two decorators to get the cartesian product. Use pytest.param(..., marks=...) to skip or xfail a single row.

The boilerplate trap with tests: writing test_subnet_24_handles_x, test_subnet_16_handles_x, test_subnet_8_handles_x as three functions when they only differ in a couple of values. Parametrize collapses these to one.

Basic parametrize

pythonimport pytest
from ipaddress import ip_network

@pytest.mark.parametrize("cidr, expected_hosts", [
    ("10.0.0.0/24", 254),
    ("10.0.0.0/30", 2),
    ("10.0.0.0/31", 2),    # point-to-point — special-case
    ("10.0.0.0/32", 1),    # single host
])
def test_usable_hosts(cidr, expected_hosts):
    assert usable_hosts(ip_network(cidr)) == expected_hosts

One function, four tests, four lines of test cases. When the third row fails, pytest reports test_usable_hosts[10.0.0.0/31-2] — the inputs are baked into the test ID.

Custom ids for readable failure messages

For complex parameters (objects, dicts, large strings) the auto-generated IDs get ugly. Spell them out: 

python@pytest.mark.parametrize(
    "config, valid",
    [
        ({"hostname": "sw-01", "role": "access"}, True),
        ({"hostname": "",      "role": "access"}, False),
        ({"hostname": "sw-01", "role": "???"},    False),
    ],
    ids=["happy_path", "empty_hostname", "unknown_role"],
)
def test_validate(config, valid):
    assert validate(config) is valid

Now failures look like test_validate[unknown_role] — immediately useful.

Stacked parametrize: cartesian product

python@pytest.mark.parametrize("vendor",  ["cisco", "arista", "juniper"])
@pytest.mark.parametrize("command", ["show version", "show interfaces"])
def test_command_supported(vendor, command):
    assert supports(vendor, command)

That's 3 vendors × 2 commands = 6 tests from one function. Useful when you want coverage of every combination; less useful (and slow) when many combinations don't make sense.

Skipping or xfailing single rows

Sometimes one row of the table is broken on purpose, or only valid on certain Python versions. Wrap it in pytest.param: 

python@pytest.mark.parametrize("cidr, count", [
    ("10.0.0.0/24", 254),
    pytest.param("::/64", 2**64,
                 marks=pytest.mark.skip(reason="v6 not yet supported")),
    pytest.param("10.0.0.0/31", 2,
                 marks=pytest.mark.xfail(reason="RFC 3021 edge case")),
])
def test_count(cidr, count): ...

skip doesn't run the test; xfail runs it and expects failure — if it unexpectedly passes you get XPASS, a signal you can remove the marker.

indirect: parametrize fixtures, not just args

When the parameter needs to feed into a fixture (so the fixture sets things up differently based on the row), use indirect=True. 

python@pytest.fixture
def device_role(request):
    # request.param is the value passed in via parametrize
    return make_device(role=request.param)

@pytest.mark.parametrize("device_role",
                          ["access", "distribution", "core"],
                          indirect=True)
def test_uplinks(device_role):
    # device_role here is a Device object, not the string
    assert device_role.uplinks > 0

Combines parametrize with the factory pattern from yesterday — one function name, three fully-different setups.

Big tables from external data

For really big tables, generate the param list in conftest.py from a JSON or CSV. 

python# conftest.py
import json, pathlib, pytest

_cases = json.loads(pathlib.Path("tests/cases.json").read_text())

def pytest_generate_tests(metafunc):
    if "case" in metafunc.fixturenames:
        metafunc.parametrize("case", _cases,
                              ids=[c["name"] for c in _cases])

Now any test taking a case argument runs over the entire JSON. Common pattern for golden-output tests.

Resist the urge to put too much in one parametrized test. If you have 50 rows and they all assert the same thing about slightly different inputs, that's perfect. If they branch into "for this row check A, for that row check B" — split into separate tests.

Find a test file with three or more test_X_does_Y functions that differ only in their inputs. Collapse to one parametrized test. The diff will likely be satisfying — and the next test case becomes one row, not one new function.
