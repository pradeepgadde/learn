factory_boy and faker: realistic test data

Test data that looks like real data. Faker for individual fields, factory_boy for whole objects. Sub-factories, sequences, post-generation hooks.

21 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

faker generates one realistic value at a time (names, addresses, IPs). factory_boy wraps it into reusable builders for whole models. Use sequences for unique fields, sub-factories for relationships, traits for variants. For pydantic-only codebases, polyfactory derives factories automatically from your models.

Device(hostname="sw-01", role="access") gets old fast. Real test data has variety, and having a builder per model means every test gets unique values without the test author having to think about it.

Faker: realistic-looking primitives

pythonfrom faker import Faker

fake = Faker()

fake.name()              # 'Allison Hill'
fake.email()             # 'kennedy24@example.com'
fake.ipv4_private()      # '10.247.62.13'
fake.mac_address()       # '7e:11:c0:cd:6e:08'
fake.hostname()          # 'desktop-79.davis.com'
fake.date_time_this_year()
fake.company()           # 'Davis-Hernandez'
fake.text(max_nb_chars=200)

Faker has hundreds of providers and supports locale (Faker("ja_JP") for Japanese names and addresses). For network-engineering tasks the IP/MAC/hostname ones are great.

factory_boy: builders that compose

pythonimport factory
from factory import Faker, Sequence, SubFactory
from myapp.models import Site, Device

class SiteFactory(factory.Factory):
    class Meta:
        model = Site
    id   = Sequence(lambda n: n)
    name = Faker("city")

class DeviceFactory(factory.Factory):
    class Meta:
        model = Device
    hostname = Sequence(lambda n: f"sw-{n:04d}")
    role     = Faker("random_element",
                       elements=["access", "core", "distribution"])
    ip       = Faker("ipv4_private")
    site     = SubFactory(SiteFactory)

Sequence gives unique values per instance — sw-0001, sw-0002, etc. SubFactory chains factories together: building a Device auto-builds a Site unless you pass one in.

Using factories in tests

pythondef test_role_filter():
    access_devices = DeviceFactory.build_batch(3, role="access")
    core_devices   = DeviceFactory.build_batch(2, role="core")
    all_devices    = access_devices + core_devices
    assert filter_by_role(all_devices, "access") == access_devices

def test_one_off():
    d = DeviceFactory.build(role="core", site__name="DC1")
    # site__name= reaches into the SubFactory and overrides Site.name
    assert d.site.name == "DC1"

build() creates the object; build_batch(N, ...) creates a list. For ORM models, create() / create_batch() also persist to the database.

Traits: named variants

pythonclass DeviceFactory(factory.Factory):
    class Meta:
        model = Device
    hostname = Sequence(lambda n: f"sw-{n}")
    role     = "access"
    decommissioned = False

    class Params:
        decom = factory.Trait(
            decommissioned=True,
            hostname=Sequence(lambda n: f"DECOM-{n}"),
        )

# Use:
DeviceFactory.build(decom=True)
# hostname='DECOM-7', decommissioned=True

Traits keep "the device but with X variation" readable. Way nicer than passing many overrides.

Post-generation hooks

For relationships you can't fit into a single constructor call — like a many-to-many. 

pythonclass SiteFactory(factory.Factory):
    class Meta:
        model = Site
    name = Faker("city")

    @factory.post_generation
    def devices(obj, create, extracted, **kwargs):
        if extracted:
            # Caller passed `devices=[...]`
            obj.devices.extend(extracted)
        elif "count" in kwargs:
            # Caller passed `devices__count=3`
            obj.devices.extend(DeviceFactory.build_batch(kwargs["count"]))

polyfactory: zero-config for pydantic and dataclasses

For codebases built around pydantic models, polyfactory derives factories automatically from the type hints — no manual factory definitions needed. 

pythonfrom polyfactory.factories.pydantic_factory import ModelFactory

class DeviceFactory(ModelFactory[Device]):
    __model__ = Device

device = DeviceFactory.build()
# Already valid against the pydantic schema; types match

Polyfactory inspects the model's type hints and generates conformant data automatically. For 90% of cases this is enough; override fields only when you need something specific.

For database-backed factories with SQLAlchemy, the SQLAlchemyModelFactory base class adds session-aware create() that inserts the row. Pair with the db_session fixture from Day 077 and you have a clean factories-on-real-DB setup.

Find a test file with hand-rolled factory functions or repeated Device(...) constructions. Replace with a single factory_boy or polyfactory definition. The tests get more variety; the next test you write needs less setup; data-related bugs start surfacing in cases you'd have missed with hand-picked values.
