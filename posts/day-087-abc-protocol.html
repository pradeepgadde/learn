ABCs and Protocols: two flavors of "this is a Foo"

ABCs enforce inheritance; Protocols enforce shape. When to use each, and why Protocols turned out to be the cleaner answer for most modern code.

28 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

An ABC (abc.ABC) defines an interface that subclasses must inherit from and implement. A Protocol (typing.Protocol) defines a shape; any class with matching methods satisfies it, no inheritance required. Use Protocols for "looks like a duck" structural typing — especially when the implementers aren't yours to modify.

Python's "duck typing" has always been runtime-only: if it walks like a duck, it's a duck. Type checkers needed a way to express that statically. The pre-PEP 544 answer was ABCs (nominal: you must declare you're a duck). The post-PEP 544 answer is Protocols (structural: type checker checks if you walk like one).

ABCs: nominal interfaces

pythonfrom abc import ABC, abstractmethod

class Driver(ABC):
    @abstractmethod
    def connect(self, host: str) -> None: ...

    @abstractmethod
    def send(self, cmd: str) -> str: ...

class CiscoDriver(Driver):
    def connect(self, host): ...
    def send(self, cmd):    ...

# Forgetting an abstract method:
class BrokenDriver(Driver):
    def connect(self, host): ...

BrokenDriver()
# TypeError: Can't instantiate abstract class BrokenDriver
# with abstract method send

Strengths:

  • Runtime enforcement. Python refuses to instantiate an incomplete subclass. Mistakes get caught early.
  • Shared implementation. ABCs can carry concrete helper methods.
  • Clear lineage. isinstance(d, Driver) is true, definitively.

Weakness: every implementer has to inherit. If you want to plug in a third-party class that happens to have connect and send, you can't — you'd have to wrap it.

Protocols: structural interfaces

pythonfrom typing import Protocol

class Driver(Protocol):
    def connect(self, host: str) -> None: ...
    def send(self, cmd: str) -> str: ...

class CiscoDriver:                         # NO inheritance from Driver
    def connect(self, host): ...
    def send(self, cmd):    ...

def use(d: Driver) -> str:
    d.connect("sw-01")
    return d.send("show ver")

use(CiscoDriver())                          # type checker: OK, structural match

CiscoDriver doesn't inherit from Driver. It just has the right shape. Mypy/pyright accept it because the method signatures match.

Strengths:

  • No coupling. Implementers don't need to import or inherit your protocol — their existing classes might already match.
  • Real duck-typing, statically checked. The runtime semantics Python always had, now with a type-checker enforcement layer.
  • Good for adapters. Need a SupportsClose, SupportsConnect? Define a tiny protocol; anything closeable works.

runtime_checkable for isinstance

Protocols are normally checked statically. Add @runtime_checkable to also get isinstance(x, Driver) at runtime — checks for matching attribute names (not signatures). 

pythonfrom typing import Protocol, runtime_checkable

@runtime_checkable
class Closeable(Protocol):
    def close(self) -> None: ...

if isinstance(obj, Closeable):
    obj.close()

Caveat: runtime checks only verify the names, not the signatures or types. For real safety, rely on the type checker; treat isinstance as a hint.

When to pick which

SituationPick
You want shared implementation in the base ABC
Implementers will inherit from your class anyway ABC
You want runtime "instantiation refused" enforcement ABC
You're typing third-party classes you don't own Protocol
You want the lightest possible "looks like a Foo" Protocol
You're a library author writing API surface Protocol (usually)

The standard library has both

collections.abc ships dozens of ABCs — Iterable, Mapping, Hashable, Sized. Many double as Protocols for type-checking purposes. So def f(xs: Iterable[int]) works for any iterable, no inheritance required.

typing.SupportsInt, SupportsFloat, SupportsAbs are protocols all the way down — they capture single-method capabilities like "convertible to int".

A practical pattern: define a Protocol for the public surface, ship an ABC implementation that base classes can inherit from when they want shared behaviour. Consumers depend on the protocol; producers can choose.

Take an ABC you have somewhere with no shared implementation — just abstract methods. Convert to a Protocol. Drop the inheritance from one or two implementing classes. Run mypy. The type-check still passes; the coupling went away. That's the structural-typing win in miniature.
