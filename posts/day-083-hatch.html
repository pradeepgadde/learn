hatch: building Python packages without setup.py

Build modern packages from a single pyproject.toml. No setup.py, no MANIFEST.in. Environments, scripts, and version bumps in one tool.

24 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

hatch new myproject scaffolds a modern, src-layout package. Edit pyproject.toml for metadata. hatch build produces wheel + sdist; hatch publish uploads to PyPI. hatch version patch bumps the version. Drop-in replacement for setuptools+twine for new projects.

The Python packaging story has been confused for a decade. As of 2026, pyproject.toml + a PEP 517 build backend is the consensus. hatch is one of the cleanest of those backends, and includes a CLI for the rest of the workflow.

Start a new project

bashuv tool install hatch
hatch new my-tool
cd my-tool
textmy-tool/
├── src/
│   └── my_tool/
│       ├── __init__.py
│       └── __about__.py     # __version__ = "0.0.1"
├── tests/
│   └── __init__.py
├── pyproject.toml
├── README.md
└── LICENSE.txt

Note the src-layout (Python package inside src/). It prevents a class of bug where import my_tool picks up the local source folder instead of the installed version — tests pass when they shouldn't. src-layout is the modern best practice.

The pyproject.toml in full

toml[build-system]
requires      = ["hatchling"]
build-backend = "hatchling.build"

[project]
name            = "my-tool"
description     = "A small useful tool"
readme          = "README.md"
requires-python = ">=3.10"
license         = "MIT"
authors         = [{ name = "Pradeep Gadde" }]
keywords        = ["networking", "automation"]
classifiers     = [
    "Development Status :: 3 - Alpha",
    "Programming Language :: Python :: 3",
]
dependencies    = [
    "httpx>=0.27",
    "pydantic>=2",
]
dynamic = ["version"]    # pulled from __about__.py

[project.urls]
Homepage   = "https://github.com/pradeepgadde/my-tool"
Issues     = "https://github.com/pradeepgadde/my-tool/issues"

[project.scripts]
my-tool = "my_tool.cli:main"     # installs the `my-tool` console command

[project.optional-dependencies]
dev = ["pytest", "ruff", "mypy"]

[tool.hatch.version]
path = "src/my_tool/__about__.py"

Single file. Replaces setup.py, setup.cfg, MANIFEST.in, versioneer — the whole pile of legacy artifacts.

Build and inspect

bashhatch build
ls dist/
# my_tool-0.1.0-py3-none-any.whl
# my_tool-0.1.0.tar.gz

Both built — the wheel for end users (pip install), the sdist as a fallback. Inspect them with unzip -l for the wheel, tar tzf for the sdist; verify only the files you wanted ended up included.

Version management

bashhatch version              # show current
hatch version patch        # 0.1.0 -> 0.1.1
hatch version minor        # 0.1.1 -> 0.2.0
hatch version major        # 0.2.0 -> 1.0.0
hatch version 1.2.3        # set explicitly

Edits the __version__ in __about__.py. The dynamic entry in pyproject means the package version is sourced from there — one place to change.

Hatch environments (optional)

Hatch can also manage virtual environments and scripts — an alternative to using uv for the same job. If you're already on uv, ignore these. If not: 

toml[tool.hatch.envs.default]
dependencies = ["pytest", "pytest-cov"]

[tool.hatch.envs.default.scripts]
test     = "pytest {args:tests}"
test-cov = "pytest --cov=src {args:tests}"
lint     = "ruff check ."
fmt      = "ruff format ."
bashhatch run test
hatch run lint

The competitors, briefly

  • setuptools + setup.py — the legacy. Still works, still ubiquitous, still has too many configuration files. New projects shouldn't pick this.
  • poetry — older, opinionated, has its own dependency resolver. Many people use it; some find it slow and prefer different conventions.
  • pdm — PEP 582 / PEP 621 focused, fast, full-featured.
  • flit — minimalist; great for pure-Python single-package projects.
  • uv — the fast newcomer (Day 019). Handles dependency management; pairs with hatch (or any PEP 517 backend) for the build step.

For most projects in 2026, the natural pairing is uv for environment + dependency management and hatchling for the build backend. Both small, both modern, both stick to the standards.

Take any small Python script you've been meaning to clean up. Run hatch new my-script-name, move the code in, edit pyproject.toml. Then hatch build. The full "package-shaped" version of your script in ten minutes. Whether you publish it or not, it's now installable, importable, and version-able.
