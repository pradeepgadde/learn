SSH to devices: paramiko vs netmiko

paramiko is the SSH library. netmiko is the network-device wrapper that knows about prompts, enable mode, and config sessions. Pick by the job.

24 Jan 2026 • ~3 min read • Python 3.8+
TL;DR

For Linux servers, paramiko (or fabric on top of it). For network gear — Cisco, Arista, Juniper, anything with an enable prompt and a prompt-driven CLI — netmiko handles the device-specific quirks so you don't have to. Both run on top of OpenSSH-compatible servers.

SSH automation in Python splits cleanly into "I'm running Linux commands on a server" and "I'm interacting with a network device over its CLI." Different problems, different libraries.

paramiko: server-style SSH

paramiko is a pure-Python SSH client and server. It's the foundation under fabric, netmiko, and most Python-based deployment tools. 

pythonimport paramiko

client = paramiko.SSHClient()
client.load_system_host_keys()
client.set_missing_host_key_policy(paramiko.RejectPolicy())

client.connect("server.local", username="pradeep", key_filename="~/.ssh/id_ed25519")

_, stdout, stderr = client.exec_command("systemctl status nginx")
print(stdout.read().decode())
client.close()

Notice RejectPolicy, not AutoAddPolicy. Auto-adding unknown host keys is the SSH equivalent of skipping TLS verification — tempting in tutorials, wrong in production.

netmiko: device-aware SSH

Network devices have prompts, paging (--More--), enable modes, and configure-terminal sessions. paramiko knows none of that. netmiko ships a device_type registry with the right behavior for each platform. 

pythonfrom netmiko import ConnectHandler

device = {
    "device_type": "cisco_ios",
    "host":        "core-sw-01",
    "username":    "netadmin",
    "password":    os.environ["NET_PASS"],
    "secret":      os.environ["NET_ENABLE"],
}

with ConnectHandler(**device) as net:
    net.enable()
    output = net.send_command("show ip interface brief")
    print(output)

send_command waits for the device prompt to return, strips paging, and times out cleanly. Compare that to writing your own prompt-detection regex over a raw paramiko channel — which is what people did for years before netmiko existed.

Pushing config with netmiko

pythoncommands = [
    "interface GigabitEthernet1/0/24",
    " description uplink to dc-router-02",
    " switchport mode trunk",
    " no shutdown",
]

with ConnectHandler(**device) as net:
    net.enable()
    result = net.send_config_set(commands)
    net.save_config()
    print(result)

send_config_set handles entering and exiting config mode for you. save_config issues the right command for the platform — copy run start on Cisco IOS, commit on Junos, etc.

Structured output with TextFSM

Network commands return formatted text, not JSON. netmiko bundles ntc-templates — pass use_textfsm=True and you get a list of dicts back instead of raw text. 

pythoninterfaces = net.send_command(
    "show ip interface brief",
    use_textfsm=True,
)
# [{'interface': 'Gi1/0/1', 'ipaddr': '10.0.0.1', 'status': 'up', ...}, ...]

up = [i for i in interfaces if i["status"] == "up"]

For dozens of devices in parallel, neither library is async. Wrap your per-device function in a ThreadPoolExecutor — SSH is I/O-bound, so threads work fine. For hundreds of devices, nornir or asyncssh become worth the migration cost.

Pick three devices on your network. Write a script that connects to each, runs show version, and prints a one-line summary: hostname, model, software version. Use use_textfsm=True so you can extract fields by name instead of parsing strings.
