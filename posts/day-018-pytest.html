pytest, the test you'll actually write

Plain functions. Plain assert. Four built-in helpers cover almost everything else.

18 Jan 2026 • ~3 min read • Python 3.7+
TL;DR

A pytest test is a function whose name starts with test_, with a plain assert inside. Use @pytest.fixture for setup, @pytest.mark.parametrize for tables of cases, tmp_path for filesystem isolation, and pytest.raises for exception testing.

unittest still works. But the modern Python ecosystem is on pytest for one reason: assert x == y beats self.assertEqual(x, y), and the failure messages are dramatically better.

The simplest possible test

python# test_math.py
def test_addition_works():
    assert 2 + 2 == 4

def test_lists_combine():
    assert [1, 2] + [3] == [1, 2, 3]

Run it with pytest. No subclassing, no class at all, no special imports. When an assertion fails, pytest rewrites the bytecode to show you both sides:

def test_addition_works(): > assert 2 + 2 == 5 E assert (2 + 2) == 5 E + where 4 = 2 + 2

Fixtures replace setUp

A fixture is a function that produces a value. Tests that request it as a parameter receive it. pytest handles the setup and teardown order automatically. 

pythonimport pytest

@pytest.fixture
def sample_users():
    return [
        {"name": "Ada",   "role": "founder"},
        {"name": "Linus", "role": "maintainer"},
    ]

def test_count(sample_users):
    assert len(sample_users) == 2

def test_first_is_founder(sample_users):
    assert sample_users[0]["role"] == "founder"

For setup-then-teardown, write the fixture as a generator and yield the value: 

python@pytest.fixture
def db_connection():
    conn = connect()
    yield conn            # test runs here
    conn.close()             # cleanup, even on failure

Built-in fixtures: tmp_path

Need to write to disk in a test? Don't touch /tmp directly. Request tmp_path — pytest gives you a fresh Path for each test and cleans it up afterward. 

pythondef test_writes_config(tmp_path):
    target = tmp_path / "config.toml"
    write_config(target, port=8080)
    assert "port = 8080" in target.read_text()

Other built-ins worth knowing: capsys (capture stdout/stderr), monkeypatch (patch attrs and env vars), caplog (capture log records).

Tables of cases: parametrize

When the same test should run against many inputs, don't write a loop. Parametrize. Each case becomes its own test with its own pass/fail status. 

python@pytest.mark.parametrize("input,expected", [
    ("hello",  "HELLO"),
    ("World",  "WORLD"),
    ("",       ""),
    ("123",    "123"),
])
def test_uppercase(input, expected):
    assert input.upper() == expected

Testing exceptions

pythondef test_zero_division_raises():
    with pytest.raises(ZeroDivisionError):
        1 / 0

def test_invalid_port_raises():
    with pytest.raises(ValueError, match="port must be"):
        parse_port("-1")

The match argument is a regex against the exception message. Catches the case where you raise the right exception type for the wrong reason.

Run only matching tests with pytest -k name. Stop on the first failure with -x. Verbose output with -v. Drop into the debugger on failure with --pdb. These four flags handle most day-to-day debugging.

Pick the smallest pure function in your codebase. Write three pytest tests for it — happy path, edge case, error case. Use parametrize if any of the cases share structure. Total time: under five minutes.
