Pydantic v2 deep cuts: validators, serializers, computed fields

field_validator and model_validator for custom logic. computed_field for derived attributes. field_serializer for output transforms.

03 Mar 2026 • ~3 min read • Python 3.8+
TL;DR

Use @field_validator("name") to validate or transform a single field. Use @model_validator(mode="after") for cross-field rules. Use @computed_field to expose a derived attribute as if it were stored. Use @field_serializer("name") to customize how a field shows up in model_dump.

Day 032 covered the pydantic basics — declare a model, get validation. The next layer is custom logic: validators that normalize input, cross-field rules, derived fields, and output transforms. Once you've used these you stop thinking of pydantic as "just" a validator.

field_validator: per-field rules

pythonfrom pydantic import BaseModel, field_validator

class Device(BaseModel):
    hostname: str
    role:     str

    @field_validator("hostname")
    @classmethod
    def _lowercase(cls, v: str) -> str:
        if not v.replace("-", "").isalnum():
            raise ValueError("hostname must be alnum + hyphens")
        return v.lower()

The validator runs after the type coerces successfully. It can return a transformed value (the lowercased hostname here) or raise to reject. Notice @classmethod — pydantic validators are class methods.

mode="before": run before type coercion

For inputs in the wrong shape that you want to normalize before type validation, use mode="before". 

pythonclass Tags(BaseModel):
    labels: list[str]

    @field_validator("labels", mode="before")
    @classmethod
    def _split(cls, v):
        if isinstance(v, str):
            return [s.strip() for s in v.split(",")]
        return v

Tags(labels="alpha, beta, gamma").labels
# ['alpha', 'beta', 'gamma']

model_validator: cross-field rules

pythonfrom pydantic import BaseModel, model_validator
from datetime import date

class Maintenance(BaseModel):
    start: date
    end:   date

    @model_validator(mode="after")
    def _check_order(self):
        if self.end < self.start:
            raise ValueError("end must be on or after start")
        return self

mode="after" runs after all fields validated — you get a fully-built self to inspect. Always return self at the end.

computed_field: derived attributes in dumps

For a value that's a function of other fields and you want it included in model_dump and the OpenAPI schema: 

pythonfrom pydantic import BaseModel, computed_field

class Probe(BaseModel):
    success: int
    failure: int

    @computed_field
    @property
    def success_rate(self) -> float:
        total = self.success + self.failure
        return self.success / total if total else 0.0

Probe(success=9, failure=1).model_dump()
# {'success': 9, 'failure': 1, 'success_rate': 0.9}

Without @computed_field, plain @property works for attribute access but is invisible to model_dump and the OpenAPI schema. The decorator wires both up.

field_serializer: control the output shape

Want a SecretStr to never leak in logs, or a datetime in epoch seconds, or a complex object as a string? Override the per-field serialization. 

pythonfrom pydantic import BaseModel, SecretStr, field_serializer

class User(BaseModel):
    name:  str
    token: SecretStr

    @field_serializer("token", when_used="json")
    def _redact(self, v: SecretStr):
        return "***"

User(name="pradeep", token="super-secret").model_dump_json()
# '{"name":"pradeep","token":"***"}'

when_used="json" means the redaction applies only to JSON dumps; model_dump() still returns the real value if you need it in-process.

Strict mode

Pydantic v2 coerces by default ("22" → 22). Sometimes you want exact types. Per-field, per-model, or per-call: 

pythonfrom pydantic import BaseModel, Field

class Strict(BaseModel):
    port: int = Field(strict=True)

Strict(port="22")
# ValidationError — string not accepted

# Or per-call:
Strict.model_validate({"port": "22"}, strict=True)

Strict mode pairs well with parsing data from other code (where types should already be right); permissive mode pairs better with parsing JSON from clients.

For very high-throughput parsing where every microsecond counts, look at TypeAdapter — pydantic's stand-alone validator for non-class types (raw list[Device], dict[str, int]). It's the same fast Rust core without the BaseModel overhead.

Take a model in your code that has an __init__ doing manual normalization ("uppercase this", "default that to today"). Move that logic into a field_validator. The model now works the same when constructed positionally, via model_validate, or via model_validate_json — one place to maintain.
