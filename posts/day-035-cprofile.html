cProfile: find the slow line

Profile first, optimize second. cProfile is in the standard library; snakeviz turns its output into a flame graph you can actually read.

04 Feb 2026 • ~3 min read • Python 3.0+
TL;DR

Run python -m cProfile -o out.prof your_script.py to generate a profile. Visualize with snakeviz out.prof for a browser-based flame graph that makes hot paths obvious. For interactive code, use cProfile.Profile() as a context manager.

Donald Knuth's "premature optimization is the root of all evil" gets quoted everywhere. The full quote ends with "...we should not pass up our opportunities in that critical 3%." Profiling tells you where the 3% is. Without it, you optimize on hunches and waste days speeding up code that wasn't the bottleneck.

The simplest profile

bash# Profile a whole script, dump to file
python -m cProfile -o slow.prof my_script.py

# Quick text summary, top 20 by cumulative time
python -m cProfile -s cumulative my_script.py | head -25

The text output is dense but readable. Columns: ncalls (call count), tottime (time in this function alone), cumtime (time in this function plus everything it called), filename:lineno(function).

The visual: snakeviz

bashpip install snakeviz
snakeviz slow.prof
# Opens a browser tab with an interactive icicle/sunburst graph

snakeviz turns the profile into a clickable visualization. Wide bars = a lot of cumulative time spent. The hot path jumps out instantly. Click any function to zoom in and see what it spent its time on.

Profiling a specific block

For long-running scripts where you only care about one section, use cProfile.Profile() as a context manager. 

pythonimport cProfile, pstats

with cProfile.Profile() as prof:
    expensive_function()

# Top 15 by cumulative time, written to disk for snakeviz
stats = pstats.Stats(prof)
stats.sort_stats("cumulative").print_stats(15)
stats.dump_stats("hot_block.prof")

Quick wall-clock measurements: perf_counter

For one-off "is X faster than Y" questions, you don't need a full profiler — time.perf_counter is the right tool. It's monotonic and high-resolution. 

pythonfrom time import perf_counter

t0 = perf_counter()
result = heavy_compute(data)
print(f"took {perf_counter() - t0:.3f}s")

For micro-benchmarks (comparing two implementations of a small function), use timeit — it runs the snippet many times to average out noise. 

bashpython -m timeit -s "items = list(range(1000))" "sum(items)"
# 100000 loops, best of 5: 4.21 usec per loop

For function-level granularity inside a hot path

cProfile measures at the function level. If your hot path is a single function with a complex body and you want line-level timings, install line-profiler and decorate the function with @profile. 

bashpip install line-profiler
kernprof -l -v my_script.py
# Per-line hits and per-line wall time for each @profile-decorated function

Memory, while we're here

Different problem, different tool. For peak memory and allocation tracking, use tracemalloc from the standard library, or memray (from Bloomberg) for deep allocation profiles. 

pythonimport tracemalloc

tracemalloc.start()
do_thing()
snapshot = tracemalloc.take_snapshot()
for stat in snapshot.statistics("lineno")[:10]:
    print(stat)

Profiling overhead skews results. cProfile adds non-trivial overhead for functions that are called millions of times — a tight inner loop will look slower than it really is. For this case, sampling profilers like py-spy (which can also attach to a running process) give more accurate real-world numbers.

Pick the slowest script you have. Profile it with python -m cProfile -o p.prof script.py. Open snakeviz p.prof. Find the widest bar that wasn't what you expected. That's where to optimize first.
